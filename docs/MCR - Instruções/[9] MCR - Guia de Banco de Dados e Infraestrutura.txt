⚠️ REGRAS DE REVISÃO PARA O ASSISTENTE (BANCO DE DADOS E INFRA)

Ao sugerir QUALQUER alteração no banco de dados ou na infraestrutura, o
assistente DEVE:

JAMAIS incluir senhas reais em texto. Usar SEMPRE o placeholder <Senha>.

Garantir que toda nova tabela, coluna ou banco seja criado com:
CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci.

Lembrar que o XAMPP actual utiliza MariaDB, e a sintaxe de alguns comandos
difere do MySQL puro (ex.: SET PASSWORD FOR... em vez de ALTER USER).

Incluir o comando SQL exacto para qualquer modificação estrutural.

Recomendar backup ANTES de qualquer operação destrutiva (DROP, DELETE sem
WHERE).

Manter a compatibilidade com o schema do Canary 3.4.1 (tabelas accounts,
players, etc.) e com as consultas do Login Server.

Documentar novos campos ou tabelas para que o assistente saiba da sua
existência em interacções futuras.

A coluna created_at (INT, default 0) na tabela accounts é essencial para a
limpeza de contas guest. Não a remova ou altere sem atualizar o Login Server.

A coluna player_pronoun (TINYINT, default 0) na tabela accounts armazena o
pronome do JOGADOR (0 = não definido, 1 = jogador, 2 = jogadora). É utilizada
pelo Oráculo ao se dirigir ao dono da conta.

A coluna pronoun na tabela players é NATIVA do Canary e armazena o pronome
do PERSONAGEM (0 = usa o sexo, 2 = ela, 3 = ele, etc.). NÃO utilize storages
manuais para este fim.

A coluna lastlogin na tabela players EXISTE no schema actual (confirmado via
phpMyAdmin). Pode ser utilizada para lógica de inicialização/migração.

Ao criar scripts que manipulam itens (especialmente missões, tutorial e lojas),
o assistente DEVE utilizar a Lista de Itens Úteis como referência rápida de IDs
já traduzidos. O ficheiro items.xml permanece a fonte canónica, mas a lista
facilita a busca por itens comuns.

⚠️ NOVO (v2.1.1): As tabelas do Sistema de Progressão (dominios_def,
habilidades_def, player_dominios, player_habilidades) são parte integrante do
banco. Se ainda não existirem, devem ser criadas com os comandos fornecidos na
secção 3.4. A população inicial deve incluir a Runologia (ID 25).

⚠️ NOVO (v2.1.1): A coluna vocation em players DEVE ter valor padrão 0
(Aventureiro). O script de criação de personagens (Oráculo) já a define como 0,
mas o DBA deve garantir que a alteração na tabela foi aplicada:
ALTER TABLE players MODIFY vocation INT NOT NULL DEFAULT 0;

🗄️ 1. AMBIENTE: XAMPP (MARIADB)

O Projeto MCR utiliza o XAMPP como stack de desenvolvimento local, que fornece
o MariaDB (fork do MySQL) e o phpMyAdmin.

Instalação:

Versão: XAMPP 8.2.12 (ou superior compatível).

Local: D:\Programas\Xampp (ajustar conforme o ambiente).

Componentes instalados: MySQL e phpMyAdmin (Apache opcional para o MyAAC).

Configuração do MariaDB (my.ini):
Local do arquivo: D:\Programas\Xampp\mysql\bin\my.ini

Parâmetros obrigatórios:

```text
[client]
default-character-set = utf8mb4

[mysql]
default-character-set = utf8mb4

[mysqld]
character-set-server = utf8mb4
collation-server = utf8mb4_unicode_ci
skip-character-set-client-handshake
init-connect = 'SET NAMES utf8mb4 COLLATE utf8mb4_unicode_ci'
Essas configurações garantem que todas as conexões usem UTF‑8 completo, sem
depender de configuração manual em cada sessão.

Definição da Senha do Root (MariaDB):

sql
SET PASSWORD FOR 'root'@'localhost' = PASSWORD('<Senha>');
⚠️ DIFERENÇA CRÍTICA: O comando ALTER USER ... IDENTIFIED WITH
mysql_native_password NÃO funciona no MariaDB. Use SEMPRE SET PASSWORD FOR...
(Fase 33)

🛠️ 2. CRIAÇÃO DO BANCO DE DADOS

O banco de dados oficial do Projeto MCR se chama "BancoServer".

Comando de criação:

sql
CREATE DATABASE BancoServer
  CHARACTER SET utf8mb4
  COLLATE utf8mb4_unicode_ci;
Por que utf8mb4_unicode_ci e não utf8mb4_general_ci?

unicode_ci oferece suporte superior a caracteres acentuados do português
(ç, ã, õ), tratando 'c' e 'ç' como letras distintas e ordenando correctamente.

É o padrão recomendado para aplicações multilíngues e localizadas.

📜 3. SCHEMA DO BANCO DE DADOS

O schema principal está no arquivo schema.sql, fornecido com o Canary v3.4.1.
Local: ProjetoMCR/Canary/data/schema.sql

Principais tabelas e suas funções:

accounts: armazena contas de jogador. Colunas: id, name, password (hash SHA1),
email (vazio), created_at (INT), player_pronoun (TINYINT).

players: armazena personagens. Colunas: id, name, account_id, level, vocation,
sex, posx, posy, posz, town_id, pronoun (TINYINT), lastlogin (INT), etc.

É complementado por dezenas de outras tabelas para houses, guilds, store, daily
rewards, etc.

O Login Server acessa as mesmas tabelas accounts e players. Qualquer alteração
nessas tabelas DEVE ser sincronizada com o código do Login Server.

3.1 ADIÇÕES MCR AO SCHEMA ORIGINAL

accounts.created_at: Coluna INT (timestamp Unix). Usada para controlar a
criação de contas guest e permitir a sua remoção automática após 30 minutos de
inactividade. Contas permanentes podem ter o valor 0 ou o timestamp de
criação.

accounts.player_pronoun: Coluna TINYINT. Define como o Oráculo deve se
dirigir ao JOGADOR (dono da conta). Valores:
0 = não definido (perguntará novamente)
1 = jogador (tratamento masculino)
2 = jogadora (tratamento feminino)

players.pronoun: Coluna TINYINT nativa do Canary. Define o pronome do
PERSONAGEM (como o mundo deve se referir a ele). Valores:
0 = não definido (usa o sexo do personagem como fallback)
1 = they (neutro) – disponível, mas não exposto pelo Oráculo por enquanto
2 = she (ela)
3 = he (ele)
4 = ze
5 = usar o nome (name)
O Oráculo grava directamente 2 ou 3 durante a criação/gerenciamento.

players.lastlogin: Coluna INT (timestamp Unix). EXISTENTE no schema actual do
Canary 3.4.1 (confirmado no banco de dados do projecto). Armazena o timestamp
do último login do personagem. Pode ser utilizada por scripts de inicialização
(verificar se é o primeiro login) ou migração.

Rotina de limpeza de contas guest (implementada no Login Server):
Remove contas com prefixo guest_ e created_at anterior a 30 minutos, assim
como os respectivos personagens (Guest_*).
Comandos SQL executados periodicamente:

sql
DELETE FROM players WHERE name LIKE 'Guest\_%' AND account_id IN (
    SELECT id FROM accounts WHERE name LIKE 'guest\_%'
    AND created_at < UNIX_TIMESTAMP() - 1800
);
DELETE FROM accounts WHERE name LIKE 'guest\_%'
AND created_at < UNIX_TIMESTAMP() - 1800;
3.2 VERIFICAÇÃO DE COMPATIBILIDADE

O Canary 3.4.1 POSSUI a coluna "lastlogin" na tabela players (confirmado pela
estrutura actual do banco). O Login Server foi ajustado para lidar com essa
coluna. Se o schema for atualizado, verificar novamente.

O campo "email" na tabela accounts é mantido como string vazia, pois o MCR não
utiliza e‑mail para autenticação.

A coluna created_at deve existir na tabela accounts. Se estiver ausente,
adicione‑a com:

sql
ALTER TABLE accounts ADD COLUMN created_at INT NOT NULL DEFAULT 0;
A coluna player_pronoun deve ser adicionada manualmente (ainda não faz parte do
schema.sql original):

sql
ALTER TABLE accounts ADD COLUMN player_pronoun TINYINT NOT NULL DEFAULT 0;
A coluna pronoun já existe na tabela players (nativa do Canary). Não precisa ser
criada.

3.3 VOCAÇÃO PADRÃO (v2.1.1)

A partir do Sistema de Progressão v2.1.1, todo personagem é um Aventureiro
(vocation = 0). As vocações antigas (1‑5) não são mais usadas. Para garantir que
o banco reflicta essa mudança:

sql
ALTER TABLE players MODIFY vocation INT NOT NULL DEFAULT 0;
Se houver personagens antigos com vocation diferente de 0, o script de migração
(parte do evento_login.lua na pasta MCR/Dominio) irá convertê‑los
automaticamente.

3.4 TABELAS DO SISTEMA DE PROGRESSÃO (v2.1.1)

O Sistema de Progressão do Aventureiro utiliza quatro tabelas novas, que devem
ser criadas com charset utf8mb4.

sql
-- Tabela de definição de Domínios (Primários e Especializações)
CREATE TABLE dominios_def (
    id SMALLINT AUTO_INCREMENT PRIMARY KEY,
    nome VARCHAR(50) NOT NULL,
    descricao VARCHAR(255),
    icone VARCHAR(50),
    parent_id SMALLINT DEFAULT NULL,
    FOREIGN KEY (parent_id) REFERENCES dominios_def(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

-- Tabela de definição de Habilidades
CREATE TABLE habilidades_def (
    id INT AUTO_INCREMENT PRIMARY KEY,
    nome VARCHAR(50) NOT NULL,
    dominio_id SMALLINT NOT NULL,
    tipo ENUM('ativa','passiva') NOT NULL,
    requisito_afinidade INT DEFAULT 0,
    descricao TEXT,
    sinergia_dominio_id SMALLINT DEFAULT NULL,
    sinergia_afinidade INT DEFAULT NULL,
    FOREIGN KEY (dominio_id) REFERENCES dominios_def(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

-- Tabela de progressão do jogador por Domínio
CREATE TABLE player_dominios (
    player_id INT NOT NULL,
    dominio_id SMALLINT NOT NULL,
    afinidade INT DEFAULT 0,
    despertado TINYINT DEFAULT 0,
    PRIMARY KEY (player_id, dominio_id),
    FOREIGN KEY (player_id) REFERENCES players(id) ON DELETE CASCADE,
    FOREIGN KEY (dominio_id) REFERENCES dominios_def(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

-- Tabela de habilidades aprendidas pelo jogador
CREATE TABLE player_habilidades (
    player_id INT NOT NULL,
    habilidade_id INT NOT NULL,
    PRIMARY KEY (player_id, habilidade_id),
    FOREIGN KEY (player_id) REFERENCES players(id) ON DELETE CASCADE,
    FOREIGN KEY (habilidade_id) REFERENCES habilidades_def(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

3.5 ESTRUTURA REVSCRIPT (INFORMAÇÃO RELACIONADA)

Embora não afecte directamente o banco de dados, os scripts Lua que consomem
essas tabelas residem em data-canary/scripts/MCR/, organizados em subpastas
como Dominio, Criacao, Geral. O motor Revscript carrega automaticamente todos
os ficheiros .lua nessas pastas, portanto, qualquer script que manipule os
dados de progressão (ex.: evento_login.lua, config_dominios.lua) deve ser
colocado na subpasta correspondente.

🔐 4. phpMyAdmin

URL: http://localhost/phpmyadmin (se o Apache estiver rodando).

Configuração do acesso (config.inc.php):
Local: D:\Programas\Xampp\phpMyAdmin\config.inc.php

Adicionar/alterar as linhas:

php
$cfg['Servers'][$i]['auth_type'] = 'config';
$cfg['Servers'][$i]['user'] = 'root';
$cfg['Servers'][$i]['password'] = '<Senha>';
⚠️ Se auth_type = 'config' e a senha NÃO for configurada, o phpMyAdmin tentará
conectar sem senha e falhará com "using password: NO". (Fase 33)

💾 5. BACKUP E RESTAURAÇÃO

Backup (exportar):

text
mysqldump -u root -p --single-transaction --routines --triggers --events
  --default-character-set=utf8mb4 BancoServer > backup_BancoServer_YYYYMMDD.sql
Restauração (importar):

text
mysql -u root -p --default-character-set=utf8mb4 BancoServer
  < backup_BancoServer_YYYYMMDD.sql
Procedimento recomendado:

Fazer backup ANTES de qualquer alteração no schema.

Manter backups datados em local seguro.

Testar a restauração em um banco secundário antes de aplicar em produção.

🔗 6. INTEGRAÇÃO COM CANARY E LOGIN SERVER

O Canary e o Login Server compartilham o mesmo banco de dados (BancoServer).

Configuração no Canary (config.lua):

lua
mysqlHost = "127.0.0.1"
mysqlUser = "root"
mysqlPass = "<Senha>"
mysqlDatabase = "BancoServer"
mysqlPort = 3306
mysqlInitCommand = "SET NAMES 'utf8mb4'"
Configuração no Login Server (.env):

text
MYSQL_DBNAME=BancoServer
MYSQL_HOST=127.0.0.1
MYSQL_PORT=3306
MYSQL_USER=root
MYSQL_PASS=<Senha>
Ambos os sistemas precisam estar rodando para o fluxo completo de autenticação e
jogo funcionar.

📚 7. RECURSOS COMPLEMENTARES: LISTA DE ITENS ÚTEIS (NOVA SECÇÃO)

Além do schema SQL e do ficheiro items.xml, o projecto dispõe de uma Lista de
Itens Úteis ([Aquivo Complementar] Lista de Items Uteis.txt) que contém mais de
3000 itens com os seus IDs, nomes traduzidos e atributos principais. Este
recurso é especialmente valioso para:

Criar rapidamente missões e o tutorial sem necessidade de abrir o XML completo.

Evitar erros de ID ao referenciar itens em scripts Lua.

Fornecer ao assistente um acesso rápido aos itens mais comuns (equipamentos,
poções, runas, ferramentas, etc.).

Como utilizar a lista:

Procure pelo nome do item em português (ex.: "Poção de Vida") e obtenha o ID
(266).

Use esse ID nos comandos player:addItem(266, 1).

Confirme sempre o ID no items.xml antes de implantar em produção.

A lista é um complemento, não substitui o ficheiro XML oficial. Em caso de
divergência, o XML prevalece. Atualizações futuras ao items.xml devem ser
reflectidas na lista.

⚠️ ARMADILHAS E LIÇÕES APRENDIDAS

MariaDB vs MySQL: O XAMPP actual usa MariaDB. O comando ALTER USER não
funciona. Use SET PASSWORD FOR 'root'@'localhost' = PASSWORD('senha');
(Fase 33)

Acesso negado (Access denied): Senha incorrecta no config.lua ou .env.
Verificar se a senha foi definida no MariaDB e se os serviços estão rodando.
(Fase 33)

Erro 10061 (Can't connect): O serviço MySQL/MariaDB não está rodando. Iniciar
pelo XAMPP Control Panel. (Fase 33)

Coluna lastlogin: O schema do Canary 3.4.1 POSSUI a coluna lastlogin na
tabela players (confirmado na estrutura actual do banco). Pode ser usada
normalmente em scripts Lua e consultas SQL.

Charset incorrecto: Dados salvos com charset errado (latin1) podem corromper
caracteres acentuados. Sempre force utf8mb4 em todas as camadas. (Fase 32)

phpMyAdmin sem senha configurada: Mesmo com senha no MySQL, o phpMyAdmin
precisa da senha no config.inc.php. (Fase 33)

Personagens de teste: Para testar o fluxo de criação de personagem, é útil
criar personagens manualmente via SQL:

sql
INSERT INTO players (name, account_id, level, vocation, sex, posx, posy,
                     posz, town_id, ...) VALUES (...);
Coluna created_at: Necessária para expiração de contas guest. Deve ser
adicionada manualmente caso o schema não a possua.

Coluna player_pronoun: Armazena o pronome do JOGADOR (dono da conta). Deve ser
adicionada com ALTER TABLE accounts ADD COLUMN player_pronoun TINYINT NOT NULL
DEFAULT 0;

Coluna pronoun: NATIVA do Canary. Armazena o pronome do PERSONAGEM. NÃO
utilize o storage 100001 para essa finalidade.

Conversão de loots e addItem: Scripts como 0-converter-loot.py e
0-converter-additem.py foram criados para migrar referências de itens de nome
para ID, garantindo consistência com o items.xml traduzido.

⚠️ Vocação 0 (v2.1.1): Qualquer personagem criado manualmente ou via ferramenta
