📘 Documentação Técnica do Motor SPA – Projeto MCR (v5.8 Revisada)
Última atualização: 13 de Junho de 2026
Versão do documento: 5.8 (revisada – integração completa do Sistema de Perseguição Multi‑Piso v3.2.0)

Este documento descreve todas as APIs, métodos, constantes, estruturas de dados e sistemas que o servidor Canary expõe ao Sistema de Progressão do Aventureiro (SPA). Inclui também a integração completa com o Sistema de Perseguição Multi‑Piso v3.2.0, o Mapa Cognitivo Global (GlobalMonsterMap) e o SPAManager (gestão de eventos e pets). Todas as funcionalidades aqui listadas estão implementadas, compiladas e verificadas através do comando !testesistema.

Nota sobre codificação do código C++ do servidor: Strings no código‑fonte Canary podem usar UTF‑8 literal (ex.: "Não", "Você") desde que o compilador tenha a flag `/utf-8` ativa e a string passe por `toLatin1()` antes de ser enviada ao protocolo. Escapes octais Latin‑1 (ex.: `\343` para `ã`) continuam a funcionar como alternativa segura, mas não são obrigatórios. Consulte o Guia de Compilação do Servidor e o DevLog/Sistema de Codificação.md para mais detalhes.

1.3 Curva de Progressão
Definida em constantes.lua na tabela NIVEL_PONTOS. Os valores actuais (para uma taxa de ≈1800 pontos/hora) são:

lua
NIVEL_PONTOS = {
    50, 120, 300, 740, 1800, 2500, 3450, 4750, 6550, 9000,
    12900, 18400, 26300, 37600, 54000, 69000, 87000, 110000, 140000, 180000
}
A função global getNivelPorAfinidade(acumulado) devolve o nível (1 a 20) correspondente ao total de pontos. Nível 0 significa nenhuma afinidade; nível 20 é a Maestria.

A função getNivelEfetivo(player, dominioId) retorna o nível real, bloqueando a Maestria (nível 20) enquanto houver domínios filhos que não tenham atingido a Maestria. Esta regra garante que um domínio pai só se torna Mestre quando todos os seus filhos já o são.

1.4 Hierarquia de Domínios
A tabela DOMAIN_TREE em constantes.lua define as relações de parentesco entre Primários, Secundários e Especialidades. Os IDs actualmente implementados são:

Primários: Combate (1), Magia (2), Ofícios (3), Natureza (4).

Secundários de Combate: Lâminas (10), Machados (11), Clavas (12), Precisão (13), Artes Marciais (14), Escudo (15).

Secundários de Magia: Elementos (20), Espectro (21), Runologia (22).

Especialidades de Arma: Espadas Leves (100), Espadas Pesadas (101), Machados Leves (110), Machados Pesados (111), Clavas Leves (112), Clavas Pesadas (113), Arcos (120), Arremesso (121), Lutador (130), Armas de Punho (132), Bastões Arcanos (133), Arremesso de Escudo (140).

Especialidades Mágicas: Fogo (23), Água e Gelo (24), Terra e Veneno (25), Energia (26), Sagrado e Morte (200), Wands (210), Rods (211), Runas (212).

Outros: Sobrevivência (400) – ramo de Natureza (4). Summon (28) – ramo de Magia (2).

Propagação de afinidade (rácio 4:2:1): quando um jogador ganha afinidade numa Especialidade, a função adicionarAfinidade em progressao.lua acumula os pontos no buffer. A propagação real ocorre no descarregamento do buffer (a cada 2 segundos), onde os totais acumulados são distribuídos:

100% dos pontos para a própria Especialidade.

50% (metade) para o Domínio Secundário correspondente.

25% (um quarto) para o Domínio Primário correspondente.

Os valores são arredondados para baixo (math.floor). A propagação usa os totais do ciclo, garantindo que pontos baixos sejam acumulados antes da divisão.

A tabela domainChildren (construída em constantes.lua) mapeia cada domínio para os seus filhos directos, sendo usada por getNivelEfetivo para a regra de Maestria condicionada.

1.5 Compatibilidade Elemental
Desde a v4.2, o SPA suporta sinergia elemental nativa: armas que possuem elementalBond definido no XML (ou elementType nas habilidades) desbloqueiam automaticamente as habilidades do domínio elemental correspondente.

Exemplo: uma Espada de Fogo (dano de fogo) activa tanto as habilidades de Espadas Leves (100) como as de Fogo (23). Uma Wand de Energia desbloqueia Wands (210) e Energia (26).

Isto é gerido pela função armaCompativeis no executor_habilidades.lua, que verifica o elemento da arma através de uma cadeia de detecção robusta (item:getElementalBond(), item:getElementType(), item:getCombatType(), getShootType), mapeando para o domínio usando a tabela damageDomains. A função getElementoItem em util_armas.lua centraliza esta detecção, usando pcall em todas as chamadas e mantendo uma tabela de fallback manual.

lua
damageDomains = {
    [COMBAT_FIREDAMAGE]   = 23,   -- Fogo
    [COMBAT_ICEDAMAGE]    = 24,   -- Água e Gelo
    [COMBAT_EARTHDAMAGE]  = 25,   -- Terra e Veneno
    [COMBAT_ENERGYDAMAGE] = 26,   -- Energia
    [COMBAT_DEATHDAMAGE]  = 200,  -- Sagrado e Morte
    [COMBAT_HOLYDAMAGE]   = 200,  -- Sagrado e Morte
}
1.6 Campos Elementais (Fields)
Desde a v4.3, habilidades de Maestria elemental podem criar campos elementais no chão. O conjurador é sempre imune aos seus próprios campos. Com a transformação de Maestria adequada (nível 20), o grupo também fica imune. A intensidade do campo (dano por tick, duração) escala com o nível de afinidade do conjurador.

A função criarField() em funcoes_habilidade.lua (v2.0.2) gere a criação de campos, incluindo o loop de ticks, a imunidade do conjurador, a imunidade do grupo (se Maestria) e uma animação de ignição (onda que converge para o centro). O campo é implementado como um item de MagicField (ID base 1497 + elemento) com atributos de dano e duração. O dano por tick é recalculado a cada intervalo (normalmente 1000 ms) e aplicado a todas as criaturas dentro do campo, excepto o conjurador e, opcionalmente, seu grupo.

🎯 4. Habilidades (Skills do SPA)
O SPA utiliza uma arquitectura modular em três camadas:

Motor de Habilidades (motor_habilidades.lua v2.0) — lógica central de execução, com 27 tipos de efeitos padronizados.

Definições de Habilidades (habilidades_*.lua) — ficheiros de dados que descrevem as habilidades usando parâmetros padronizados (campo efeitoConfig para gatilhos; campo efeito vazio para passivas de vida/mana/velocidade). Zero lógica manual.

Narrativa (narrativa_habilidades.lua v1.0) — frases de feedback organizadas por tipo de efeito, postura e domínio.

4.1 Motor Modular de Habilidades (motor_habilidades.lua v2.0)
O motor central processa qualquer habilidade definida por parâmetros. Cada efeito possível está mapeado na tabela MOTOR.efeitos. O executor chama MOTOR.executar(player, target, damage, habDef), que interpreta habDef.efeitoConfig.tipo e dispara a função correspondente.

Novas funcionalidades da v2.0:

Fallback automático de efeitos visuais: se uma habilidade não especificar magicEffect, o motor obtém o efeito adequado a partir do elemento da arma equipada (tabela EFEITOS_ELEMENTAIS).

Suporte a conditionMagicEffect: as habilidades que aplicam condições podem especificar um efeito visual separado.

Função auxiliar aplicarDanoCircular: unifica a lógica de dano em área circular com onda expansiva.

Adaptação elemental para Wands e Rods: as habilidades de Wands (210) e Rods (211) têm o seu efeito visual e condição automaticamente adaptados.

Novos tipos de efeitos modulares: sinergia_escalonada, storm, orbit, rain, pulse, trap, summon.

Controlo de efeitos temporais: limite de 2 efeitos temporais (storm, orbit, rain, pulse, trap, field) activos por jogador.

4.1.2 Controlo de Efeitos Temporais
Os tipos storm, orbit, rain, pulse, trap e field são considerados efeitos temporais. Para evitar sobrecarga de addEvents, o motor limita a 2 efeitos temporais activos por jogador. Se um novo efeito for solicitado e o limite estiver atingido, o efeito não é criado, a habilidade entra em cooldown normalmente, e o jogador é informado. O limite é gerido pelas funções canStartTempEffect, registerTempEffect e unregisterTempEffect no motor.

4.2 Narrativa Centralizada (narrativa_habilidades.lua v1.0)
A tabela Narrativa.frases contém frases organizadas por tipo de efeito e postura. A função Narrativa.gerar(player, target, habDef) compõe a frase final com cores, substituindo placeholders:

{alvo} → nome do alvo com COR.INIMIGO_MONSTRO

{condicao} → nome da condição com a cor correspondente (COR.COND_*)

{elemento} → nome do elemento com a cor correspondente (COR.ELEM_*)

O motor chama Narrativa.gerar() automaticamente, garantindo feedback consistente em todas as habilidades.

4.8 Gestão de Atributos Base (Vida, Mana, Velocidade)
Passivas que alteram vida, mana ou velocidade de movimento não devem modificar directamente esses valores nos seus campos efeito. Em vez disso, são geridas centralizadamente por funções dedicadas:

recalcularVidaMana(player) (em passivas.lua): usa player:getVocationBaseMaxHealth() e getVocationBaseMaxMana() para obter a base da vocação e adiciona bónus percentuais com base nos níveis dos domínios relevantes. Aplica setMaxHealth/setMaxMana uma única vez.

recalcularVelocidade(player) (em passivas.lua): usa player:getNaturalSpeed() (base + nível) e adiciona bónus percentuais de habilidades de movimento. Aplica setMovementSpeed.

4.11 Execução (executor_habilidades.lua v9.0)
O executor utiliza um motor de selecção inteligente com prioridade contextual, cooldowns individuais por domínio, e suporta:

Cooldown global por domínio: cada domínio tem o seu próprio cooldown, baseado na hierarquia (especialidade: 6s, secundário: 8s, primário: 10s), reduzido pelo nível de afinidade (2% por nível, máx. 40%) e pela barra de foco (até 30%).

Limite de habilidades por ciclo dinâmico: 1/ciclo (nível 0–4), 2 (5–14), 3 (15+).

Prioridade por especialização: bónus de +1 a cada 5 níveis de afinidade no domínio.

Disponibilidade por hierarquia: habilidades de secundários e primários só são elegíveis quando a barra de foco está em 100% (Vínculo/Maestria). Especialidades estão sempre disponíveis.

Limites de categoria: o executor impõe limites por categoria (ex.: "aoe" = 2, "single" = 3, "finisher" = 1) numa janela de 10 segundos. Habilidades que excedem o limite não são seleccionadas, não gastam cooldown, e ficam disponíveis no próximo ciclo.

Throttle global: intervalo mínimo de 600ms entre qualquer execução de habilidade (MIN_GLOBAL_DELAY = 0.6).

Selecção optimizada: a selecção de habilidades percorre apenas as primeiras 5 habilidades elegíveis (MAX_CANDIDATOS = 5), com fallback de categoria para "single" caso a habilidade não defina categoria.

Delegação ao motor modular: chama MOTOR.executar() e Narrativa.gerar(). O sistema legado de buffType foi totalmente removido.

Sistema de combos: para o gatilho "onAttack", o executor tenta encadear até 3 habilidades (com chances configuráveis por nível), com 250ms e 500ms de intervalo entre elas.

8.6 Sistema de Perseguição Multi‑Piso (v3.2.0)
8.6.1 Visão Geral
O sistema de perseguição multi‑piso permite que summons/pets e criaturas hostis sigam o seu líder através de diferentes pisos (eixo Z), utilizando escadas, buracos, teleportes e rampas. A versão actual (3.2.0) é uma reescrita completa face à versão 8.0 original, abandonando o BFS cognitivo completo em favor de uma abordagem de A* segmentado com escadas, complementada por atalhos multi‑piso e rotas alternativas com waypoints cognitivos.

Princípios fundamentais:

A* nativo do motor para navegação 2D de longa distância (maxSearchDist generoso de 500 tiles).

Escadas como pontos de ligação entre segmentos A* em pisos diferentes.

Mapa cognitivo global (GlobalMonsterMap) como cache de transições e suporte a rotas alternativas.

Avaliação de custo total para escolha de escadas, considerando o percurso completo até ao líder.

Atalho multi‑piso: quando o líder está no mesmo piso mas inacessível, o monstro pode usar escadas para contornar o obstáculo.

Rotas alternativas com waypoints: quando o A* falha no mesmo piso, o monstro usa o grafo cognitivo para encontrar um caminho alternativo.

Seguimento imediato da escada do líder: se o jogador usou uma escada e o monstro está a ≤ 3 tiles da sua origem, o monstro segue‑o imediatamente.

Summons nunca desistem: continuam a perseguição mesmo após perder o mestre de vista, sem nunca tentar voltar a um ponto de spawn.

Destravamento silencioso: se uma summon ficar completamente bloqueada, um teleporte invisível a recoloca em movimento.

Monstros normais voltam ao seu ponto de origem (spawn) caminhando, utilizando as escadas que percorreram.

8.6.2 Arquitectura da Máquina de Estados (Monster::onThinkMCR)
A máquina de estados é executada integralmente dentro do ciclo onThinkMCR, com apenas 5 estados processados por ciclo, pela seguinte ordem de prioridade:

text
┌──────────────────────────────────────────────┐
│ 1. COMBATE (prioridade máxima)                │
│    Se pode atacar → cancela tudo, ataca       │
├──────────────────────────────────────────────┤
│ 2. PERSEGUIÇÃO MULTI‑PISO (m_pursuingStairs)  │
│    Tenta alcançar m_stairOrigin               │
│    Gere travamentos, tenta movimento forçado   │
│    Monitoriza progresso, teleporta para escada │
├──────────────────────────────────────────────┤
│ 3. PLANEAMENTO DE ESCADA                      │
│    Se líder noutro piso → planStairTrajectory │
│    Se líder no mesmo piso e inacessível →     │
│      Avalia atalho multi‑piso vs rota alt.    │
├──────────────────────────────────────────────┤
│ 4. RETORNO AO SPAWN / CASA                   │
│    (Apenas monstros normais)                  │
│    processarRetornoSpawn / processarRetornoSimples │
│    Summons NUNCA entram neste estado           │
├──────────────────────────────────────────────┤
│ 5. IDLE                                       │
│    Se sem alvo → tenta follow, deambula ou    │
│    retorna ao spawn (só normais)              │
│    Inclui destravamento silencioso para summons│
└──────────────────────────────────────────────┘
Regra de ouro: O BFS do mapa cognitivo nunca toma o controlo primário. É sempre uma ferramenta auxiliar para planear uma rota até um alvo.

Bloco 3 em detalhe (Planeamento de Escada)
O Bloco 3 é executado quando o monstro não está em perseguição ativa nem em retorno, e o líder existe. A lógica exacta é:

Se o líder está num piso diferente (leaderPos.z != myPos.z):

Chama planStairTrajectory (throttle 2500 ms via m_lastStairCheck).

Se o líder está no mesmo piso:

Verifica se o A* nativo encontra um caminho até ao líder (leaderAcessivel).

Se acessível, não faz nada (o Bloco 1 ou 5 tratará do follow).

Se inacessível:

Calcula multiFloorCost = evaluateMultiFloorCost(leaderPos).

Calcula alternativeCost = findNearestApproachTo(myPos, leaderPos).size().

Se multiFloorCost < alternativeCost e multiFloorCost < INT_MAX:

Inicia planStairTrajectory imediatamente (atalho multi‑piso).

Atualiza m_lastStairCheck = now.

Senão, se alternativeCost < INT_MAX:

Tenta tryAlternativeRouteToLeader (throttle 5000 ms).

Se a rota alternativa já estava ativa (m_followingWaypoints) e m_waypointStuckCycles >= 3:

Força o atalho: clearWaypoints() seguido de planStairTrajectory.

8.6.4 Funções Centrais
Monster::onThinkMCR() – Função principal chamada a cada ciclo de pensamento (1 Hz). Implementa a máquina de 5 estados descrita em 8.6.2.

Monster::planStairTrajectory(const Position& leaderPos) – Planeia a melhor escada para alcançar o líder. Fluxo:

Regra de seguimento imediato da escada do líder: percorre as transições do jogador; se alguma tiver origem no piso do monstro, destino no piso do líder, distância ≤ 3 tiles e for válida → perseguição imediata.

Rota global completa: tenta GlobalMonsterMap::findPath. Se encontrar, extrai a primeira transição de piso e inicia perseguição.

Fallback heurístico: recolhe escadas candidatas (grafo + líder + scan real), avalia cada uma com A* até à origem + estimativa da distância do destino ao líder (via findNearestApproachTo). Bónus de -1 para escadas do líder. Escolhe a de menor custo total.

Monster::evaluateMultiFloorCost(const Position& leaderPos) – Versão "seca" do planStairTrajectory que apenas calcula o custo da melhor escada, sem iniciar perseguição. Usada no Bloco 3 para comparar atalho multi‑piso com rota alternativa. Fluxo interno:

Obtém o líder (com fallback para getTargetList e jogadores no mesmo piso).

Recolhe candidatas: transições do grafo, transições do líder e, se vazio, findStairNearReal.

Para cada candidata:

Verifica direção, m_recentlyFailedStairs, canStairBeUsed.

Calcula A* até à origem.

Estima distância do destino ao líder: tenta findNearestApproachTo; se falhar, faz scanPlayerTrail(dest, 8) e tenta de novo.

Custo = |A*| + 1 + distFromDest, com bónus de -1 para escadas do líder.

Devolve o menor custo encontrado, ou INT_MAX se nenhuma escada for viável.

Monster::walkToWaypoint(const Position& target) – Inicia a caminhada para um waypoint. Se o movimento forçado falhar, invalida o tile e incrementa m_waypointIndex para o próximo ponto da rota (ou chama clearWaypoints se for o último). Também tenta tiles adjacentes e fallback para BFS.

Monster::tryAlternativeRouteToLeader() – Encontra uma rota alternativa no mesmo piso usando o mapa cognitivo (findNearestApproachTo).

Monster::clearStairPursuit() – Limpa o estado de perseguição, restaura o follow original e faz reset dos flags. Importante: Não zera m_pursuitDeadline, permitindo que a perseguição continue enquanto o líder estiver visível. Também faz reset de m_multiFloorShortcut.

Monster::clearWaypoints() – Limpa a lista de waypoints e faz reset dos contadores associados, incluindo m_waypointStuckCycles e m_lastDistToLeaderWaypoints.

Monster::canStairBeUsed(const Position& origin, int deltaZ) – Valida se uma escada pode ser usada na direção pretendida. Algoritmo:

Obtém o tile em origin. Se não tiver TILESTATE_FLOORCHANGE, retorna false.

Se o tile tem destino explícito (getDestination()), compara o deltaZ.

Se não, verifica sub‑flags: FLOORCHANGE_DOWN → sempre desce (Z aumenta); FLOORCHANGE_NORTH/SOUTH/EAST/WEST e variantes → sempre sobe (Z diminui).

Se não tem sub‑flags, é bidirecional (buracos, etc.) → retorna true.

Se tem sub‑flags, valida conforme deltaZ: < 0 requer flags de subida, > 0 requer FLOORCHANGE_DOWN.

Monster::findStairNearReal(const Position& leaderPos, int radius) – Procura escadas reais no mapa ao redor do monstro (scan no mesmo piso, raio padrão 15 tiles). Itera todos os tiles no raio, determina deltaZ com base nas sub‑flags, valida com canStairBeUsed, e escolhe a de menor custo (distância Manhattan até à escada + distância Manhattan do destino ao líder).

8.6.4.1 Funções de Retorno (apenas Hostis)
processarRetornoSimples() – Usado quando o monstro está no mesmo piso que m_homePosition. Tenta A* direto até casa. Se falhar, usa waypoints do mapa cognitivo. Timeout de 15 segundos: se exceder, força teleporte para casa. Após 10 falhas totais, desiste e deambula.

processarRetornoSpawn() – Usado quando o monstro tem um caminho de escadas (m_returnPath) para voltar. Percorre m_returnPath no sentido inverso. Para cada transição, tenta caminhar até à origem da escada e usá‑la. Timeout de caminhada de 5s por etapa. Após 15 falhas totais, reinicia o percurso.

8.6.5 Mecanismos de Decisão
Custo Total da Rota:

text
custo = distância A* até à escada + 1 (transição) + distância estimada do destino ao líder
A distância do destino ao líder é obtida via findNearestApproachTo. Se o grafo não tiver dados suficientes, é feito um mini‑scan (scanPlayerTrail) ao redor do destino antes de reavaliar.

Atalho Multi‑Piso vs Rota Alternativa (Bloco 3): Ver detalhes na secção 8.6.2 (Bloco 3 em detalhe).

Seguimento Imediato da Escada do Líder: No início do planStairTrajectory, se existir uma transição do jogador que cumpra os critérios (origem no piso do monstro, destino no piso do líder, distância ≤ 3 tiles, válida), o monstro inicia a perseguição por essa escada imediatamente, sem avaliar outras opções.

Anti‑Ping‑Pong: O flag m_multiFloorShortcut é ativado no planStairTrajectory quando leaderPos.z == myPos.z no momento do planeamento. Isto protege a perseguição de ser cancelada quando o líder "volta" ao mesmo piso (pois o monstro está deliberadamente a contornar um obstáculo). Escadas falhadas têm cooldown de 10 segundos em m_recentlyFailedStairs.

8.6.6 Tratamento de Summons vs Hostis
Funcionalidade	Summons	Hostis
Perda de líder (5 ticks)	Continua perseguição normal (nunca desiste)	Inicia retorno ao spawn
Deadline expirado	Continua perseguição normal	Inicia retorno ao spawn
Retorno ao spawn	Nunca – continua perseguição	Sim, caminha de volta
Destravamento silencioso	Sim, se presa ≥ 3 ciclos (teleporte invisível para waypoint)	Não
isInSpawnRange	Retorna sempre true	Verifica distância real ao spawn
Porque as summons nunca retornam ao spawn: Nos blocos de perda de líder (m_sightLostTicks >= maxSightLost e now > m_pursuitDeadline), o código para summons simplesmente limpa a perseguição ativa (clearStairPursuit), reseta m_sightLostTicks = 0 e retorna sem definir m_returningHome. No ciclo seguinte, a summon volta ao planeamento normal (Blocos 3 e 5).

Destravamento Silencioso (apenas summons): Se uma summon estiver a seguir waypoints e m_waypointStuckCycles >= 3, o Bloco 5 (IDLE) executa:

Obtém um caminho do findNearestApproachTo entre a posição atual e o líder.

Se o caminho não estiver vazio:

Teleporta silenciosamente para o primeiro tile (g_game().internalTeleport).

Limpa os waypoints antigos.

Define os novos waypoints como o caminho obtido, começando no índice 1.

Reseta m_waypointStuckCycles = 0.

Se o caminho estiver vazio, limpa os waypoints e tentará de novo no próximo ciclo.

O teleporte não tem efeitos visuais e mantém a imersão.

8.6.7 Integração com o Motor (Tile::queryAdd)
Foram corrigidas as verificações de TILESTATE_FLOORCHANGE e TILESTATE_TELEPORT no método Tile::queryAdd:

Antes (errado): hasFlag(TILESTATE_FLOORCHANGE | TILESTATE_TELEPORT) – exigia ambos os bits simultaneamente.

Depois (corrigido): hasFlag(TILESTATE_FLOORCHANGE) || hasFlag(TILESTATE_TELEPORT).

Comportamento durante pathfinding (FLAG_PATHFINDING): O A* só considera um tile de escada/teleporte como destino se o monstro estiver em perseguição (isPursuingStairs()) e for a escada planeada (getStairOrigin() == getPosition()).

Comportamento durante movimento real (sem FLAG_PATHFINDING): Se o monstro está em perseguição e o tile é exatamente a escada planeada, a entrada é permitida. Caso contrário, tiles com TILESTATE_FLOORCHANGE ou TILESTATE_TELEPORT são bloqueados.

8.6.8 Sistema de Engajamento e Deadline
Barra de Engajamento (m_engagement, 0–100): sobe +2 ao causar/receber dano, +1/s com líder visível; desce -3/s sem ver o líder.

Cálculo do Deadline (computePursuitDeadline):

cpp
uint64_t Monster::computePursuitDeadline() {
    int baseTimeMs = 30000;                         // 30 segundos base
    int engagementTimeMs = m_engagement * 500;      // até 50s adicionais
    int totalMs = std::min(60000, baseTimeMs + engagementTimeMs); // máx. 60s
    return getCurrentTimeMs() + totalMs;
}
O deadline é renovado enquanto o líder está visível.

8.6.11 Comportamento Esperado dos Monstros
Perseguição multi‑piso: segue o jogador através de escadas usando a fila de transições e o grafo cognitivo.

Múltiplas mudanças de piso: pode planear rotas que atravessam vários andares para cercar o jogador.

Uso de escadas nunca pisadas pelo jogador: escolhe qualquer escada do grafo cognitivo que minimize o custo total.

Atalho multi‑piso: se o caminho direto está bloqueado no mesmo piso, usa escadas para contornar o obstáculo, desde que o custo total seja menor.

Rota alternativa: quando o A* falha, usa o mapa cognitivo para encontrar um caminho alternativo no mesmo piso.

Seguimento imediato: se o jogador usa uma escada e o monstro está a ≤ 3 tiles, segue‑o instantaneamente.

Reação a obstáculos: tiles bloqueados são invalidados no grafo; o monstro recalcula a rota.

Summons nunca desistem: continuam a perseguição mesmo após perder o mestre de vista, com destravamento silencioso em último caso.

Hostis retornam ao spawn: caminham de volta usando escadas; se o líder reaparecer a ≤ 10 tiles, cancelam o retorno.

Anti‑ping‑pong: não ficam presos a subir e descer a mesma escada repetidamente.
