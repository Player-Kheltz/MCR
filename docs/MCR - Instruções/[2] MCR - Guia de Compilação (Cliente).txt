🎯 OBJETIVO DESTE GUIA
Fornecer todas as instruções, dependências, customizações e verificações necessárias para compilar o OTClient Redemption do Projeto MCR a partir do código‑fonte. Este guia também estabelece as regras que o assistente DEVE seguir ao auxiliar em tarefas relacionadas à compilação, modificação de arquivos C++ do cliente ou ajustes nos módulos Lua que dependem de alterações no motor.

Esta versão atualiza o cliente para a versão 2.1.0 do Sistema de Progressão do Aventureiro: o HUD de Estados de Alma (opcodes 182/183), a janela de conhecimento (opcode 184) e a confirmação de que as Posturas de Combate (Ímpeto, Equilíbrio, Guarda) reutilizam os modos de luta nativos (Full Attack, Balanced, Full Defense) sem necessidade de novas interfaces.

🚧 ESTADO ATUAL DA IMPLEMENTAÇÃO (FASE 4 – CLIENTE)

De acordo com o [Plano de Ação] Consolidado, a **Fase 4 (Cliente) ainda não foi iniciada**. Isto significa que:
- Os listeners para os opcodes 182 (ícone de Vínculo), 183 (partículas de Estado de Alma) e 184 (janela de conhecimento) **ainda não foram implementados** nos módulos Lua do OTClient.
- As Posturas de Combate reutilizam os modos de luta nativos (Full Attack, Balanced, Full Defense) que já funcionam no cliente, portanto nenhuma modificação é necessária para que o servidor receba o fightMode correcto. Os efeitos visuais e de jogabilidade dependem dos scripts do servidor (e, por sua vez, dos métodos C++ da Fase 2).
- O opcode 200 (migração de conta) já está implementado e funcional.
- As customizações de máscara de Alma, cache portátil, tradução da interface e conversão de nome (UTF‑8 para Latin‑1) estão aplicadas e em funcionamento.
- Qualquer novo módulo Lua para interface deve ser salvo como ISO‑8859‑1.
- As directrizes de codificação C++ (escapes octais para strings, comentários em PT‑BR) aplicam-se integralmente a qualquer modificação no código‑fonte do OTClient.

⚠️ REGRAS DE REVISÃO PARA O ASSISTENTE (C++ CLIENTE / OTCLIENT)

Ao modificar QUALQUER arquivo .cpp/.h no diretório OTClient/src/, o assistente DEVE:

- Garantir que TODAS as strings de interface (menus, mensagens de toast, labels) estejam em PT‑BR, com acentuação correcta. Utilizar OBRIGATORIAMENTE escapes octais Latin‑1 (ex.: "Configura\347\343o") para garantir compatibilidade entre compiladores e sistemas. NÃO usar UTF‑8 diretamente nas strings C++.
- Manter comentários em PT‑BR; nomes de funções e variáveis em inglês (camelCase).
- Verificar se a alteração preserva a compatibilidade com o servidor Canary (protocolo 15.00).
- Se modificar a comunicação de rede (HTTP, ProtocolGame, Extended Opcodes), garantir que as portas usadas são 8080 (Login Server) e 7173 (Game), conforme documentado.
- Toda nova interface gráfica em C++ deve utilizar os recursos existentes do motor (g_ui, displays) e ser documentada para integração Lua.
- Após a edição, listar explicitamente os arquivos modificados para compilação selectiva.

🔨 PROCEDIMENTO DE COMPILAÇÃO

Clone o repositório MCR (já contém os fontes customizados):
git clone https://github.com/Player-Kheltz/ProjetoMCR.git
cd ProjetoMCR\OTClient

Configure o CMake (substitua o caminho do vcpkg):
mkdir build
cd build
cmake .. -DCMAKE_TOOLCHAIN_FILE=C:/vcpkg/scripts/buildsystems/vcpkg.cmake -DVCPKG_TARGET_TRIPLET=x64-windows

Compile:
cmake --build . --config Release

O executável será gerado em build\Release\otclient.exe.

Copie o otclient.exe para a raiz do diretório do OTClient (onde está o init.lua e as pastas data/, modules/). O cliente NÃO funciona sem os arquivos de configuração. Uma alternativa é executar o cliente a partir do diretório build com os caminhos ajustados, mas a prática recomendada é manter o exe na raiz junto com data/.

🎮 REUTILIZAÇÃO DOS MODOS DE LUTA NATIVOS PARA POSTURAS (v2.1.1)

O cliente já possui nativamente os modos de luta Full Attack, Balanced e Full Defense, acessíveis por hotkeys (Ctrl+A, Ctrl+B, Ctrl+D) e ícones no painel de combate. Para o Sistema de Progressão v2.1.1, estes mesmos modos representam as Posturas de Combate:

- Full Attack → Ímpeto
- Balanced → Equilíbrio
- Full Defense → Guarda

Nenhuma modificação é necessária no cliente. O servidor interpreta o fightMode recebido e aplica os efeitos correspondentes (danos, alcances, sangramentos, defesas) conforme a arma equipada e a Especialização.

Os ícones e as hotkeys permanecem exactamente os mesmos. Apenas o feedback narrativo (mensagens no chat) usará a terminologia MCR quando a postura for alterada, mas essa comunicação é gerida pelo servidor.

📡 NOVOS OPCODES PARA O SISTEMA DE PROGRESSÃO (A IMPLEMENTAR NO CLIENTE)

⚠️ Os listeners para estes opcodes ainda não foram implementados. A descrição abaixo serve de especificação para quando a Fase 4 for iniciada.

O cliente deve escutar e processar os seguintes opcodes estendidos (faixa 180‑255):

- Opcode 182 – Ícone do Vínculo (Estado de Alma): o servidor envia o ID do domínio dominante para atualizar o ícone no HUD e a tooltip poética.
- Opcode 183 – Partículas de Estado de Alma: dispara efeitos visuais temáticos (fagulhas, folhas, brilho) sobre o personagem.
- Opcode 184 – Examinar Conhecimento (janela modal): acionado quando o jogador clica com o botão direito num item de conhecimento. Exibe a janela modal com título, descrição narrativa, botão "Aprender" e "Fechar". Ao confirmar, envia um pacote de volta ao servidor.

A implementação desses opcodes pode ser feita nos módulos Lua do cliente (ex.: adicionar listeners em `modules/core/opcodes.lua` ou módulo dedicado). Exemplo de registo para o opcode 184:

```lua
ProtocolGame.registerExtendedOpcode(184, function(protocol, opcode, buffer)
    if not buffer or #buffer == 0 then return end
    local data = json.decode(buffer)
    -- exibir janela modal com data.title, data.description, etc.
end)
Para os opcodes 182 e 183, o tratamento será semelhante, atualizando o HUD e disparando sistemas de partículas (se disponíveis).

Parser Lua estendido (resolveLocalVar, parseNpcConfig)

Modificações em monsters.cpp e npcs.cpp

Consulte o Guia [2] (este documento) e o repositório para compilar o RME.

⚠️ ARMADILHAS E LIÇÕES APRENDIDAS

Cache em %APPDATA%: use a cache local (cache/).

Módulos de bot/walking: remova ou desabilite se causarem travamentos.

Codificação de scripts Lua: salvar como ISO‑8859‑1.

Métodos setModal, center: use propriedades OTML ou cálculo manual.

Login HTTP: use HTTP.post, não ProtocolLogin.

Máscara de nome via Lua: requer modificação em C++ (creature::setName).

Escapes octais: usar SEMPRE para acentos no C++ do cliente. NÃO usar UTF‑8 diretamente nas strings.

A coluna lastlogin EXISTE na tabela players do schema actual do Canary (confirmado no banco de dados do projecto). O cliente não interage diretamente com ela, mas scripts podem usar essa informação.
