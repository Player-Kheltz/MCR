⚠️ REGRAS DE REVISÃO PARA O ASSISTENTE (OTUI + LUA CLIENTE)

Sintaxe OTUI: NUNCA usar chaves { } para agrupar propriedades. NUNCA usar comentários com --. A indentação é EXCLUSIVAMENTE com espaços. Widgets devem ser filhos directos da MainWindow para getChildById funcionar.

Fontes: Usar apenas fontes do catálogo oficial (cipsoftFont, verdana‑11px‑rounded, small‑9px, etc.). Não usar fontes TrueType sem teste prévio.

Strings: Toda string fixa na interface (labels, botões) em PT‑BR.

Codificação dos scripts Lua carregados via dofile: ISO‑8859‑1 (Latin‑1). Esta é uma regra absoluta.

Métodos proibidos no Redemption: Não usar setModal(true), setSize(w,h), center(). Preferir propriedades OTML (modal: true, center: true) ou cálculo manual de posição (setPosition com coordenadas calculadas).

Comunicação com servidor: Requisições HTTP.get devem ter callback onError. Listeners de Extended Opcode devem verificar se buffer não é nil.

Não usar HorizontalLayout em .otui; usar anchors e margens.

Máscara de Alma: Qualquer alteração na lista de personagens deve preservar a substituição de "Guest_*" por "Alma" (ver secção 7).

Tradução de interfaces: Todas as strings fixas devem estar em PT‑BR. Para modificações em massa, pode‑se utilizar a pipeline de tradução automática (scripts Python descritos no Índice Geral). Scripts Lua resultantes devem ser convertidos para ISO‑8859‑1 antes de serem salvos.

⚠️ SPA v4.2: Os opcodes 182, 183 e 184 são dedicados ao feedback do SPA. O cliente DEVE ter listeners implementados para eles (nos módulos Lua apropriados). Consulte a secção 9.

⚠️ SPA v4.2: As Posturas de Combate (Ímpeto, Equilíbrio, Guarda) reutilizam os modos de luta nativos do cliente (Full Attack, Balanced, Full Defense). Nenhuma alteração visual ou estrutural é necessária.

⚠️ SPA v4.2 – Cores: As mensagens coloridas geradas pelo servidor (via enviarMsgColorida e canal 24) são exibidas nativamente. O cliente não precisa de alterações para suportar este recurso. A tabela COR está na versão 9.1.

📐 1. SINTAXE OTUI (ARQUIVOS .otui)

Arquivos .otui descrevem a hierarquia e as propriedades de widgets de forma declarativa. O parser do OTClient Redemption é extremamente restrito.

REGRAS ABSOLUTAS:

NÃO use comentários com --. O parser não os reconhece e gera erros.

NÃO use chaves { } para aninhar estilos ou propriedades. Cada propriedade deve ser uma linha independente dentro do widget.

Use APENAS espaços para indentação. Tabs podem causar "invalid indentation".

Para strings fixas, use a propriedade text (ex.: text: Criar Conta). Evite !text, que tem comportamento inconsistente.

Exemplo CORRECTO de um arquivo .otui:

text
MainWindow
  id: registerWindow
  width: 300
  height: 220
  modal: true
  center: true

  Label
    id: titleLabel
    text: Criação de Conta
    font: verdana-11px-rounded
    anchors.top: parent.top
    anchors.left: parent.left
    anchors.right: parent.right
    margin-top: 15

  TextEdit
    id: accountName
    font: verdana-10px
    anchors.top: titleLabel.bottom
    anchors.left: parent.left
    anchors.right: parent.right
    margin-top: 10
    margin-left: 20
    margin-right: 20

  Button
    id: createButton
    text: Criar Conta
    font: verdana-10px
    anchors.top: accountName.bottom
    anchors.left: parent.left
    anchors.right: parent.right
    margin-top: 15
    margin-left: 50
    margin-right: 50
CARACTERÍSTICAS IMPORTANTES:

Widgets são declarados como filhos directos da janela principal (MainWindow).

Cada widget tem um id único que permite recuperá‑lo via getChildById.

Ancoragem (anchors) define a posição relativa.

Margens (margin-*) ajustam o espaçamento.

O QUE NUNCA FAZER:

text
MainWindow {
    id: registerWindow
    -- isso quebra
}
🖼️ 2. CRIAÇÃO DE WIDGETS VIA LUA

Para janelas complexas, o .otui é preferível. Mas se for necessário criar widgets programaticamente, use g_ui.createWidget ou g_ui.displayUI.

Exemplo de carregamento de .otui:

lua
local window = g_ui.displayUI('register_window.otui')
local nameField = window:getChildById('accountName')
local createButton = window:getChildById('createButton')
Exemplo de criação programática (para casos simples):

lua
local window = g_ui.createWidget('MainWindow', rootWidget)
window:setWidth(300)
window:setHeight(220)
-- NÃO usar window:setModal(true) – inválido no Redemption
-- Para centralizar, calcular posição manualmente:
local rootWidth = rootWidget:getWidth()
local rootHeight = rootWidget:getHeight()
local posX = (rootWidth - 300) / 2
local posY = (rootHeight - 220) / 2
window:setPosition({ x = posX, y = posY })
Função segura para buscar widgets recursivamente:

lua
function recursiveGetChildById(widget, id)
    if widget:getId() == id then return widget end
    for i = 1, widget:getChildCount() do
        local child = widget:getChildByIndex(i)
        local found = recursiveGetChildById(child, id)
        if found then return found end
    end
    return nil
end
🔤 3. FONTES BITMAP DISPONÍVEIS (CATÁLOGO OFICIAL)

O OTClient utiliza fontes bitmap pré‑renderizadas (.otfont). NÃO há suporte a font‑size; o tamanho é intrínseco ao arquivo da fonte.

Fontes recomendadas (testadas e funcionais):

Nome da fonte	Tamanho	Uso sugerido
cipsoftFont	~10px	Texto geral, padrão Tibia
verdana-11px-rounded	11px	Títulos, destaques
verdana-10px	10px	Corpo de texto, rótulos
small-9px	9px	Textos pequenos, status
sans-bold-16px	16px	Títulos principais
pVerdana Bold-11px	11px	Títulos alternativos
Fontes NÃO testadas (usar com cautela):

Fontes TrueType (.ttf) na pasta data/fonts/ttf/ podem exigir configuração extra. Não as utilize sem teste explícito de compatibilidade.

Regra prática: se não está na lista acima, NÃO use sem antes verificar no diretório data/fonts/otfont/ do seu OTClient.

📏 4. AJUSTE FINO DE ALINHAMENTO (TEXT-OFFSET)

A propriedade text-offset permite deslocar o texto dentro de widgets como TextEdit e Label. Sintaxe: text-offset: <horizontal> <vertical> (em pixels).

Exemplo:

text
text-offset: 0 1
→ desloca 1 pixel para baixo, corrigindo alinhamento vertical.

Este recurso é valioso para centralizar visualmente o texto em campos de entrada com altura reduzida (ex.: 28px).

💻 6. CODIFICAÇÃO DE SCRIPTS LUA COM ACENTOS (REGRA CRÍTICA)

⚠️ Esta é uma das regras mais importantes e negligenciadas do projecto.

Scripts .lua carregados pelo OTClient via dofile() ou como módulos são interpretados como LATIN-1 (ISO‑8859‑1), NÃO como UTF-8.

Portanto:

TODOS os scripts Lua que contêm strings acentuadas em PT‑BR e são executados no lado do CLIENTE devem ser salvos como ISO‑8859‑1 (Latin‑1).

Isso inclui: registro.lua, entergame.lua, spa_opcodes.lua (futuro), e qualquer módulo customizado.

Se o arquivo for salvo como UTF‑8, caracteres como "ç", "ã", "é" aparecerão como "Ã§", "Ã£", "Ã©" na interface.

Ferramentas para salvar como ISO‑8859‑1:

Notepad++: Codificação → Codificar em ANSI (que é Latin‑1 no Windows Ocidental).

VS Code: Clique em "UTF-8" na barra inferior → "Salvar com codificação" → "Western (ISO 8859‑1)".

Verificação rápida: abra o arquivo em um editor hex. O caractere "ç" deve ser representado pelo byte 0xE7 (e não 0xC3 0xA7 do UTF‑8).

ARQUIVOS QUE DEVEM SER ISO‑8859‑1 (lista não exaustiva):

modules/client_entergame/registro.lua

modules/client_entergame/entergame.lua

modules/game_alma/alma.lua (se existir)

modules/game_spa/spa_opcodes.lua (futuro)

Qualquer script que contenha strings com acentos mostradas na UI.

EXCEÇÃO: Arquivos .otui podem ser UTF‑8 sem BOM (o parser do OTUI trata correctamente).

🖥️ 7. FLUXO DO SALÃO DOS DESTINOS E MÁSCARA DE ALMA (INTEGRAÇÃO CLIENTE)

O cliente possui um botão "Salão dos Destinos" no lugar de "Criar Conta". O fluxo é:

Botão chama EnterGame.openSalaoDestinos().

A função esconde a janela de login e faz HTTP.get para /guest_login.

Com as credenciais recebidas, chama EnterGame.doAutoLogin(account, password, character).

doAutoLogin autentica via /login, obtém os dados do personagem e chama g_game.loginWorld directamente.

Após logout, EnterGame.showPanels decide se exibe a lista de personagens ou a tela de login, usando a flag isGuestSession.

⚠️ IMPLEMENTAÇÃO DA MÁSCARA DE ALMA NA LISTA DE PERSONAGENS:

No ficheiro modules/client_entergame/entergame.lua, a função onCharacterList() foi alterada para substituir o nome Guest_* por "Alma" antes de criar a lista visual:

lua
for _, characterInfo in pairs(characters) do
    if characterInfo.name and characterInfo.name:sub(1,6) == "Guest_" then
        characterInfo.name = "Alma"
    end
    -- ... restante do loop
end
A máscara também foi implementada a nível do motor (C++) para garantir que o nome apareça como "Alma" em TODOS os contextos (nome sobre a cabeça, chat, barras). Ver Guia de Compilação (Cliente), secção "Customizações MCR – Máscara de Alma".

Certifique‑se de que entergame.lua esteja salvo como ISO‑8859‑1.

O .otui correspondente deve ter o botão com id="createAccountButton" (mantido para compatibilidade), texto "Salão dos Destinos" e onClick: EnterGame.openSalaoDestinos().

📡 8. CALLBACK DO OPCODE 200 (MIGRAÇÃO DE CONTA)

O cliente escuta o opcode 200 através da seguinte função, registada no módulo entergame.lua:

lua
ProtocolGame.registerExtendedOpcode(200, function(protocol, opcode, buffer)
    if not buffer or #buffer == 0 then return end

    local success, data = pcall(json.decode, buffer)
    if not success then return end

    -- Desconecta a Alma imediatamente (método C++ síncrono)
    g_game.logout(false)

    -- Inicia o novo login com as credenciais recebidas
    EnterGame.doAutoLogin(data.account, data.password, data.character)
end)
Este código garante uma transição praticamente instantânea entre a Alma e o personagem definitivo, sem atrasos artificiais. A função g_game.logout(false) foi implementada em C++ (Game::logout) e registada em luafunctions.cpp.

📡 9. NOVOS OPCODES DO SPA v4.2 (A IMPLEMENTAR NO CLIENTE)

O SPA introduz três Extended Opcodes para feedback visual. O cliente DEVE possuir listeners para todos eles. Actualmente, estes listeners ainda não foram implementados — as especificações abaixo servem de guia para a implementação, que pode ser feita num módulo dedicado (ex.: modules/game_spa/spa_opcodes.lua, salvo como ISO‑8859‑1).

9.1 OPCODE 182 – ÍCONE DO VÍNCULO / MAESTRIA (ESTADO DE ALMA)

Objectivo: Actualizar o ícone no HUD que representa a especialização dominante (aquela em Vínculo activo ou Maestria permanente).

Dados recebidos (JSON): { "dominioId": 100, "tooltip": "O aço canta nas suas mãos.", "maestria": false }

Comportamento do cliente: Exibir ou actualizar o ícone no canto superior direito (hud.otui). A tooltip aparece ao passar o rato. Se "maestria" for true, o ícone deve indicar que o Vínculo é permanente.

Exemplo de listener:

lua
ProtocolGame.registerExtendedOpcode(182, function(protocol, opcode, buffer)
    if not buffer or #buffer == 0 then return end
    local data = json.decode(buffer)
    -- atualizar HUD com data.dominioId, data.tooltip e data.maestria
end)
9.2 OPCODE 183 – PARTÍCULAS DE ESTADO DE ALMA

Objectivo: Disparar efeitos visuais temáticos (fagulhas, folhas, brilho) sobre o personagem quando um Lampejo, Vínculo ou Maestria se forma.

Dados recebidos (JSON): { "tipo": "fogo", "intensidade": "maestria" }

Comportamento do cliente: Iniciar uma animação de partículas sobre o jogador local. A intensidade ("lampejo", "vinculo" ou "maestria") pode alterar a quantidade/densidade.

Exemplo de listener:

lua
ProtocolGame.registerExtendedOpcode(183, function(protocol, opcode, buffer)
    if not buffer or #buffer == 0 then return end
    local data = json.decode(buffer)
    -- disparar sistema de partículas com data.tipo e data.intensidade
end)
9.3 OPCODE 184 – EXAMINAR CONHECIMENTO (JANELA MODAL)

Objectivo: Quando o jogador clica com o botão direito num item de conhecimento (ex.: Livro de Feitiços), o servidor envia as informações para exibição numa janela modal.

Dados recebidos (JSON): { "titulo": "Livro de Feitiços", "descricao": "Um antigo tratado...", "habilidadeId": 10, "compreensivel": true }

Comportamento do cliente:

Exibir uma janela modal com o título e a descrição.

Se compreensivel for true, mostrar botão "Aprender" e "Fechar".

Se compreensivel for false, mostrar a mensagem enigmática e apenas o botão "Fechar".

Ao clicar "Aprender", enviar um pacote de volta ao servidor (ex.: opcode 185 ou mensagem no chat) confirmando o consumo.

Exemplo de listener:

lua
ProtocolGame.registerExtendedOpcode(184, function(protocol, opcode, buffer)
    if not buffer or #buffer == 0 then return end
    local data = json.decode(buffer)
    -- criar janela modal com data.titulo, data.descricao, botoes condicionais
end)
Todos estes opcodes dependem de extendedOpcode = true no OTCFEATURES (já configurado).

🎮 10. POSTURAS DE COMBATE – REUTILIZAÇÃO DOS MODOS DE LUTA NATIVOS

As Posturas de Combate (Ímpeto, Equilíbrio, Guarda) definidas no SPA v4.2 utilizam os modos de luta nativos do cliente Tibia:

Full Attack → Ímpeto

Balanced → Equilíbrio

Full Defense → Guarda

Estes modos são acessíveis pelas hotkeys padrão (Ctrl+A, Ctrl+B, Ctrl+D) e pelos ícones no painel de combate do cliente.

NENHUMA modificação é necessária no cliente para suportar esta mecânica. O servidor interpreta o fightMode (0=full attack, 1=balanced, 2=full defense) enviado pelo cliente e aplica os efeitos correspondentes (dano, alcance, sangramento, defesa, furtividade, etc.) com base na arma equipada e na Especialização.

O feedback narrativo (ex.: "Postura ajustada: Ímpeto adaptada à sua arma.") é enviado pelo servidor via chat colorido (canal 24) e exibido normalmente pelo cliente.

📟 11. SISTEMA DE CORES DO SPA v4.2 – EXIBIÇÃO NO CLIENTE (CANAL 24)

O SPA v4.2 introduz um sistema de cores abrangente, definido no módulo texto_colorido.lua v9.1 (tabela COR) e utilizado pela função c(). O servidor envia as mensagens coloridas através de player:sendTextMessage(24, texto), onde texto contém blocos no formato {Texto, #Cor}.

O OTClient Redemption suporta nativamente a exibição de texto colorido no canal 24, processando a sintaxe de chaves e cores. Nenhuma modificação no motor do cliente é necessária para que as cores do SPA apareçam correctamente.

O que o desenvolvedor precisa garantir:

O módulo de chat do OTClient está activo e a processar mensagens do canal 24 (comportamento padrão).

As fontes utilizadas suportam os caracteres acentuados enviados nas mensagens (as fontes bitmap padrão suportam).

Os scripts Lua do cliente que porventura processem mensagens do canal 24 não interferem na formatação.

Se as cores não aparecerem, verificar:

Se o servidor está a enviar as mensagens correctamente (use o comando !exibircores para testar).

Se o módulo game_console ou equivalente está a filtrar o canal 24.

🎨 12. FORMATO DE CORES EXPANDIDO ({texto, #hexcolor})
Para além do canal 24, o MCR expandiu o suporte a cores para TODOS os widgets
de texto do cliente através de uma modificação C++ em uiwidgettext.cpp.
O método setText() agora detecta o formato {texto, #cor} e chama
setColoredText() automaticamente quando detecta a presença de { e , no texto.

Isto significa que o formato funciona em:
- Mensagens do console (canal 24 e outros)
- Balões de fala (speech bubbles) sobre NPCs/criaturas
- Labels de interface (widgets UIText)
- Dicas de ferramentas (tooltips)

Exceção: UITextEdit (campo de input do console) ignora esta formatação para
não interferir com a digitação do jogador.

A implementação é transparente para scripts Lua do lado do servidor: qualquer
mensagem enviada com c() e canal 24, ou com o formato {texto, #cor} em qualquer
contexto, será renderizada com a cor apropriada.

12.1 CORES DAS POSTURAS NO CHAT

Quando o jogador alterna de postura, o servidor envia uma mensagem como:

text
Postura ajustada: {Ímpeto, #FF4040} adaptada à sua arma.
O cliente exibe "Ímpeto" na cor vermelha (#FF4040) e o restante do texto na cor padrão do canal. Essa é a mesma mecânica usada para todas as mensagens coloridas do SPA (Lampejos, Vínculos, Maestria, habilidades, etc.).

⚠️ ARMADILHAS E LIÇÕES APRENDIDAS

Erro "VerticalLayout is not defined": Causado por uso de layouts não suportados. Prefira sempre ancoragem manual (anchors).

Botão "Criar Conta" não funcionava: O evento onClick deve estar dentro do bloco do botão no .otui ou criado programaticamente com bind. Múltiplos comandos em onClick devem usar pipe | como separador.

Caminhos relativos no dofile: Dentro de módulos, o contexto de execução pode mudar. Use caminho absoluto com barra inicial (ex.: dofile('/registro.lua')) para garantir a localização correcta.

Widget não encontrado (nil): Se getChildById retorna nil, verifique se o widget é filho directo da janela. Use recursiveGetChildById como fallback.

Janela não centralizava: center: true no .otui pode falhar em algumas versões. Calcule as coordenadas manualmente usando getWidth/getHeight do rootWidget.

Notificações com acentos corrompidos: Resolvido com error_code no backend e mapeamento no cliente Lua ISO‑8859‑1. NUNCA envie strings acentuadas no JSON de resposta.

Tentativa de máscara de nome via Lua falhou: O motor não expõe método para alterar o nome de exibição do jogador local. A solução foi modificar a função setName() da classe Creature em C++.

⚠️ SPA v4.2 – Listeners de opcode em falta: Se os opcodes 182, 183 e 184 não tiverem listeners, o HUD de Vínculo/Maestria, as partículas e a janela de conhecimento simplesmente não aparecerão. Verifique se os módulos Lua apropriados contêm os listeners antes de testar.

⚠️ SPA v4.2 – Cores no canal 24: As cores são suportadas nativamente. Se não aparecerem, o problema está no servidor (envio incorrecto) ou no módulo de chat do cliente (filtros), não na ausência de suporte. A tabela COR está na versão 9.1.

⚠️ SPA v4.2 – Posturas: Não confundir as mensagens de postura com as de modo de luta. O servidor pode enviar ambas; certifique-se de que o chat não está a filtrar mensagens do canal 24.

⚠️ SPA v4.2 – Maestria: O opcode 182 deve suportar um campo "maestria" (boolean) para diferenciar Vínculo temporário de Vínculo permanente.

📟 13. SISTEMA DE NOTIFICAÇÕES TOAST (MÓDULO game_toast)

O MCR adicionou um sistema de notificações toast (pop-ups temporários) ao
cliente, implementado no módulo modules/game_toast/. O sistema é carregado
automaticamente por game_toast.otmod e expõe:

Funções Lua:
  Toast.show(text, type, duration)
  - text: string a exibir
  - type: "success" (verde), "error" (vermelho), "info" (azul, padrão)
  - duration: milissegundos (padrão 3000)

Comportamento:
  - As notificações são enfileiradas e exibidas sequencialmente
  - Cada toast aparece no centro superior da tela
  - Estilo: fundo azul/vermelho/verde, texto branco, cantos arredondados, sombra
  - A fila garante que toasts não se sobreponham

Uso típico:
  - Feedback de acções do jogador (item recebido, missão completada, etc.)
  - Mensagens de erro não críticas
  - Notificações de progressão (nível de domínio, etc.)

Estilo OTUI do toast:
  text
  ToastNotification {
      background-color: #4169E1  -- azul info
      margin: 2
      padding: 8
      text-align: center
      text-auto-resize: true
  }
