📝 2. CODIFICAÇÃO POR TIPO DE ARQUIVO (REVISADA)

Esta é a secção MAIS CRÍTICA deste guia. A codificação errada é a causa número 1 de bugs visuais (acentos corrompidos) no Projeto MCR.

Tipo de Arquivo | Codificação | BOM? | Exemplo
Scripts Lua (servidor e cliente) | UTF‑8 | Não | mount_summon.lua, constantes.lua
Código C++ (servidor e cliente) | UTF‑8 (`/utf-8` no MSVC) | Não | canary_server.cpp, player.cpp
Código Go (Login Server) | UTF‑8 | Não | register.go, login.go
Arquivos .otui (interface) | UTF‑8 | Não | register_window.otui
Arquivos .xml (spawn, itens) | UTF‑8 real | Não | MCR-npc.xml, items.xml
Arquivos de configuração (.lua, .env) | UTF‑8 | Não | config.lua, .env
Documentação (.txt) | UTF‑8 | Não | MCR - Guia de *.txt

Detalhamento:

⚠️ CÓDIGO C++ (SERVIDOR E CLIENTE):
Codificação UTF‑8 com a flag `/utf-8` no compilador MSVC (configurada nas 3 configurações do .vcxproj: Debug, Release, ReleaseDebug). Strings com acentos podem ser escritas como UTF‑8 literal (ex.: "Você", "Configuração"). A função `toLatin1()` (em encoding.hpp) converte UTF‑8 → Latin‑1 antes do envio ao protocolo, e é chamada automaticamente em `sendCancelMessage` e `sendTextMessage`. Escapes octais continuam a funcionar como alternativa segura, mas não são obrigatórios.

⚠️ PROTEÇÃO DE BINDINGS LUA:
Durante a tradução, foram preservados todos os nomes de funções e classes registados no Lua (ex.: Player::getPronoun, Game::logout). Nomes de parâmetros em C++ podem ser traduzidos nos comentários, mas nunca nos registerMethod ou registerClass.

⚠️ UTF‑8 (SCRIPTS LUA – SERVIDOR E CLIENTE):
Scripts Lua devem ser salvos como UTF‑8 sem BOM. O `toLatin1()` nas funções do servidor (`sendCancelMessage`, `sendTextMessage`) converte automaticamente para Latin‑1 antes de enviar ao protocolo. No cliente OTClient, o InputMessage::getString() retorna bytes crus, e a renderização usa a codificação configurada (Latin‑1 por padrão). Caracteres como 'ç', 'ã', 'é' (UTF‑8: 2 bytes) são convertidos para Latin‑1 (1 byte) pelo servidor antes da transmissão. A antiga regra ISO‑8859‑1 está obsoleta.

⚠️ IMPACTO NO MAP EDITOR (RME)
Os scripts de monstros e NPCs salvos como ISO‑8859‑1 causam um problema adicional no Remere's Map Editor (RME). O RME carrega esses ficheiros como bytes e depois assume que são UTF‑8 para criar as strings internas (wxString). Isto faz com que nomes com acentos apareçam em branco nas paletas de monstros e NPCs. Para resolver este problema, o código‑fonte do RME foi modificado para: converter o conteúdo de ISO‑8859‑1 para UTF‑8 ao ler os ficheiros .lua (função latin1_to_utf8 em encoding_utils.h). Estender o parser Lua com as funções resolveLocalVar e parseNpcConfig para reconhecer padrões comuns nos scripts do Projeto MCR. Consulte o Guia [2] para detalhes das modificações e compilação do RME.

⚠️ UTF‑8 COM BOM (C++, GO):
O compilador C++ e o compilador Go reconhecem o BOM e tratam o arquivo como UTF‑8. Isso garante acentuação correta em logs e mensagens internas.

⚠️ UTF‑8 SEM BOM (OTUI, XML):
Arquivos .otui e .xml são processados por parsers que podem rejeitar o BOM. Salvar sem BOM, mas manter como UTF‑8.

🌐 3. COMUNICAÇÃO BACKEND → FRONTEND (EVITAR CORRUPÇÃO)

Devido à incompatibilidade entre o backend Go (UTF‑8) e a interpretação do cliente Lua (ISO‑8859‑1 para respostas HTTP), strings acentuadas NÃO devem ser enviadas directamente no JSON de resposta do Login Server.

PROTOCOLO OBRIGATÓRIO (ERROR_CODE):

O backend (Login Server) retorna um código de erro padronizado em INGLÊS, SEM acentos, no campo "error_code". Exemplo: "account_name_taken", "password_too_weak", "invalid_credentials".

O cliente (Lua) possui uma TABELA DE MAPEAMENTO que traduz o error_code para a mensagem em PT‑BR (com acentos). Exemplo:

lua
local ERROR_MESSAGES = {
    account_name_taken = "O nome de conta já está em uso.",
    password_too_weak = "A senha deve ter 8+ caracteres com letras e números.",
    invalid_credentials = "Nome de conta ou senha incorretos.",
    registration_success = "Conta criada com sucesso!",
    guest_creation_failed = "Falha ao criar conta convidada.",
    alma_creation_failed = "Falha ao criar personagem convidado.",
}
Esta tabela EXISTE no script Lua do cliente (registro.lua, entergame.lua, etc.), que está salvo como ISO‑8859‑1. Portanto, as strings em PT‑BR exibidas ao jogador terão os acentos CORRECTOS. NOVOS error_codes devem ser adicionados tanto no backend quanto na tabela de mapeamento do cliente.

🔄 4. NOME DO PERSONAGEM VINDO DO JSON DE LOGIN (HTTP)

O Login Server envia a lista de personagens e a resposta de autenticação em JSON com codificação UTF‑8 (padrão obrigatório do formato). No entanto, o OTClient interpreta strings Lua como ISO‑8859‑1 (Latin‑1) e o protocolo de jogo também utiliza Latin‑1. Para evitar que o nome do próprio jogador apareça corrompido na interface (ex.: JoÃ£o em vez de João), todo o nome de personagem recebido via HTTP deve ser convertido de UTF‑8 para Latin‑1 antes de ser armazenado ou utilizado na interface.

A conversão é feita com a função utf8ToLatin1, que deve ser inserida no módulo entergame.lua (ou num módulo de utilidades carregado antes). O procedimento é:

Adicionar a função utf8ToLatin1 ao ficheiro:

lua
-- Converte uma string UTF-8 para Latin‑1 (ISO 8859‑1)
local function utf8ToLatin1(str)
    local result = {}
    local i = 1
    while i <= #str do
        local c = str:byte(i)
        if c < 0x80 then
            result[#result + 1] = string.char(c)
            i = i + 1
        elseif c >= 0xC2 and c <= 0xDF then -- sequência de 2 bytes
            local c2 = str:byte(i + 1)
            if c2 and c2 >= 0x80 and c2 <= 0xBF then
                local codepoint = ((c - 0xC0) * 64) + (c2 - 0x80)
                if codepoint <= 0xFF then
                    result[#result + 1] = string.char(codepoint)
                else
                    result[#result + 1] = '?'
                end
                i = i + 2
            else
                result[#result + 1] = '?'
                i = i + 1
            end
        else -- sequência inválida ou > 2 bytes
            result[#result + 1] = '?'
            i = i + 1
        end
    end
    return table.concat(result)
end
Aplicar a conversão em dois pontos dentro do entergame.lua:

Na função onCharacterList, após o loop que aplica a máscara de Alma e antes de entregar os personagens ao módulo de lista:

lua
for _, characterInfo in pairs(characters) do
    characterInfo.name = utf8ToLatin1(characterInfo.name)
end
Na função doAutoLogin, imediatamente após a construção da tabela characters a partir da resposta HTTP e antes de procurar o personagem alvo:

lua
for _, char in ipairs(characters) do
    char.name = utf8ToLatin1(char.name)
end
Garantir que o ficheiro entergame.lua está salvo como ISO‑8859‑1 (embora a função só contenha caracteres ASCII, o ficheiro em si pode conter outras strings acentuadas).

Após esta conversão, o nome do personagem fica em Latin‑1, igual aos nomes que chegam via protocolo de jogo, e será exibido corretamente em toda a interface (chat, lista de personagens, etc.).

🔑 5. DESTAQUE NO CHAT E SISTEMA DE CORES (v9.1)

O OTClient suporta exclusivamente o destaque de palavras envolvidas por {chaves}. Esse realce é herdado do protocolo Tibia e não requer nenhuma modificação no cliente.

Regras:

Utilize {chaves} para destacar ações ({aceitar}, {entregar}), itens ({10 Poções de Vida}), locais ({Floresta Sombria}) e recompensas ({500 pontos de experiência}).

Para cores, utilize EXCLUSIVAMENTE a função c() do SPA, definida em texto_colorido.lua v9.1 (MCR/Geral/). Exemplo: c("Texto", COR.SISTEMA_INFO). Isso gera a string "{Texto, #5DADE2}", que é interpretada corretamente pelo OTClient.

NÃO utilize tags {{#RRGGBB}}, HTML, ou qualquer outra sintaxe – elas não produzem efeito visual ou corrompem a exibição.

NUNCA use "}?" junto; sempre "} ?" para evitar corrupção visual no cliente.

✍️ 6. ESTILO E QUALIDADE DO TEXTO

Ser NATURAL: evitar traduções literais do inglês ("você completou a quest" soa artificial; prefira "você concluiu a missão").

Usar o padrão culto da língua, mas com flexibilidade para personagens caricatos (um bêbado pode falar errado; um dragão ancião pode usar arcaísmos).

Manter consistência: se um termo foi traduzido de uma forma, usar a mesma tradução em todo o projecto.

🎭 7. MÁSCARA DE ALMA EM MENSAGENS DO SERVIDOR (REVISADA)

O personagem convidado (Guest_*) deve ser sempre apresentado como "Alma" para o jogador. Esta substituição deve ser aplicada em TODAS as mensagens de sistema, incluindo:

Mensagens de "look" (examinar personagem)

Mensagens de entrada/saída de jogadores

Qualquer broadcast que mencione o nome do personagem convidado

Descrições em janelas modais ou tooltips

A máscara é implementada nos scripts do servidor substituindo o nome real pelo nome "Alma" antes de enviar a mensagem ao jogador:

lua
local name = creature:getName()
if name:sub(1,6) == "Guest_" then
    name = "Alma"
end
Arquivos que já possuem a máscara:

data/creaturescripts/scripts/player.lua (onLookInBattleList)

data/events/scripts/on_look.lua (handleCreatureDescription)

Novos scripts que gerem mensagens com o nome do personagem devem incluir a mesma verificação. Na estrutura Revscript, scripts de progressão que emitam mensagens de sistema também devem aplicar a máscara se mencionarem o nome do Alma.

⚠️ Nota sobre a lista de personagens (cliente): O nome do personagem convidado (Guest_*) é recebido via JSON (UTF‑8) e depois mascarado para "Alma". Após a máscara, também deve ser aplicada a conversão utf8ToLatin1 para garantir que o nome exibido na interface está na codificação correcta.

A partir da versão 4.0, todas as mensagens que se dirijam ao personagem devem respeitar o pronome definido na coluna players.pronoun, obtido preferencialmente pelo módulo pronouns.lua. ⚠️ ARMADILHAS E LIÇÕES APRENDIDAS (ACTUALIZADAS)

Acentos corrompidos no cliente e servidor: Scripts Lua de ambas as pontas DEVEM ser salvos como ISO‑8859‑1 para evitar artefactos.

Mensagens do backend com acentos: Enviar strings acentuadas no JSON do Login Server resultava em "jÃi estÃi" no cliente. A solução foi o padrão error_code com mapeamento no cliente.

Console do Canary com "Vers├úo": Os logs em português apareciam corrompidos porque o console do Windows usava CP 850. A solução foi adicionar SetConsoleOutputCP(CP_UTF8) no main.cpp.

Destaque no chat: Apenas {chaves} são reconhecidas; qualquer tentativa de usar {{#RRGGBB}} falha silenciosamente.

ISO‑8859‑1 vs ANSI: No Windows, "ANSI" é um apelido para a codificação Windows‑1252, que é um superset do ISO‑8859‑1. Salvar como "ANSI" no Notepad++ ou VS Code geralmente funciona, desde que não haja caracteres específicos do Windows‑1252 (como €, ‚, ƒ) nos scripts, o que é raro.

Mensagens de look do servidor estavam em inglês: Foram traduzidas ("You see" → "Você vê") e a máscara de Alma foi aplicada para consistência total.

⚠️ SPA v4.2 – Termos novos: Ao usar "Maestria", "Sinergia Elemental", "Arremesso de Escudo", "Chi", "Virtude", "Serenidade" em mensagens narrativas, assegure‑se de que a string está em ISO‑8859‑1 e que o termo não é confundido com comandos ou variáveis. Use as cores apropriadas da tabela COR para cada estado/domínio.

⚠️ SPA v4.2 – Cores: A função c() está disponível globalmente e deve ser a única forma de gerar texto colorido. Não construa manualmente strings como "{Texto, #Cor}" a menos que esteja a implementar a própria função c(). A tabela COR está na versão 9.1, com novas constantes para missões e NPCs.

⚠️ SPA v4.2 – Estrutura Revscript: Com scripts espalhados por subpastas (core/, habilidades/, comandos/), é fácil esquecer a codificação de um ficheiro. Crie o hábito de verificar a codificação ao criar qualquer novo script .lua que contenha acentos.

⚠️ SPA v4.2 – Coluna lastlogin: Confirmada a existência; sem impacto na localização.
