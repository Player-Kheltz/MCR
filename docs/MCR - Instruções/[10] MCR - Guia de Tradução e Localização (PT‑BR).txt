📝 2. CODIFICAÇÃO POR TIPO DE ARQUIVO (REVISADA)

Esta é a secção MAIS CRÍTICA deste guia. A codificação errada é a causa número 1 de bugs visuais (acentos corrompidos) no Projeto MCR.

Tipo de Arquivo | Codificação | BOM? | Exemplo
Scripts Lua do CLIENTE (dofile, módulos) | ISO‑8859‑1 | Não | registro.lua, entergame.lua
Scripts Lua do SERVIDOR (todas as subpastas) | ISO‑8859‑1 | Não | oraculo.lua, evento_kill.lua, player.lua
Código C++ (servidor e cliente) com escapes octais (preferível) | UTF‑8 ou ASCII | COM | canary_server.cpp, main.cpp
Código Go (Login Server) | UTF‑8 | COM | register.go, login.go
Arquivos .otui (interface) | UTF‑8 | SEM | register_window.otui
Arquivos .xml (spawn, itens) | UTF‑8 | SEM | MCR-npc.xml, items.xml
Arquivos de configuração (.lua, .env) | UTF‑8 | SEM | config.lua, .env
Documentação (.txt) | UTF‑8 | COM | MCR - Guia de *.txt

Detalhamento:

⚠️ CÓDIGO C++ (SERVIDOR E CLIENTE):
A partir da versão 5.0, todo o código‑fonte C++ do servidor e do cliente foi traduzido para PT‑BR utilizando uma pipeline de tradução automática supervisionada. Para garantir a máxima compatibilidade e evitar quaisquer artefactos de encoding, todas as strings que contêm acentos ou caracteres especiais são representadas exclusivamente através de escapes octais da tabela Latin‑1 (ISO‑8859‑1). Exemplo: "bem‑vindo ao Projeto MCR" pode ser escrito como "bem‑vindo ao Projeto MCR" (sem acentos) ou, se necessário, "bem‑vindo \340o Projeto MCR" para "à". NÃO se utiliza UTF‑8 directamente nas strings; o ficheiro pode ser salvo como ASCII puro, sem BOM, pois os escapes garantem a representação correcta. Comentários também seguem a mesma regra, embora seja preferível usar apenas caracteres ASCII nos comentários para evitar problemas.

⚠️ PROTEÇÃO DE BINDINGS LUA:
Durante a tradução, foram preservados todos os nomes de funções e classes registados no Lua (ex.: Player::getPronoun, Game::logout). Nomes de parâmetros em C++ podem ser traduzidos nos comentários, mas nunca nos registerMethod ou registerClass.

⚠️ ISO‑8859‑1 (SCRIPTS LUA – CLIENTE E SERVIDOR):
Scripts Lua que contenham strings acentuadas e sejam exibidos ao jogador (tanto no cliente quanto no servidor) DEVEM ser salvos como ISO‑8859‑1 (Latin‑1 / ANSI no Windows). Isso porque o OTClient interpreta todas as strings recebidas como Latin‑1, mesmo quando originadas de um script do servidor. Caracteres como 'ç', 'ã', 'é' são representados por um ÚNICO byte (ex.: 0xE7 = ç). Se salvo como UTF‑8, aparecerão como "Ã§", "Ã£", "Ã©" na interface. Essa regra se aplica a todos os scripts Lua que geram texto visível ao jogador, sejam eles do cliente (dofile) ou do servidor (em qualquer subpasta de data-canary/scripts/MCR/). A estrutura Revscript não altera este requisito.

⚠️ IMPACTO NO MAP EDITOR (RME)
Os scripts de monstros e NPCs salvos como ISO‑8859‑1 causam um problema adicional no Remere's Map Editor (RME). O RME carrega esses ficheiros como bytes e depois assume que são UTF‑8 para criar as strings internas (wxString). Isto faz com que nomes com acentos apareçam em branco nas paletas de monstros e NPCs. Para resolver este problema, o código‑fonte do RME foi modificado para: converter o conteúdo de ISO‑8859‑1 para UTF‑8 ao ler os ficheiros .lua (função latin1_to_utf8 em encoding_utils.h). Estender o parser Lua com as funções resolveLocalVar e parseNpcConfig para reconhecer padrões comuns nos scripts do Projeto MCR. Consulte o Guia [2] para detalhes das modificações e compilação do RME.

⚠️ UTF‑8 COM BOM (C++, GO):
O compilador C++ e o compilador Go reconhecem o BOM e tratam o arquivo como UTF‑8. Isso garante acentuação correta em logs e mensagens internas.

⚠️ UTF‑8 SEM BOM (OTUI, XML):
Arquivos .otui e .xml são processados por parsers que podem rejeitar o BOM. Salvar sem BOM, mas manter como UTF‑8.

🌐 3. COMUNICAÇÃO BACKEND → FRONTEND (EVITAR CORRUPÇÃO)

Devido à incompatibilidade entre o backend Go (UTF‑8) e a interpretação do cliente Lua (ISO‑8859‑1 para respostas HTTP), strings acentuadas NÃO devem ser enviadas directamente no JSON de resposta do Login Server.

PROTOCOLO OBRIGATÓRIO (ERROR_CODE):

O backend (Login Server) retorna um código de erro padronizado em INGLÊS, SEM acentos, no campo "error_code". Exemplo: "account_name_taken", "password_too_weak", "invalid_credentials".

O cliente (Lua) possui uma TABELA DE MAPEAMENTO que traduz o error_code para a mensagem em PT‑BR (com acentos). Exemplo:

```lua
local ERROR_MESSAGES = {
    account_name_taken = "O nome de conta já está em uso.",
    password_too_weak = "A senha deve ter 8+ caracteres com letras e números.",
    invalid_credentials = "Nome de conta ou senha incorretos.",
    registration_success = "Conta criada com sucesso!",
    guest_creation_failed = "Falha ao criar conta convidada.",
    alma_creation_failed = "Falha ao criar personagem convidado.",
}
Esta tabela EXISTE no script Lua do cliente (registro.lua, entergame.lua, etc.), que está salvo como ISO‑8859‑1. Portanto, as strings em PT‑BR exibidas ao jogador terão os acentos CORRECTOS. NOVOS error_codes devem ser adicionados tanto no backend quanto na tabela de mapeamento do cliente.

🔄 4. NOME DO PERSONAGEM VINDO DO JSON DE LOGIN (HTTP)

O Login Server envia a lista de personagens e a resposta de autenticação em JSON com codificação UTF‑8 (padrão obrigatório do formato). No entanto, o OTClient interpreta strings Lua como ISO‑8859‑1 (Latin‑1) e o protocolo de jogo também utiliza Latin‑1. Para evitar que o nome do próprio jogador apareça corrompido na interface (ex.: JoÃ£o em vez de João), todo o nome de personagem recebido via HTTP deve ser convertido de UTF‑8 para Latin‑1 antes de ser armazenado ou utilizado na interface.

A conversão é feita com a função utf8ToLatin1, que deve ser inserida no módulo entergame.lua (ou num módulo de utilidades carregado antes). O procedimento é:

Adicionar a função utf8ToLatin1 ao ficheiro:

lua
-- Converte uma string UTF-8 para Latin‑1 (ISO 8859‑1)
local function utf8ToLatin1(str)
    local result = {}
    local i = 1
    while i <= #str do
        local c = str:byte(i)
        if c < 0x80 then
            result[#result + 1] = string.char(c)
            i = i + 1
        elseif c >= 0xC2 and c <= 0xDF then -- sequência de 2 bytes
            local c2 = str:byte(i + 1)
            if c2 and c2 >= 0x80 and c2 <= 0xBF then
                local codepoint = ((c - 0xC0) * 64) + (c2 - 0x80)
                if codepoint <= 0xFF then
                    result[#result + 1] = string.char(codepoint)
                else
                    result[#result + 1] = '?'
                end
                i = i + 2
            else
                result[#result + 1] = '?'
                i = i + 1
            end
        else -- sequência inválida ou > 2 bytes
            result[#result + 1] = '?'
            i = i + 1
        end
    end
    return table.concat(result)
end
Aplicar a conversão em dois pontos dentro do entergame.lua:

Na função onCharacterList, após o loop que aplica a máscara de Alma e antes de entregar os personagens ao módulo de lista:

lua
for _, characterInfo in pairs(characters) do
    characterInfo.name = utf8ToLatin1(characterInfo.name)
end
Na função doAutoLogin, imediatamente após a construção da tabela characters a partir da resposta HTTP e antes de procurar o personagem alvo:

lua
for _, char in ipairs(characters) do
    char.name = utf8ToLatin1(char.name)
end
Garantir que o ficheiro entergame.lua está salvo como ISO‑8859‑1 (embora a função só contenha caracteres ASCII, o ficheiro em si pode conter outras strings acentuadas).

Após esta conversão, o nome do personagem fica em Latin‑1, igual aos nomes que chegam via protocolo de jogo, e será exibido corretamente em toda a interface (chat, lista de personagens, etc.).

🔑 5. DESTAQUE NO CHAT

O OTClient suporta exclusivamente o destaque de palavras envolvidas por {chaves}. Esse realce é herdado do protocolo Tibia e não requer nenhuma modificação no cliente.

Regras:

Utilize {chaves} para destacar ações ({aceitar}, {entregar}), itens ({10 Poções de Vida}), locais ({Floresta Sombria}) e recompensas ({500 pontos de experiência}).

NÃO utilize tags {{#RRGGBB}}, HTML, ou qualquer outra sintaxe – elas não produzem efeito visual.

NUNCA use "}?" junto; sempre "} ?" para evitar corrupção visual no cliente.

🔍 CHECKLIST DE REVISÃO PARA O ASSISTENTE (TRADUÇÃO E LOCALIZAÇÃO)

Este checklist DEVE ser aplicado a QUALQUER modificação, independentemente da camada, pois a localização é transversal.

☐ Todo texto visível ao jogador está em PT‑BR com acentuação correta?
☐ Comentários de código estão em PT‑BR (exceto nomes de variáveis)?
☐ A codificação do arquivo está correta conforme a tabela da Secção 2?
☐ Scripts Lua (cliente e servidor, em todas as subpastas) estão salvos como ISO‑8859‑1?
☐ Arquivos .otui estão salvos como UTF‑8 sem BOM?
☐ Strings enviadas do backend (Go) usam o padrão error_code (sem acentos)?
☐ O mapeamento error_code → mensagem PT‑BR existe no cliente Lua?
☐ Destaque no chat usa apenas {chaves} e NUNCA "}?" junto?
☐ O texto soa NATURAL e COERENTE com o universo MCR?
☐ A terminologia está padronizada (Aventureiro, personagem, missão)?
☐ Os novos termos do Sistema de Progressão (Lampejo, Vínculo, etc.) são usados correctamente e sem expor números?
☐ Mensagens do servidor aplicam a máscara de Alma para personagens Guest_*?
☐ Mensagens de look usam "Você vê" (não "You see")?
☐ Mensagens dirigidas ao personagem respeitam o pronome (via pronouns.lua)?
☐ Código C++ traduzido seguiu a pipeline automática supervisionada?
☐ Strings acentuadas em C++ foram convertidas para escapes octais Latin‑1 (OBRIGATÓRIO)?
☐ Nomes de bindings Lua (métodos, classes) permanecem inalterados?
☐ Ficheiros de cabeçalho (.h/.hpp) não tiveram macros ou declarações alteradas?
☐ O comando look (servidor) utiliza o módulo pronouns.lua para personalizar a descrição?
☐ Os nomes de personagens provenientes do JSON de login são convertidos de UTF‑8 para Latin‑1 no entergame.lua?
☐ O módulo pronouns.lua é acedido como global (sem dofile) a partir de MCR/Geral/?

📤 COMO ATUALIZAR O REPOSITÓRIO GIT

git add "<arquivos alterados>"
git commit -m "feat(lang): Atualiza guia de tradução com termos v2.1.1 e Revscript"
git push origin main
