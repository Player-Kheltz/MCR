🎯 OBJETIVO DESTE DOCUMENTO
Este documento descreve de forma exaustiva e canónica o sistema de perseguição multi‑piso do Projeto MCR na sua versão 8.0. Inclui todos os ficheiros de código relevantes, a arquitetura da máquina de estados, as estruturas de dados, os mecanismos de timeout e desistência, o mapa cognitivo global partilhado (GlobalMonsterMap), os algoritmos de cercamento e flanqueamento, o retorno ao spawn sem teleportes, o sistema de engajamento e deadline adaptativo, a integração entre hostis e summons, os comandos de diagnóstico e as regras de log.

A versão 8.0 é a consolidação de todas as melhorias introduzidas após a 7.2.1, com especial foco em:

Throttles independentes para procura de escadas e rotas alternativas.

Anti‑ping‑pong — detecção e prevenção de ciclos de sobe‑e‑desce na mesma escada.

Múltiplas mudanças de piso no cercamento.

Retorno ao spawn sempre a pé, sem teleportes (exceto as próprias transições de escada).

Cancelamento imediato da perseguição ao receber dano.

Reação mais rápida — throttles globalmente reduzidos.

Invalidação reativa melhorada com mini‑scans e limpeza imediata da rota.

BFS preguiçoso com expansão ao vivo e ponte de corredor (bridgeCorridorUnlocked).

Alimentação contínua do grafo via scanPlayerTrail.

Persistência assíncrona do grafo em disco com validação de transições.

CÓDIGOS (Referência)

1. VISÃO GERAL
O sistema permite que criaturas hostis e summons persigam um jogador através de escadas, buracos, teleportes e rampas. A base do sistema é a fila de transições de escada (StairTransition) que o jogador vai deixando ao mudar de piso. As criaturas consultam essa fila, o mapa cognitivo global (GlobalMonsterMap) e, se necessário, realizam scans reativos para planear uma rota até ao líder.

Quando o caminho direto (A*) falha, o monstro utiliza o grafo cognitivo para planear uma rota de cercamento que pode atravessar múltiplos pisos. Se nenhuma rota for encontrada após várias tentativas, o monstro desiste e regressa ao spawn caminhando, utilizando as escadas percorridas no sentido inverso, nunca usando teleportes (exceto a própria mecânica de subir/descer escadas).

A máquina de estados é executada integralmente dentro do ciclo onThinkMCR, sem threads adicionais (exceto a gravação assíncrona do ficheiro de cache).

1.1 Tipos de Criaturas e Comportamentos
Tipo	Perseguição	Timeout	Retorno	Mapa Cognitivo
Hostil	Persegue o jogador que está a atacar	Adaptativo (10–45s via engagement)	Caminha de volta ao spawn; nunca teleporta (exceto escadas)	Sim, para planear cercos e atalhos
Summon	Segue o mestre através de escadas	120 ciclos (~120s)	Teleporta para o mestre	Sim, para regressar ao mestre (se ativado)
2. ARQUITETURA DA MÁQUINA DE ESTADOS
A cada ciclo de onThinkMCR, o monstro avalia as seguintes prioridades pela ordem indicada. Apenas um estado é processado por ciclo, exceto a verificação de combate que pode cancelar tudo.

text
┌──────────────────────────────────────────────┐
│ 1. COMBATE (prioridade máxima)                │
│    Se pode atacar → cancela tudo, ataca       │
├──────────────────────────────────────────────┤
│ 2. RETORNO AO SPAWN / CASA                   │
│    Se m_returningToSpawn / m_returningHome    │
│    → processarRetornoSpawn/Simples            │
│    (pode ser cancelado se líder visível)      │
├──────────────────────────────────────────────┤
│ 3. SEGUIR ROTA GLOBAL (BFS)                  │
│    Se m_usingGlobalPath == true               │
│    → followGlobalPathStep()                   │
│    (verifica constantemente se líder acessível)│
├──────────────────────────────────────────────┤
│ 4. PROCURAR ESCADAS (checkStairPursuit)       │
│    Se não estiver a perseguir nem a retornar  │
│    → throttle 1500ms, procura melhor escada   │
│    via grafo cognitivo ou fila do jogador     │
├──────────────────────────────────────────────┤
│ 5. PERSEGUIÇÃO ATIVA (m_pursuingStairs)       │
│    Tenta alcançar m_stairOrigin               │
│    Gere travamentos, tenta atalhos,           │
│    tenta cercamento se líder inacessível      │
├──────────────────────────────────────────────┤
│ 6. CERCAMENTO / ROTA ALTERNATIVA / PROATIVA   │
│    Se líder no mesmo piso e inacessível       │
│    → tryFlankingRoute()                       │
│    → tryAlternativeRouteToLeader()            │
│    → tryProactiveGlobalPursuit()              │
├──────────────────────────────────────────────┤
│ 7. OCIOSO                                      │
│    Se sem alvo → tryIdleWander()              │
└──────────────────────────────────────────────┘
Regra de ouro: O BFS nunca toma o controlo primário. É sempre uma ferramenta auxiliar para planear uma rota até um alvo. Enquanto uma rota BFS está ativa, a máquina de estados continua a verificar se o jogador ficou atacável.

Novidade v8.0: Ao receber dano (onAttackedByPlayer), todas as perseguições e retornos são cancelados imediatamente, e o monstro foca‑se no jogador.

4. MECANISMOS DE TIMEOUT E DESISTÊNCIA
4.1 Timeout Adaptativo (Hostis)
Deadline base: 25 segundos.

Fórmula: deadline = currentTime + (25000 + engagement × 350) ms.

Máximo: 45 segundos (engagement = 100).

Enquanto o líder está visível, o deadline é constantemente renovado.

Se o engagement aumentar significativamente durante a perseguição, o deadline é estendido.

4.2 Timeout por Perda de Visão
Máximo sem ver o líder: 5 segundos (5 ciclos de onThinkMCR).

Contador m_sightLostTicks incrementa quando o líder não está visível.

Após 5 ticks, o monstro desiste e inicia retorno ao spawn.

4.3 Timeout de Progresso (escada)
Se o monstro não reduzir a distância à escada alvo por 1,5 segundos, abandona a escada e avança para a próxima transição.

Utiliza m_minDistToStair e m_lastDistImprovementTime.

4.4 Falhas de Movimento Forçado
3 falhas consecutivas de internalMoveCreature → abandona a escada atual.

4.5 Timeout de Summons
120 ciclos de onThink → teleporta para o destino.

4.6 Anti‑Ping‑Pong
Se a mesma escada (origem e destino idênticos) for usada 3 vezes num intervalo de 5 segundos, é aplicado um cooldown longo de 45 segundos, forçando o sistema a explorar outras alternativas.

Escadas falhadas são registadas em m_recentlyFailedStairs com cooldown de 30 segundos.

4.7 Desistência no Retorno
Retorno por escadas: após 15 falhas totais, reinicia o percurso (iniciarRetornoSpawn).

Retorno simples: após 10 falhas totais, desiste e deambula (tryIdleWander).

Timeout de caminhada: se a caminhada de retorno exceder 5 segundos, é cancelada e o sistema avança para a próxima etapa.

5. WAYPOINTS DE SUPERFÍCIE (MIGALHAS)
Os waypoints de superfície são pontos de passagem que o jogador larga ao caminhar. São usados como fallback quando o mapa cognitivo não consegue ajudar.

O jogador adiciona um waypoint a cada 5 tiles percorridos (Player::addSurfaceWaypoint).

Expiração: 2 minutos sem uso.

Hostis usam waypoints quando ficam bloqueados a caminho da escada (após tentar o mapa cognitivo).

Summons usam waypoints para encontrar o mestre no mesmo piso quando inacessível.

6. MAPA COGNITIVO GLOBAL (GlobalMonsterMap)
O GlobalMonsterMap é um grafo partilhado por todos os monstros que armazena tiles caminháveis e transições de piso. É implementado como um singleton com acesso thread‑safe (std::shared_mutex).

6.1 Alimentação do Grafo
O grafo é alimentado de quatro formas principais:

Transições do jogador – injetadas diretamente via addStairTransition sempre que checkStairPursuit, tryFlankingRoute ou a perseguição proativa são executados. v8.0: addStairTransition já não força walkable = true nos tiles de origem/destino, evitando que tiles bloqueados sejam considerados transitáveis.

scanPlayerTrail (rasto do jogador) – mini‑scan de raio 10 tiles e profundidade Z=1, executado automaticamente quando o tile atual do monstro ou do líder é desconhecido. Também chamado em onWalkComplete e em várias funções de perseguição.

Scans reativos do monstro – tryFlankingRoute faz mini‑scans duplos (posição do líder + posição do monstro) com scanPlayerTrail. walkToWaypoint e processarRetornoSpawn também usam scanPlayerTrail em caso de falha de pathfinding.

Descoberta automática de escadas – durante qualquer scanArea, tiles com flag TILESTATE_FLOORCHANGE geram automaticamente transições para z±1 (ou para o destino exato, se disponível via Tile::getDestination). Os 8 vizinhos do destino são também mapeados para garantir conectividade.

6.2 Persistência e Auto‑Reparação
Gravação: a cada saveServer, em thread separada, para data/world/global_monster_map.bin. Apenas tiles estáticos (dynamic == false) são salvos. Antes de gravar, validateTransitions() limpa transições inválidas.

Carregamento: no arranque do servidor (Game::loadMap). Após a leitura, validateTransitions() remove automaticamente transições para tiles inexistentes ou bloqueados.

Limpeza de emergência: se o ficheiro estiver corrompido, basta apagar o .bin e o servidor iniciará com o grafo vazio, reconstruindo‑o progressivamente.

6.3 Invalidação Reativa e Revalidação de Obstáculos Temporários
invalidateTileBlocked() determina se o bloqueio é permanente (TILESTATE_BLOCKSOLID) ou temporário (TILESTATE_BLOCKPATH sem BLOCKSOLID).

Para bloqueios temporários, addTemporaryObstacle() marca o tile como walkable = false, dynamic = true com um TTL de 30 segundos.

cleanupTemporaryObstacles() (chamada a cada 5 segundos pelo onThinkMCR) percorre a lista de obstáculos expirados e revalida o estado real do tile.

v8.0: invalidateTileBlocked agora também executa scanPlayerTrail(pos, 3) e clearGlobalPath(), forçando o recálculo imediato da rota após a deteção de um bloqueio.

Validação preguiçosa: isWalkableUnlocked() consulta o estado real do jogo para tiles marcados como dynamic, garantindo reação imediata ao desaparecimento de um obstáculo.

6.4 Descoberta Automática de Escadas
Durante o scanArea, para cada tile com flag TILESTATE_FLOORCHANGE:

É chamado tile->getDestination(). Se o destino for válido (não Position()), a transição é registada com as coordenadas reais.

Caso contrário, assume‑se z±1 na mesma posição (x,y).

Em ambos os casos, os 8 tiles vizinhos do destino são também mapeados com addPos para garantir conectividade.

v8.0 – Modo de sementes: Se a origem do scan já está mapeada, o scanArea procura tiles conhecidos que tenham vizinhos ainda não explorados e usa‑os como sementes adicionais para a BFS de expansão, acelerando a cobertura de áreas já parcialmente conhecidas.

6.5 BFS Preguiçoso com Expansão ao Vivo (findPath)
O coração do sistema de cercamento. Características:

8 direções com validação de diagonais (evita cortar paredes).

Expansão preguiçosa: isReachableForPathUnlocked() consulta o mapa real se o tile não estiver no grafo, e registerDiscoveredTileUnlocked() adiciona‑o automaticamente. Isto elimina o problema das "duas bolhas" desconectadas.

Permite alcançar o destino mesmo que este seja uma escada bloqueada (goalIsStair).

Limite de 2000 nós expandidos.

BFS bidirecional ativado como fallback se o BFS principal falhar e existirem transições.

Logs de diagnóstico detalhados em caso de falha (nós expandidos, estado dos tiles, total de tiles e transições no grafo).

6.6 Ponte de Corredor (bridgeCorridorUnlocked)
Quando o BFS principal falha e start e goal estão no mesmo piso, é tentada uma ponte leve de até 64 passos entre os dois pontos. Esta ponte:

Usa apenas tiles caminháveis no mapa real.

Regista os tiles descobertos no grafo.

Não reinicia a busca – apenas expande o grafo para que futuros findPath possam ter sucesso.

7. CERCAMENTO E FLANQUEAMENTO (tryFlankingRoute)
Ativado quando o líder está no mesmo piso mas inacessível via A* (caminho direto bloqueado). Fluxo completo:

Verifica condições: throttle de 600 ms (líder visível) ou 1000 ms (líder não visível).

Mini‑scan duplo: scanPlayerTrail na posição do líder (raio 10) e na posição do monstro (raio 10), ambos com force=true.

BFS: findPath(getPosition(), leaderPos). Se falhar, tenta findNearestApproachTo para uma aproximação progressiva.

Aceitação da rota: a rota é aceite se a distância Manhattan do último tile ao líder for menor que a distância atual (newDist < currentDist). Isto garante que o monstro avança sempre para mais perto do líder, mesmo que temporariamente se afaste.

Primeiro passo ocupado: se o primeiro tile da rota estiver ocupado por outra criatura, tenta avançar para o próximo passo da rota.

Se encontrar rota, ativa m_usingGlobalPath e segue o caminho. Se estava em perseguição por escada, cancela‑a.

Contador de falhas: m_flankingFailCount é incrementado apenas no onThinkMCR em caso de insucesso. Após 12 falhas consecutivas, o monstro desiste e inicia retorno ao spawn.

8. RETORNO AO SPAWN / POSIÇÃO INICIAL
Quando a perseguição termina por timeout ou desistência, o monstro regressa ao ponto de origem sem nunca usar teleportes (exceto as transições de escada, que são parte do mundo).

Com escadas percorridas: processarRetornoSpawn() percorre o m_returnPath no sentido inverso, usando as escadas de volta. Apenas a transição de escada em si pode envolver um teleporte (ex.: internalTeleport para a origem da escada quando o monstro está no destino e a escada é ativa).

Sem escadas: processarRetornoSimples() caminha diretamente para m_homePosition usando pathfinding A* normal.

Novidades da v8.0:

Nunca teleporta como fallback; toda a locomoção é por movimento normal e escadas.

Timeout de caminhada de 5 segundos: se a caminhada não progredir, é cancelada e o sistema avança para a próxima escada ou etapa.

Após 15 falhas totais no retorno por escadas: reinicia o percurso (iniciarRetornoSpawn).

Após 10 falhas totais no retorno simples: desiste e deambula (tryIdleWander).

Cancelamento do retorno: se o líder ficar visível a ≤ 25 tiles, o retorno é cancelado e o monstro reengaja.

9. SISTEMA DE ENGAJAMENTO E DEADLINE ADAPTATIVO
9.1 Barra de Engajamento
Sobe: +2 ao causar ou receber dano do jogador; +1 por segundo enquanto o jogador está visível.

Desce: −3 por segundo sem ver o jogador.

Limites: 0–100.

9.2 Cálculo do Deadline
text
deadline = currentTime + (25000 + engagement × 350) ms
Recálculo no início da perseguição.

Renovado enquanto o líder está visível (nunca desiste se vir o jogador).

Estendido se o engagement aumentar significativamente durante a perseguição.

13. COMPORTAMENTO ESPERADO DOS MONSTROS
Perseguição multi‑piso: segue o jogador através de escadas usando a fila de transições e o grafo cognitivo.

Múltiplas mudanças de piso: pode planear rotas que atravessam vários andares para cercar o jogador.

Uso de escadas nunca pisadas pelo jogador: escolhe qualquer escada do grafo cognitivo que o aproxime do líder.

Cercamento: se o caminho direto está bloqueado, faz mini‑scans ao redor do líder e de si próprio, e usa o BFS para encontrar uma rota alternativa.

Aproximação progressiva: se o BFS não encontra uma rota completa, findNearestApproachTo encontra o tile mais próximo do líder, e o monstro aproxima‑se (desde que a distância diminua).

Reação imediata a obstáculos: se uma Magic Wall desaparece, o monstro deteta a mudança na próxima consulta ao tile (validação preguiçosa). Ao detetar um bloqueio, a rota é imediatamente recalculada.

Retorno sem teleportes: regressa ao spawn caminhando, usando as escadas percorridas. Se o líder reaparecer (≤ 25 tiles), o retorno é cancelado.

Foco imediato no jogador: ao receber dano, abandona qualquer perseguição ou retorno e foca‑se no jogador.

Persistência adaptativa: o deadline aumenta com o engajamento; nunca desiste enquanto vê o jogador.

Prevenção de ping‑pong: não fica preso a subir e descer a mesma escada repetidamente.
