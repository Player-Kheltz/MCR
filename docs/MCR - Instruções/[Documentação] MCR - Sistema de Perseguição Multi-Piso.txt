1. VISÃO GERAL
O sistema permite que criaturas hostis e summons persigam um jogador através de escadas, buracos, teleportes e rampas. A base do sistema é:

A* nativo do motor para navegação 2D de longa distância.

Escadas como pontos de ligação entre segmentos A* em pisos diferentes.

Mapa cognitivo global (GlobalMonsterMap) como cache de transições e suporte a rotas alternativas.

Avaliação de custo total para escolha de escadas, considerando o percurso completo até ao líder.

Atalho multi‑piso: quando o líder está no mesmo piso mas inacessível, o monstro pode usar escadas para contornar o obstáculo.

Rotas alternativas com waypoints: quando o A* falha no mesmo piso, o monstro usa o grafo cognitivo para encontrar um caminho alternativo.

Seguimento imediato da escada do líder: se o jogador usou uma escada e o monstro está a ≤ 3 tiles da sua origem, o monstro segue‑o imediatamente.

Summons nunca desistem: continuam a perseguição mesmo após perder o mestre de vista, sem nunca tentar voltar a um ponto de spawn. Se ficarem completamente bloqueadas, um destravamento silencioso (teleporte invisível) as recoloca em movimento.

Monstros normais voltam ao seu ponto de origem (spawn) caminhando, utilizando as escadas que percorreram.

2. ARQUITETURA DA MÁQUINA DE ESTADOS
A máquina de estados é executada integralmente dentro do ciclo Monster::onThinkMCR(). Apenas um estado é processado por ciclo, pela seguinte ordem de prioridade:

text
┌──────────────────────────────────────────────┐
│ 1. COMBATE (prioridade máxima)                │
│    Se pode atacar → cancela tudo, ataca       │
├──────────────────────────────────────────────┤
│ 2. PERSEGUIÇÃO MULTI‑PISO (m_pursuingStairs)  │
│    Tenta alcançar m_stairOrigin               │
│    Gere travamentos, tenta movimento forçado   │
│    Monitoriza progresso, teleporta para escada │
├──────────────────────────────────────────────┤
│ 3. PLANEAMENTO DE ESCADA                      │
│    Se líder noutro piso → planStairTrajectory │
│    Se líder no mesmo piso e inacessível →     │
│      Avalia atalho multi‑piso vs rota alt.    │
├──────────────────────────────────────────────┤
│ 4. RETORNO AO SPAWN / CASA                   │
│    (Apenas monstros normais)                  │
│    processarRetornoSpawn / processarRetornoSimples │
│    Summons NUNCA entram neste estado           │
├──────────────────────────────────────────────┤
│ 5. IDLE                                       │
│    Se sem alvo → tenta follow, deambula ou    │
│    retorna ao spawn (só normais)              │
│    Inclui destravamento silencioso para summons│
└──────────────────────────────────────────────┘
Regra de ouro: O BFS do mapa cognitivo nunca toma o controlo primário. É sempre uma ferramenta auxiliar para planear uma rota até um alvo. Enquanto uma rota BFS está ativa, a máquina de estados continua a verificar se o jogador ficou acessível por A*.

4. FUNÇÕES CENTRAIS
4.1 Monster::onThinkMCR()
Função principal chamada a cada ciclo de pensamento (1 Hz). Implementa a máquina de 5 estados descrita na secção 2.

Pré‑processamento:

Safeguard do auto‑walk: corrige fuga de estado isWalkingToPosition quando a fila de passos está vazia.

Limpeza de escadas falhadas: remove escadas cujo cooldown expirou.

Limpeza de obstáculos temporários no GlobalMonsterMap (a cada 5s).

Obtenção do líder: para summons, getMaster(); para hostis, m_attackedCreature ou m_persistentTargetPlayer.

Monitorização de waypoints: se m_followingWaypoints ativo, calcula distância ao líder; se preso há ≥ 2 ciclos, limpa waypoints.

Estados (executados sequencialmente, apenas um por ciclo):

COMBATE – Se o monstro tem um alvo atacável, inicia follow direto.

PERSEGUIÇÃO MULTI‑PISO (m_pursuingStairs == true)

Cancela perseguição se o líder voltou ao mesmo piso e a escada não é um atalho (m_multiFloorShortcut == false).

Movimento para a escada:

Distância 1: movimento direto (internalMoveCreature); se falhar 3×, teleporta para o tile.

Distância > 1 e não está a caminhar: tenta A* para a escada; se falhar, procura tile adjacente; se tudo falhar, movimento forçado na direção da escada.

Monitorização de progresso: se isWalkingToPosition(), reseta contador de stuck; senão, se a distância diminuiu, reseta; caso contrário, incrementa m_stairStuckCount. Após 5 ciclos preso, abandona a escada e regista em m_recentlyFailedStairs.

Chegada à escada: se myPos == m_stairOrigin, efetua a transição (nativa ou ativa). Após transição, verifica se o líder ficou acessível; se sim, inicia follow; senão, limpa perseguição e permite reavaliação no próximo ciclo.

PLANEAMENTO DE ESCADA – Se não está em perseguição nem retorno e o líder existe:

Líder noutro piso: chama planStairTrajectory (throttle 2.5 s).

Líder no mesmo piso mas inacessível via A*:

Verifica se o A* nativo encontra caminho (leaderAcessivel). Se sim, não faz nada.
Se inacessível:
Calcula custo do atalho multi‑piso (evaluateMultiFloorCost).
Calcula custo da rota alternativa (findNearestApproachTo).
Se multiFloorCost < alternativeCost e multiFloorCost < INT_MAX → inicia planStairTrajectory imediatamente.
Caso contrário, se alternativeCost < INT_MAX, tenta tryAlternativeRouteToLeader (throttle 5 s).
Se a rota alternativa já estava ativa mas presa há ≥ 3 ciclos (m_followingWaypoints && m_waypointStuckCycles >= 3), força o atalho (limpa waypoints e chama planStairTrajectory).
RETORNO AO SPAWN / CASA – Apenas monstros normais. Summons nunca entram neste estado (são barradas no início do bloco). Se o líder reaparecer a ≤ 10 tiles, o retorno é cancelado e o monstro reengaja. O retorno usa processarRetornoSimples (caminhada) ou processarRetornoSpawn (escadas).

IDLE – Se não está a perseguir nem a retornar:

Se m_followingWaypoints ativo, chama walkToWaypoint para o waypoint atual. Inclui destravamento silencioso para summons (ver secção 7).

Caso contrário, tenta follow normal, deambulação ou retorno ao spawn (apenas normais, se distToSpawn > 10).

4.2 Monster::planStairTrajectory(const Position& leaderPos)
Planeia a melhor escada para alcançar o líder.

Fluxo detalhado:

Obtenção do líder (com fallback para getTargetList e jogadores no mesmo piso).

Regra de seguimento imediato da escada do líder:

Percorre as transições do jogador.

Condições: origem no piso do monstro, destino no piso do líder, distância ≤ 3 tiles, não falhada recentemente, válida (canStairBeUsed).

Se encontrada, inicia perseguição imediata (movimento direto se dist=1, A* se dist>1, movimento forçado se A* falhar).

Rota global completa:

Tenta GlobalMonsterMap::findPath(myPos, leaderPos).

Se falhar, faz scans rápidos (scanPlayerTrail) ao redor do monstro e do líder e tenta de novo.

Se encontrar:

Procura a primeira transição de piso no caminho.

Se existir, inicia perseguição para essa escada.

Se não existir (caminho todo no mesmo piso), inicia waypoints.

Fallback heurístico (quando não há rota global):

Recolhe escadas candidatas: transições do grafo, transições do líder, e (se vazio) findStairNearReal.

Para cada candidata:

Valida direção (não oposta ao líder), não falhada recentemente, válida (canStairBeUsed).

Calcula A* até à origem da escada.

Estima distância do destino ao líder: usa findNearestApproachTo (com scan extra se falhar).

Custo = |A\*| + 1 + distFromDest, com bónus de -1 para escadas do líder.

Escolhe a de menor custo e inicia perseguição.

Se nenhuma escada viável, inicia retorno ao spawn (apenas normais).

4.3 Monster::evaluateMultiFloorCost(const Position& leaderPos)
Versão "seca" do planStairTrajectory que apenas calcula o custo da melhor escada, sem iniciar perseguição. Usada no Bloco 3 para comparar atalho multi‑piso com rota alternativa.

Fluxo detalhado:

Obtenção do líder: idêntica à do planStairTrajectory, com fallback para getTargetList e jogadores no mesmo piso.

Recolha de candidatas: transições do grafo (getAllTransitions), transições do líder e, se vazio, findStairNearReal.

Avaliação de cada candidata:

Verifica deltaZ e direção (não oposta ao líder).

Verifica se não está em m_recentlyFailedStairs.

Valida com canStairBeUsed (exceto escadas do líder, que têm bypass).

Calcula A* até à origem (getPathTo).

Estima distância do destino ao líder: primeiro tenta findNearestApproachTo; se falhar, faz scanPlayerTrail(dest, 8) e tenta de novo.

Custo = |A\*| + 1 + distFromDest, com bónus de -1 para escadas do líder.

Devolve o menor custo encontrado, ou INT_MAX se nenhuma escada for viável.

Não modifica nenhum estado do monstro – é apenas uma função de avaliação.

4.4 Monster::walkToWaypoint(const Position& target)
Inicia a caminhada para um waypoint.

Melhorias face à versão original:

Se o alvo é a própria posição do monstro, avança para o próximo waypoint.

Tenta A* nativo; se falhar e o alvo estiver a 1 tile, tenta movimento forçado (internalMoveCreature).

Se o movimento forçado falhar, invalida o tile e avança para o próximo waypoint (ou limpa a rota se for o último). Isto evita que o monstro fique eternamente a tentar o mesmo tile bloqueado.

Tenta tiles adjacentes ao alvo.

Fallback para mapa cognitivo (BFS) e aproximação máxima.

4.5 Monster::tryAlternativeRouteToLeader()
Encontra uma rota alternativa no mesmo piso usando o mapa cognitivo.

Verifica se o líder está no mesmo piso (senão, aborta).

Garante que o próprio tile está marcado como caminhável.

Efetua scan rápido se necessário.

Usa findNearestApproachTo para obter uma aproximação ao líder.

Converte o caminho em waypoints e força o primeiro passo imediatamente.

4.6 Monster::clearStairPursuit()
Limpa o estado de perseguição, restaura o follow original e faz reset dos flags.

Importante: Não zera m_pursuitDeadline, permitindo que a perseguição continue enquanto o líder estiver visível. Também faz reset de m_multiFloorShortcut.

4.7 Monster::clearWaypoints()
Limpa a lista de waypoints e faz reset dos contadores associados, incluindo m_waypointStuckCycles e m_lastDistToLeaderWaypoints.

4.8 Monster::canStairBeUsed(const Position& origin, int deltaZ)
Valida se uma escada pode ser usada na direção pretendida.

Algoritmo completo:

Obtém o tile na posição origin. Se não existir ou não tiver flag TILESTATE_FLOORCHANGE, retorna false.

Verifica se o tile tem um destino explícito (getDestination()). Se sim, compara o deltaZ do destino com o deltaZ pretendido.

Se não tem destino explícito, verifica as sub‑flags de direção:

TILESTATE_FLOORCHANGE_DOWN: a escada sempre desce (Z aumenta).

TILESTATE_FLOORCHANGE_NORTH, SOUTH, EAST, WEST e variantes: a escada sempre sobe (Z diminui).

Se o tile não tem nenhuma sub‑flag de direção, é considerado bidirecional (buracos, etc.) e retorna true.

Se tem sub‑flags, valida conforme a direção pretendida:

deltaZ < 0 (subir) → requer flags de subida.

deltaZ > 0 (descer) → requer flag FLOORCHANGE_DOWN.

Retorna true se a direção coincide; caso contrário, false.

4.9 Monster::findStairNearReal(const Position& leaderPos, int radius)
Procura escadas reais no mapa ao redor do monstro (scan no mesmo piso).

Algoritmo completo:

Itera todos os tiles num raio de radius (padrão 15) ao redor da posição do monstro, no mesmo piso.

Para cada tile:

Verifica se tem flag TILESTATE_FLOORCHANGE.

Determina deltaZ com base nas sub‑flags:

FLOORCHANGE_DOWN → deltaZ = 1 (desce).

FLOORCHANGE_NORTH, SOUTH, EAST, WEST e variantes → deltaZ = -1 (sobe).

Sem sub‑flags: assume deltaZ com base na diferença de Z entre líder e monstro (targetDeltaZ).

Valida com canStairBeUsed(origin, deltaZ).

Calcula destino (dest.z = origin.z + deltaZ).

Custo = distância Manhattan até à escada + distância Manhattan do destino ao líder.

Mantém a escada com menor custo.

Retorna a posição da melhor escada encontrada, ou Position(0,0,0) se nenhuma.

4.10 processarRetornoSimples() e processarRetornoSpawn()
processarRetornoSimples() – Usado quando o monstro está no mesmo piso que m_homePosition.

Tenta A* direto até casa.

Se falhar, usa waypoints do mapa cognitivo.

Timeout de 15 segundos: se exceder, força teleporte para casa (apenas para normais).

Se não conseguir chegar após 10 falhas totais, desiste e deambula.

processarRetornoSpawn() – Usado quando o monstro tem um caminho de escadas (m_returnPath) para voltar.

Percorre m_returnPath no sentido inverso.

Para cada transição, tenta caminhar até à origem da escada e usá‑la.

Timeout de caminhada de 5s por etapa.

Após 15 falhas totais, reinicia o percurso.

Se ficar preso no mesmo tile 3×, invalida o tile e avança.

5. MECANISMOS DE DECISÃO
5.1 Custo Total da Rota
A escolha da escada é baseada no custo total estimado até ao líder:

text
custo = distância A* até à escada + 1 (transição) + distância estimada do destino ao líder
A distância do destino ao líder é calculada via findNearestApproachTo (se disponível) ou distância Manhattan como fallback. Se o grafo não tiver dados suficientes, é feito um mini‑scan (scanPlayerTrail) ao redor do destino antes de reavaliar.

5.2 Atalho Multi‑Piso vs Rota Alternativa
Quando o líder está no mesmo piso mas inacessível via A* (Bloco 3):

Calcula-se o custo do atalho multi‑piso (evaluateMultiFloorCost). Este custo considera usar uma escada para sair do piso, caminhar noutro andar e regressar ao piso do líder.

Calcula-se o custo da rota alternativa (findNearestApproachTo), que permanece no mesmo piso.

Se o atalho for estritamente menor, inicia perseguição multi‑piso imediatamente (sem esperar por throttle adicional).

Se a rota alternativa for viável, tenta‑a (com throttle de 5 s para evitar chamadas excessivas).

Se a rota alternativa estiver ativa mas presa há ≥ 3 ciclos, força o atalho (limpa waypoints e chama planStairTrajectory).

Isto garante que o monstro escolhe sempre o caminho com menor número total de passos, podendo usar escadas para contornar obstáculos mesmo quando o líder está no mesmo piso.

5.3 Seguimento Imediato da Escada do Líder
No início do planStairTrajectory, é executada uma verificação rápida:

Se existir uma transição do jogador que:

Tenha origem no piso atual do monstro.

Tenha destino no piso atual do líder.

Esteja a ≤ 3 tiles de distância do monstro.

Não esteja na lista de escadas falhadas recentemente.

Seja válida segundo canStairBeUsed.

→ O monstro inicia a perseguição por essa escada imediatamente, sem avaliar outras opções. Isto garante que, quando o jogador usa uma escada próxima, o monstro o segue de imediato.

5.4 Anti‑Ping‑Pong
O flag m_multiFloorShortcut é ativado no planStairTrajectory quando leaderPos.z == myPos.z (líder no mesmo piso que o monstro no momento do planeamento). Isto protege a perseguição de ser cancelada quando o líder "volta" ao mesmo piso (pois o monstro está deliberadamente a contornar um obstáculo). Para perseguições normais (líder noutro piso), o flag é false.

O flag é limpo em clearStairPursuit.

Escadas que falham (abandonadas por falta de progresso) são registadas em m_recentlyFailedStairs com cooldown de 10 segundos (reduzido de 30s nas versões anteriores para permitir reavaliação mais rápida).

6. TRATAMENTO DE SUMMONS VS HOSTIS
Funcionalidade	Summons	Hostis
Iniciar perseguição	Quando o mestre muda de piso	Quando o alvo muda de piso
Perda de líder (5 ticks)	Continua perseguição normal (nunca desiste)	Inicia retorno ao spawn
Deadline expirado	Continua perseguição normal	Inicia retorno ao spawn
Retorno ao spawn	Nunca – continua perseguição	Sim, caminha de volta
Destravamento silencioso	Sim, se presa ≥ 3 ciclos (teleporte invisível para waypoint)	Não
Mapa cognitivo	Sim, para seguir o mestre	Sim, para atalhos e cercamentos
Cancelamento ao receber dano	Sim (limpa perseguição)	Sim
isInSpawnRange	Retorna sempre true	Verifica distância real ao spawn
6.1 Porque as summons nunca retornam ao spawn
No onThinkMCR, quando uma summon perde o mestre de vista (m_sightLostTicks >= 5) ou o deadline expira (now > m_pursuitDeadline), o código executa:

clearStairPursuit() – limpa perseguição ativa.

setFollowCreature(nullptr) e setAttackedCreature(nullptr) – remove follow direto.

m_sightLostTicks = 0 – reseta o contador para reavaliar assim que o mestre reaparecer.

Não define m_returningHome = true – ao contrário dos hostis.

Retorna (m_inOnThinkMCR = false).

No ciclo seguinte, a summon volta ao Bloco 3 (planeamento) ou Bloco 5 (waypoints), continuando a perseguição normal.

6.2 Destravamento silencioso (apenas summons)
Ver secção 7.3.

7. SISTEMA DE WAYPOINTS E DESTRAVAMENTO SILENCIOSO
7.1 Rotas Alternativas com Waypoints
Quando o A* nativo falha no mesmo piso, o sistema chama tryAlternativeRouteToLeader() para obter um caminho do mapa cognitivo. Esse caminho é convertido numa lista de waypoints (m_waypoints), e o monstro segue‑os chamando walkToWaypoint a partir do Bloco 5 (IDLE).

7.2 Monitorização de Progresso
A cada ciclo, enquanto m_followingWaypoints está ativo, o sistema calcula a distância Manhattan ao líder. Se a distância diminuir, o contador m_waypointStuckCycles é resetado. Se não diminuir durante 2 ciclos, os waypoints são limpos (clearWaypoints).

7.3 Destravamento Silencioso (apenas Summons)
Condições:

É uma summon.

Está a seguir waypoints (m_followingWaypoints == true).

m_waypointStuckCycles >= 3.

Ação:

Obtém um caminho do findNearestApproachTo entre a posição atual e o líder (ou a última posição conhecida).

Se o caminho não estiver vazio:

Teleporta silenciosamente para o primeiro tile desse caminho (g_game().internalTeleport).

Limpa os waypoints antigos.

Define os novos waypoints como o caminho obtido, começando no índice 1 (já que está no primeiro).

Ativa m_followingWaypoints = true.

Reseta m_waypointStuckCycles = 0.

Se o caminho estiver vazio, limpa os waypoints e tentará de novo no próximo ciclo.

Nota: Este teleporte não tem efeitos visuais e é apenas uma medida de segurança para evitar que summons fiquem permanentemente bloqueadas. Nenhum jogador vê a imersão ser quebrada.

8. INTEGRAÇÃO COM O MOTOR – Tile::queryAdd
Para que o A* nativo pudesse traçar rotas até escadas e o monstro conseguisse pisá‑las, foram necessárias correções no método Tile::queryAdd.

8.1 Correção das máscaras de bits
As verificações originais usavam o operador | (bitwise OR) em hasFlag, que exige que todos os bits estejam presentes simultaneamente. Como TILESTATE_FLOORCHANGE e TILESTATE_TELEPORT são mutuamente exclusivos, as condições nunca disparavam.

Antes (errado):

cpp
hasFlag(TILESTATE_FLOORCHANGE | TILESTATE_TELEPORT)
Depois (corrigido):

cpp
hasFlag(TILESTATE_FLOORCHANGE) || hasFlag(TILESTATE_TELEPORT)
8.2 Pathfinding (FLAG_PATHFINDING)
Quando o motor está a calcular uma rota A*:

Só permite que o A* considere um tile de escada/teleporte como destino se o monstro estiver em perseguição (isPursuingStairs()) e for a escada planeada (getStairOrigin() == getPosition()).

Isto impede que o A* use escadas indevidamente durante a navegação normal.

8.3 Movimento real (sem FLAG_PATHFINDING)
Quando o monstro tenta pisar um tile:

Se o monstro está em perseguição e o tile é exatamente a escada planeada (monster->isPursuingStairs() && monster->getStairOrigin() == getPosition()), a entrada é permitida.

Caso contrário, tiles com TILESTATE_FLOORCHANGE ou TILESTATE_TELEPORT são bloqueados para monstros que não estejam em perseguição direcionada.

9. MAPA COGNITIVO GLOBAL (GlobalMonsterMap)
O GlobalMonsterMap é um grafo partilhado por todos os monstros que armazena tiles caminháveis e transições de piso. É implementado como um singleton com acesso thread‑safe (std::shared_mutex).

9.1 Alimentação do Grafo
Transições do jogador: injetadas via addStairTransition.

scanPlayerTrail: mini‑scan de raio 10 tiles e profundidade Z=1.

Scans reativos do monstro: tryAlternativeRouteToLeader faz scans ao redor do monstro e do líder.

Descoberta automática de escadas: durante scanArea, tiles com flag TILESTATE_FLOORCHANGE geram automaticamente transições.

9.2 Persistência
Gravação: em thread separada para data/world/global_monster_map.bin. Apenas tiles estáticos (dynamic == false) são salvos. As transições estáticas também são guardadas.

Carregamento: no arranque do servidor (Game::loadMap). Após a leitura, validateTransitions() remove transições inválidas.

Limpeza de emergência: se o ficheiro estiver corrompido, basta apagar o .bin e o servidor iniciará com o grafo vazio.

9.3 Invalidação Reativa
invalidateTileBlocked(): marca tiles como obstáculos temporários (30 s) ou permanentes (BLOCKSOLID).

cleanupTemporaryObstacles(): chamada a cada 5 segundos para revalidar tiles expirados.

Validação preguiçosa: isWalkableUnlocked() consulta o estado real do jogo para tiles marcados como dynamic.

10. SISTEMA DE ENGAJAMENTO E DEADLINE
10.1 Barra de Engajamento (m_engagement, 0–100)
Sobe: +2 ao causar ou receber dano do jogador; +1 por segundo enquanto o jogador está visível.

Desce: -3 por segundo sem ver o jogador.

10.2 Cálculo do Deadline (computePursuitDeadline)
cpp
uint64_t Monster::computePursuitDeadline() {
    int baseTimeMs = 30000;                         // 30 segundos base
    int engagementTimeMs = m_engagement * 500;      // até 50 segundos adicionais (engagement 100)
    int totalMs = std::min(60000, baseTimeMs + engagementTimeMs); // máximo de 60 segundos
    return getCurrentTimeMs() + totalMs;
}
O deadline é recalculado no início da perseguição e renovado enquanto o líder está visível.

Se o engagement aumentar significativamente durante a perseguição, o deadline é estendido.

10.3 Timeout por Perda de Visão
Hostis: 10 ciclos (10 segundos) sem ver o líder → iniciam retorno ao spawn.

Summons: 5 ciclos (5 segundos) sem ver o mestre → continuam perseguição normal (nunca retornam).

14. COMPORTAMENTO ESPERADO DOS MONSTROS
Perseguição multi‑piso: segue o jogador através de escadas usando a fila de transições e o grafo cognitivo.

Múltiplas mudanças de piso: pode planear rotas que atravessam vários andares para cercar o jogador.

Uso de escadas nunca pisadas pelo jogador: escolhe qualquer escada do grafo cognitivo que minimize o custo total.

Atalho multi‑piso: se o caminho direto está bloqueado no mesmo piso, usa escadas para contornar o obstáculo, desde que o custo total seja menor.

Rota alternativa: quando o A* falha, usa o mapa cognitivo para encontrar um caminho alternativo no mesmo piso.

Seguimento imediato: se o jogador usa uma escada e o monstro está a ≤ 3 tiles, segue‑o instantaneamente.

Reação a obstáculos: tiles bloqueados são invalidados no grafo; o monstro recalcula a rota.

Summons nunca desistem: continuam a perseguição mesmo após perder o mestre de vista, com destravamento silencioso em último caso.

Hostis retornam ao spawn: caminham de volta usando escadas; se o líder reaparecer a ≤ 10 tiles, cancelam o retorno.

Anti‑ping‑pong: não ficam presos a subir e descer a mesma escada repetidamente.
