1. VISÃO GERAL
O sistema permite que criaturas hostis e summons persigam um jogador através de escadas, buracos, teleportes e rampas. A base do sistema é:

A nativo do motor* para navegação 2D de longa distância.

Escadas como pontos de ligação entre segmentos A* em pisos diferentes.

Mapa cognitivo global (GlobalMonsterMap) como cache de transições e suporte a rotas alternativas.

Avaliação de custo total (holística) para escolha de escadas, considerando o percurso completo até ao líder.

Atalho multi‑piso: quando o líder está no mesmo piso mas inacessível, o monstro pode usar escadas para contornar obstáculos.

Rotas alternativas com waypoints: quando o A* falha no mesmo piso, usa o grafo cognitivo para encontrar um caminho alternativo.

Seguimento imediato da escada do líder: se o jogador usou uma escada e o monstro está a ≤3 tiles da sua origem, segue‑o imediatamente.

Sistema híbrido de planeamento: após cada transição de piso, recalcula a rota global e substitui a rota original se a nova for ≥20% mais curta.

Anti‑ping‑pong por penalização: escadas usadas recentemente entram em cooldown de 20 segundos e recebem um acréscimo de +10000 ao custo, mas não são bloqueadas totalmente – podem ser usadas como último recurso.

Summons nunca desistem: continuam a perseguição mesmo após perder o mestre de vista, sem nunca tentar voltar ao spawn. Se ficarem completamente bloqueadas, um destravamento silencioso (teleporte invisível) as recoloca em movimento.

Monstros normais voltam ao seu ponto de origem (spawn) caminhando, utilizando as escadas que percorreram.

1.1 Tipos de Criaturas e Comportamentos
Tipo	Perseguição	Timeout sem ver líder	Retorno ao spawn	Mapa cognitivo	Limite por líder
Hostil	Persegue o jogador que está a atacar	10 ciclos (10s)	Caminha de volta usando escadas	Sim, para atalhos e rotas alternativas	15 (não conta summons)
Summon	Segue o mestre através de escadas	5 ciclos (5s) → continua perseguição	Nunca retorna; continua perseguição; destravamento silencioso	Sim, para regressar ao mestre	Ilimitado
Jogador	Navegação multi‑piso via GlobalMonsterMap	N/A	N/A	Sim	Ilimitado

1.2 Crowd Control e Jitter
Para evitar picos de processamento quando múltiplos monstros perseguem o mesmo líder:
- Jitter aleatório de 100-500ms é adicionado ao throttle de planeamento (tryPlanStairsIfNeeded)
- Cada monstro tem um atraso pseudo-aleatório, evitando que todos planeiem no mesmo frame
- O registro de perseguidores é mantido no GlobalMonsterMap (pursuerCount, stairOverload)
- Escadas com 5+ perseguidores são consideradas congestionadas e recebem cooldown temporário
- O monstro então busca rotas alternativas naturalmente via planStairTrajectory
2. ARQUITETURA DA MÁQUINA DE ESTADOS
A máquina de estados é executada integralmente dentro do ciclo Monster::onThinkMCR(). Apenas um estado é processado por ciclo, pela seguinte ordem de prioridade:

text
┌─────────────────────────────────────────────────────────────────┐
│ 1. COMBATE (prioridade máxima)                                   │
│    Se pode atacar → cancela tudo, ataca                          │
├─────────────────────────────────────────────────────────────────┤
│ 2. PERSEGUIÇÃO MULTI‑PISO (m_pursuingStairs)                     │
│    - Verifica se já pode atacar (ranged) → cancela escada        │
│    - Tenta alcançar m_stairOrigin                                │
│    - Movimento forçado / A* / teleporte após falhas              │
│    - Monitoriza stuck (5 ciclos → abandona)                      │
│    - Chegou à escada? → efetua transição                         │
│    - Após transição: sistema híbrido (replaneia rota)            │
├─────────────────────────────────────────────────────────────────┤
│ 3. PLANEAMENTO DE ESCADA                                         │
│    Se líder noutro piso → planStairTrajectory (throttle 2.5s)    │
│    Se líder no mesmo piso e inacessível:                         │
│        Avalia atalho multi‑piso (evaluateMultiFloorCost)         │
│        Avalia rota alternativa (tryAlternativeRouteToLeader)     │
│        Escolhe a de menor custo                                  │
├─────────────────────────────────────────────────────────────────┤
│ 4. RETORNO AO SPAWN / CASA                                       │
│    (Apenas monstros normais)                                     │
│    processarRetornoSimples / processarRetornoSpawn               │
│    Summons NUNCA entram neste estado                             │
├─────────────────────────────────────────────────────────────────┤
│ 5. IDLE                                                          │
│    Se sem alvo → tenta follow, deambula ou retorna ao spawn      │
│    (só normais). Inclui destravamento silencioso para summons    │
└─────────────────────────────────────────────────────────────────┘
Regra de ouro: O BFS do mapa cognitivo nunca toma o controlo primário. É sempre uma ferramenta auxiliar para planear uma rota até um alvo. Enquanto uma rota BFS está ativa, a máquina de estados continua a verificar se o jogador ficou acessível por A*.

4. FUNÇÕES CENTRAIS (detalhadas)
4.1 Monster::onThinkMCR()
Função principal chamada a cada ciclo de pensamento (1 Hz). Implementa a máquina de 5 estados.

Pré‑processamento:

Corrige fuga de estado isWalkingToPosition quando a fila de passos está vazia.

Limpa escadas com cooldown expirado (m_recentlyFailedStairs, m_recentlyUsedStairs).

Limpa obstáculos temporários no GlobalMonsterMap (a cada 5s).

Obtém o líder (getMaster() para summons, m_attackedCreature/m_persistentTargetPlayer para hostis).

Monitoriza waypoints: se m_followingWaypoints ativo, calcula distância ao líder; se preso há ≥2 ciclos, limpa waypoints.

Estados (executados sequencialmente):

COMBATE – Se tem alvo atacável, inicia follow direto e ataca.

PERSEGUIÇÃO MULTI‑PISO (m_pursuingStairs == true)

Se canUseAttack() (ranged) → cancela escada e ataca.

Se líder voltou ao mesmo piso e !m_multiFloorShortcut → cancela perseguição.

Movimento para a escada:

Distância 1: internalMoveCreature; se falhar 3× → teleporta para o tile.

Distância >1 e não está a caminhar: tenta A* até m_stairOrigin; se falhar, procura tile adjacente; se tudo falhar, movimento forçado.

Monitorização de progresso:

Se isWalkingToPosition() → reseta m_stairStuckCount.

Se a distância à escada diminuiu → reseta.

Senão incrementa m_stairStuckCount. Se ≥5 → abandona escada e regista em m_recentlyFailedStairs.

Chegada à escada (posição == m_stairOrigin):

Efetua transição (nativa ou ativa).

Sistema híbrido (ver secção 5.4):

Marca escada como usada (m_recentlyUsedStairs).

Remove primeira escada da m_stairQueue.

Recalcula rota global (GlobalMonsterMap::findPath) a partir da nova posição.

Se newCost < m_originalPathCost * 0.8 e a primeira escada da nova rota não estiver em cooldown → substitui m_stairQueue.

Se a fila não estiver vazia, inicia a próxima escada.

Se vazia e líder no mesmo piso → tenta follow nativo; senão limpa perseguição.

PLANEAMENTO DE ESCADA – Se não está em perseguição nem retorno e o líder existe:

Líder noutro piso → planStairTrajectory (throttle 2.5s).

Líder no mesmo piso mas inacessível via A*:

Calcula custo do atalho (evaluateMultiFloorCost) e da rota alternativa (tryAlternativeRouteToLeader).

Se multiFloorCost < alternativeCost e multiFloorCost < INT_MAX → planStairTrajectory imediato.

Senão se alternativeCost < INT_MAX → tenta tryAlternativeRouteToLeader (throttle 5s).

Se rota alternativa ativa mas presa há ≥3 ciclos → força atalho (clearWaypoints + planStairTrajectory).

RETORNO AO SPAWN – Apenas monstros normais. Summons são barradas. Se líder reaparecer a ≤10 tiles, cancela retorno. Usa processarRetornoSimples (caminhada) ou processarRetornoSpawn (escadas).

IDLE – Se não está a perseguir nem a retornar:

Se m_followingWaypoints → walkToWaypoint.

Caso contrário, tenta follow normal, deambulação ou retorno ao spawn (normais, se distToSpawn >10).

4.2 Monster::calculateStairCost(origin, dest, leaderPos)
Função auxiliar que calcula o custo holístico de uma escada candidata.
Passos:

Executa A* até origin. Se falhar → retorna INT_MAX.

Se distância(monstro, origin) ≤ 2 e a distância Manhattan ao líder não diminui → retorna INT_MAX - 1 (escada inútil).

Penalização por cooldown (anti‑ping‑pong):
Se a escada está em m_recentlyUsedStairs e o cooldown ainda não expirou → retorna distToStair + 10000.
(Isto torna o custo extremamente alto mas não bloqueia totalmente.)

Calcula o custo a partir de dest até leaderPos:

Tenta findPath(dest, leaderPos). Se falhar, faz scan (raio 12) e tenta novamente.

Se ainda falhar → usa distância Manhattan + 50 (penalização).

Se encontrar → costFromDest = tamanho_do_caminho - 1.

Retorna distToStair + costFromDest.

Código completo:

cpp
int Monster::calculateStairCost(const Position& origin, const Position& dest, const Position& leaderPos) {
    Position myPos = getPosition();

    // 1. Custo A* até à origem da escada
    std::vector<Direction> pathToStair;
    FindPathParams fpp;
    getPathSearchParams(getMonster(), fpp);
    fpp.minTargetDist = 0; fpp.maxTargetDist = 1;
    fpp.fullPathSearch = true; fpp.clearSight = false; fpp.maxSearchDist = 500;
    if (!getPathTo(origin, pathToStair, fpp) || pathToStair.empty())
        return std::numeric_limits<int>::max();

    int distToStair = static_cast<int>(pathToStair.size());

    // 2. Ignora escadas adjacentes que não reduzem distância Manhattan
    int manhattanBefore = std::abs(myPos.x - leaderPos.x) + std::abs(myPos.y - leaderPos.y);
    int manhattanAfter  = std::abs(dest.x - leaderPos.x) + std::abs(dest.y - leaderPos.y);
    if (distToStair <= 2 && manhattanAfter >= manhattanBefore)
        return std::numeric_limits<int>::max() - 1;

    // 3. Penalização por cooldown (anti‑ping‑pong) – sem bloqueio total
    auto itUsed = m_recentlyUsedStairs.find(origin);
    if (itUsed != m_recentlyUsedStairs.end() && getCurrentTimeMs() < itUsed->second) {
        return distToStair + 10000;   // custo muito alto, mas ainda somável
    }

    // 4. Custo a partir do destino da escada até ao líder
    int costFromDest = std::numeric_limits<int>::max();
    auto pathFromDest = g_globalMonsterMap().findPath(dest, leaderPos);
    if (pathFromDest.empty()) {
        g_globalMonsterMap().scanPlayerTrail(dest, 12);
        pathFromDest = g_globalMonsterMap().findPath(dest, leaderPos);
    }
    if (!pathFromDest.empty()) {
        costFromDest = static_cast<int>(pathFromDest.size()) - 1;
    } else {
        costFromDest = manhattanAfter + 50;
    }

    return distToStair + costFromDest;
}
4.3 Monster::planStairTrajectory(const Position& leaderPos)
Planeia a melhor escada para alcançar o líder. Três fases:

Fase 1 – Seguimento imediato da escada do líder
Percorre as transições do jogador.

Condições: origem no piso do monstro, destino no piso do líder, distância ≤3 tiles, não falhada recentemente.

Nota: escadas em cooldown não são filtradas nesta fase – o monstro pode seguir o líder imediatamente mesmo que a escada tenha sido usada há pouco tempo (prioridade máxima).

Se encontrada, inicia perseguição imediata.

Fase 2 – Rota global via GlobalMonsterMap::findPath
Tenta GlobalMonsterMap::findPath(myPos, leaderPos).

Se falhar, faz scans rápidos (raio 15 ao redor do monstro e do líder) e tenta novamente.

Se encontrar:

Extrai a primeira transição de piso do caminho.

Se existir, inicia perseguição para essa escada e guarda o resto da fila em m_stairQueue.

Se não existir (caminho todo no mesmo piso), inicia waypoints.

Guarda o custo total em m_originalPathCost.

Fase 3 – Avaliação holística de escadas individuais (fallback)
Recolhe candidatas: transições do grafo, transições do líder, e (se vazio) findStairNearReal (raio 20).

Para cada candidata:

Verifica deltaZ, direção (não oposta ao líder), cooldown de falha (m_recentlyFailedStairs).

Não filtra por m_recentlyUsedStairs – a penalização será aplicada dentro de calculateStairCost.

Valida com canStairBeUsed (exceto escadas do líder, que têm bypass).

Calcula A* até à origem (getPathTo).

Estima distância do destino ao líder: findNearestApproachTo (com scan se falhar).

Custo = calculateStairCost(origin, dest, leaderPos) – que já inclui a penalização de cooldown.

Bónus de -1 para escadas do líder (apenas se o custo base não estiver já muito alto, i.e., <10000).

Escolhe a de menor custo e inicia perseguição.

Nota importante: Se a melhor escada for a que está em cooldown, ela será escolhida somente se nenhuma outra escada tiver custo menor – garantindo que o monstro nunca fique sem opções.

4.4 Monster::evaluateMultiFloorCost(const Position& leaderPos)
Versão “seca” do planStairTrajectory que apenas calcula o melhor custo entre todas as escadas candidatas, sem iniciar perseguição. Usada no Bloco 3 para comparar atalho multi‑piso com rota alternativa.

Mesma recolha de candidatas e mesmas regras: não filtra por m_recentlyUsedStairs, usa calculateStairCost para obter o custo (incluindo penalização).

Devolve o menor custo (ou INT_MAX se nenhuma viável).

Não modifica nenhum estado do monstro.

4.5 Monster::walkToWaypoint(const Position& target)
Inicia a caminhada para um waypoint.
Melhorias:

Se target == posição atual → avança para o próximo waypoint.

Tenta A* nativo; se falhar e target a 1 tile → tenta movimento forçado.

Se movimento forçado falhar → invalida o tile e avança para o próximo waypoint (evita bloqueios eternos).

Tenta tiles adjacentes ao alvo.

Fallback para mapa cognitivo (BFS) e aproximação máxima.

4.6 Monster::tryAlternativeRouteToLeader()
Encontra uma rota alternativa no mesmo piso usando o mapa cognitivo.

Verifica se líder está no mesmo piso (senão aborta).

Garante que o próprio tile está marcado como caminhável.

Executa scan rápido se necessário.

Usa findNearestApproachTo para obter uma aproximação ao líder.

Converte o caminho em waypoints e força o primeiro passo imediatamente.

4.7 Monster::clearStairPursuit() e Monster::clearWaypoints()
clearStairPursuit(): limpa estado de perseguição, restaura follow original, reseta flags (m_pursuingStairs, m_multiFloorShortcut, etc.). Não zera m_pursuitDeadline.

clearWaypoints(): limpa m_waypoints, reseta m_waypointIndex, m_followingWaypoints = false, m_waypointStuckCycles = 0.

4.8 Monster::canStairBeUsed(const Position& origin, int deltaZ)
Valida se uma escada pode ser usada na direção pretendida.

Obtém o tile; se não existir ou não tiver TILESTATE_FLOORCHANGE → false.

Se tile tem destino explícito (getDestination()), compara deltaZ com o destino.

Caso contrário, verifica sub‑flags:

FLOORCHANGE_DOWN → sempre desce (Z+1).

FLOORCHANGE_NORTH/SOUTH/EAST/WEST → sempre sobe (Z-1).

Sem sub‑flags → bidirecional (buracos, etc.) → true.

Retorna true se a direção coincide com a sub‑flag, senão false.

4.9 Monster::findStairNearReal(const Position& leaderPos, int radius=20)
Procura escadas reais no mapa ao redor do monstro (scan no mesmo piso).

Itera todos os tiles num raio de 20 ao redor de myPos.

Para cada tile com TILESTATE_FLOORCHANGE:

Determina deltaZ pelas sub‑flags (ou targetDeltaZ se ausente).

Valida com canStairBeUsed.

Calcula destino (dest.z = origin.z + deltaZ).

Custo = distância Manhattan até à escada + distância Manhattan do destino ao líder.

Mantém a escada com menor custo e retorna-a, ou Position(0,0,0) se nenhuma.

4.10 processarRetornoSimples() e processarRetornoSpawn()
processarRetornoSimples() – usado quando monstro está no mesmo piso que m_homePosition:

Tenta A* direto até casa.

Se falhar, usa waypoints do mapa cognitivo.

Timeout de 15s → força teleporte para casa (apenas normais).

Após 10 falhas totais, desiste e deambula.

processarRetornoSpawn() – usado quando o monstro tem um caminho de escadas (m_returnPath):

Percorre m_returnPath no sentido inverso.

Para cada transição, caminha até à origem e usa a escada.

Timeout de 5s por etapa.

Após 15 falhas totais, reinicia o percurso.

Se preso no mesmo tile 3×, invalida o tile e avança.

5. MECANISMOS DE DECISÃO
5.1 Custo Total da Rota (Avaliação Holística)
A escolha da escada baseia‑se no custo total estimado até ao líder:

text
custo = distância A* até à escada + 1 (transição) + distância estimada do destino ao líder
A distância do destino ao líder é calculada via findNearestApproachTo (se disponível) ou Manhattan com penalização.
Se o grafo não tiver dados suficientes, é feito um mini‑scan (scanPlayerTrail) ao redor do destino antes de reavaliar.

5.2 Atalho Multi‑Piso vs Rota Alternativa (Bloco 3)
Quando o líder está no mesmo piso mas inacessível via A*:

Calcula‑se o custo do atalho multi‑piso (evaluateMultiFloorCost).
Este custo considera usar uma escada para sair do piso, caminhar noutro andar e regressar ao piso do líder.

Calcula‑se o custo da rota alternativa (findNearestApproachTo), que permanece no mesmo piso.

Se multiFloorCost < alternativeCost → inicia perseguição multi‑piso imediatamente.

Caso contrário, se alternativeCost viável → tenta rota alternativa (throttle 5s).

Se a rota alternativa estiver ativa mas presa por ≥3 ciclos → força o atalho (limpa waypoints e chama planStairTrajectory).

5.3 Seguimento Imediato da Escada do Líder (Fase 1 do planStairTrajectory)
Verificação rápida no início do planeamento:

Se existir uma transição do jogador que:

Origem no piso atual do monstro.

Destino no piso atual do líder.

Distância ≤3 tiles do monstro.

Não esteja em m_recentlyFailedStairs.

Seja válida segundo canStairBeUsed.

→ O monstro inicia a perseguição por essa escada imediatamente, sem avaliar outras opções.
(Escadas em cooldown são aceites nesta fase – a prioridade é não perder o líder.)

5.4 Sistema Híbrido de Planeamento (pós‑transição)
Após cada transição de piso (quando o monstro chega à escada e muda de andar), o sistema executa:

Marca a escada como usada (m_recentlyUsedStairs[origin] = now + USED_STAIR_COOLDOWN).

Remove a primeira escada da m_stairQueue (se existir).

Recalcula a rota global a partir da nova posição do monstro até ao líder, usando GlobalMonsterMap::findPath.

Obtém o custo da nova rota (newCost = número de passos totais).

Compara com o custo da rota original guardado em m_originalPathCost:

Se newCost < m_originalPathCost * 0.8 (ou seja, pelo menos 20% mais curta) e a primeira escada da nova rota não estiver em cooldown (m_recentlyUsedStairs) → substitui m_stairQueue pela nova rota e actualiza m_originalPathCost.

Caso contrário, mantém a rota original (se a fila ainda tiver escadas).

Se a fila não estiver vazia, inicia a próxima escada; se vazia e líder no mesmo piso, tenta follow nativo; senão, limpa perseguição e volta a planear.

Motivação: Evita que o monstro persista numa rota longa e ineficiente se, entretanto, o líder se moveu ou se uma rota mais curta foi descoberta.

5.5 Anti‑Ping‑Pong (actualizado – penalização de custo)
O sistema evita que o monstro fique a subir e descer repetidamente a mesma escada usando:

m_recentlyUsedStairs: mapa que regista o timestamp de expiração (20 segundos) para cada escada que o monstro acabou de usar.

calculateStairCost – quando uma escada está em cooldown, retorna distToStair + 10000 (em vez de um valor bloqueante). Isto torna o custo extremamente alto, mas não bloqueia completamente a escada.

Nas fases de planeamento (planStairTrajectory FASE 3 e evaluateMultiFloorCost), os continue que ignoravam escadas em cooldown foram removidos. Assim, a escada é sempre considerada, mas o seu custo elevado só a torna elegível se não existir nenhuma outra escada viável.

Na FASE 1 (seguimento imediato da escada do líder), as escadas em cooldown não são filtradas – o monstro pode segui‑las imediatamente, pois a prioridade é não perder o líder quando está muito próximo.

No sistema híbrido pós‑transição, a nova rota só substitui a original se a primeira escada não estiver em cooldown (evita trocar para uma rota que acabou de ser usada).

Comportamento resultante:
✅ O monstro evita repetir a mesma escada enquanto existirem alternativas.
✅ Se não houver absolutamente nenhuma outra escada (ou se a única escada disponível for a que está em cooldown), o monstro ainda pode usá‑la, evitando que fique preso.
✅ O sistema mantém a protecção contra ping‑pong sem criar situações de impasse.

6. TRATAMENTO DE SUMMONS VS HOSTIS
Funcionalidade	Summons	Hostis
Iniciar perseguição	Quando o mestre muda de piso	Quando o alvo muda de piso

6.1 Porque as summons nunca retornam ao spawn
No onThinkMCR, quando uma summon perde o mestre de vista (m_sightLostTicks >= 5) ou o deadline expira (now > m_pursuitDeadline):

Chama clearStairPursuit().

Define setFollowCreature(nullptr) e setAttackedCreature(nullptr).

Reseta m_sightLostTicks = 0.

Não define m_returningHome = true.

No ciclo seguinte, a summon volta ao Bloco 3 (planeamento) ou Bloco 5 (waypoints), continuando a perseguição normal.

6.2 Destravamento silencioso (apenas summons)
Condições:

É uma summon.

Está a seguir waypoints (m_followingWaypoints == true).

m_waypointStuckCycles >= 3.

Ação:

Obtém um caminho do findNearestApproachTo entre a posição atual e o líder (ou última posição conhecida).

Se o caminho não estiver vazio:

Teleporta silenciosamente para o primeiro tile desse caminho (g_game().internalTeleport).

Limpa os waypoints antigos.

Define os novos waypoints como o caminho obtido, começando no índice 1.

Reseta m_waypointStuckCycles = 0.

Se o caminho estiver vazio, limpa os waypoints e tentará de novo no próximo ciclo.

(Nota: Este teleporte não tem efeitos visuais – é apenas uma medida de segurança.)

7. SISTEMA DE WAYPOINTS E DESTRAVAMENTO SILENCIOSO
7.1 Rotas Alternativas com Waypoints
Quando o A* nativo falha no mesmo piso, o sistema chama tryAlternativeRouteToLeader() para obter um caminho do mapa cognitivo. Esse caminho é convertido numa lista de waypoints (m_waypoints), e o monstro segue‑os chamando walkToWaypoint a partir do Bloco 5 (IDLE).

7.2 Monitorização de Progresso
A cada ciclo, enquanto m_followingWaypoints está ativo:

Calcula a distância Manhattan ao líder.

Se a distância diminuir → reseta m_waypointStuckCycles.

Se não diminuir durante 2 ciclos → limpa waypoints (clearWaypoints).

7.3 Destravamento Silencioso (apenas Summons) – já descrito na secção 6.2.
8. INTEGRAÇÃO COM O MOTOR – Tile::queryAdd
Para que o A* nativo pudesse traçar rotas até escadas e o monstro conseguisse pisá‑las, foram necessárias correções no método Tile::queryAdd.

8.1 Correção das máscaras de bits
Antes (errado): hasFlag(TILESTATE_FLOORCHANGE | TILESTATE_TELEPORT)
Depois (corrigido): hasFlag(TILESTATE_FLOORCHANGE) || hasFlag(TILESTATE_TELEPORT)

8.2 Pathfinding (modo FLAG_PATHFINDING)
Só permite que o A* considere um tile de escada/teleporte como destino se o monstro estiver em perseguição (isPursuingStairs()) e for a escada planeada (getStairOrigin() == getPosition()).

8.3 Movimento real (sem FLAG_PATHFINDING)
Se o monstro está em perseguição e o tile é exatamente a escada planeada → entrada permitida.

Caso contrário, tiles com TILESTATE_FLOORCHANGE ou TILESTATE_TELEPORT são bloqueados para monstros que não estejam em perseguição direcionada.

9. MAPA COGNITIVO GLOBAL (GlobalMonsterMap)
O GlobalMonsterMap é um grafo partilhado por todos os monstros que armazena tiles caminháveis e transições de piso. Implementado como singleton com acesso thread‑safe (std::shared_mutex).

9.1 Alimentação do Grafo
Transições do jogador (addStairTransition).

scanPlayerTrail: mini‑scan de raio 10 tiles e profundidade Z=1.

Scans reativos do monstro (tryAlternativeRouteToLeader).

Descoberta automática de escadas (scanArea): tiles com TILESTATE_FLOORCHANGE geram transições.

9.2 Persistência
Gravação em data/world/global_monster_map.bin (apenas tiles estáticos e transições estáticas).

Carregamento no arranque do servidor (Game::loadMap). Após leitura, validateTransitions() remove transições inválidas.

9.3 Invalidação Reativa
invalidateTileBlocked(): marca tiles como obstáculos temporários (30s) ou permanentes.

cleanupTemporaryObstacles(): chamada a cada 5s para revalidar tiles expirados.

Validação preguiçosa: isWalkableUnlocked() consulta o estado real do jogo para tiles dinâmicos.

10. SISTEMA DE ENGAJAMENTO E DEADLINE
10.1 Barra de Engajamento (m_engagement, 0–100)
Sobe: +2 ao causar ou receber dano do jogador; +1 por segundo enquanto o jogador está visível.

Desce: -3 por segundo sem ver o jogador.

10.2 Cálculo do Deadline (computePursuitDeadline)
cpp
uint64_t Monster::computePursuitDeadline() {
    int baseTimeMs = 30000;                         // 30 segundos base
    int engagementTimeMs = m_engagement * 500;      // até 50s adicionais (engagement 100)
    int totalMs = std::min(60000, baseTimeMs + engagementTimeMs); // máximo 60s
    return getCurrentTimeMs() + totalMs;
}
O deadline é recalculado no início da perseguição e renovado enquanto o líder está visível.

Se o engagement aumentar significativamente, o deadline é estendido.

10.3 Timeout por Perda de Visão
Hostis: 10 ciclos (10s) sem ver o líder → iniciam retorno ao spawn.

Summons: 5 ciclos (5s) sem ver o mestre → continuam perseguição normal (nunca retornam).

14. COMPORTAMENTO ESPERADO DOS MONSTROS
Perseguição multi‑piso: segue o jogador através de escadas usando a fila de transições e o grafo cognitivo.

Múltiplas mudanças de piso: pode planear rotas que atravessam vários andares para cercar o jogador.

Uso de escadas nunca pisadas pelo jogador: escolhe qualquer escada do grafo cognitivo que minimize o custo total.

Atalho multi‑piso: se o caminho direto está bloqueado no mesmo piso, usa escadas para contornar o obstáculo, desde que o custo total seja menor.

Rota alternativa: quando o A* falha, usa o mapa cognitivo para encontrar um caminho alternativo no mesmo piso.

Seguimento imediato: se o jogador usa uma escada e o monstro está a ≤3 tiles, segue‑o instantaneamente.

Sistema híbrido: após cada transição, compara a rota original com uma nova rota global; se a nova for ≥20% mais curta, substitui a rota.

Anti‑ping‑pong por penalização: evita repetir a mesma escada, mas permite‑a como último recurso (custos altos, não bloqueio).

Reação a obstáculos: tiles bloqueados são invalidados no grafo; o monstro recalcula a rota.

Summons nunca desistem: continuam a perseguição mesmo após perder o mestre de vista, com destravamento silencioso em último caso.

Hostis retornam ao spawn: caminham de volta usando escadas; se o líder reaparecer a ≤10 tiles, cancelam o retorno.

17. INTEGRAÇÃO COM O SISTEMA DE ATAQUE EFETIVO MULTI‑PISO
A partir de Junho de 2026, o sistema de perseguição passou a considerar
a distância efetiva de ataque (effectiveAttackDistance) como fator de
decisão para monstros em pisos diferentes.

17.1 effectiveAttackDistance
Calculada em src/mcr/core/monster_ai.cpp:

  effectiveAttackDistance = targetDistance + penaltyZ
  penaltyZ = deltaZ * (deltaZ > 0 ? 2.0f : 1.0f)

- Atirar para cima (deltaZ < 0): penalidade 2.0 por piso
- Atirar para baixo (deltaZ > 0): penalidade 1.0 por piso

A penalidade assimétrica reflete a dificuldade de atacar alvos em
andares superiores (cobertura, ângulo) vs. inferiores.

17.2 Impacto no getDistanceStep (monster.cpp)
O método getDistanceStep (usado no A* para determinar quando o monstro
parou de andar e começou a atacar) agora usa:

  if (effectiveDist > targetDistance) → continua a andar

Isto faz com que monstros em pisos diferentes caminhem para mais
perto do líder (ou para escadas) até que a distância efetiva seja
igual ou inferior ao alcance da arma, mesmo que a distância XY já
seja suficiente.

17.3 CROSS_FLOOR_RANGE_LIMIT = 10
Para spells de range 0 (corpo a corpo), foi adicionado um limite
absoluto de 10 tiles efetivos para ataques cross‑floor:

  if (range == 0) {
      return effectiveDist <= CROSS_FLOOR_RANGE_LIMIT;
  }

Isto impede que monstros ataquem de distâncias absurdas através de
pisos, mesmo com range 0.

17.4 Fallback de Escada no handleCombatState
Quando o líder está visível mas o monstro está fora do alcance efetivo
em piso diferente, handleCombatState chama tryPlanStairsIfNeeded como
fallback. Sem isto, o monstro via que o líder estava "ao alcance" pela
distância bruta mas nunca conseguia atacar, ficando parado.

17.5 Navegação do Jogador (Player Navigation)
A partir de Junho de 2026, os jogadores também podem usar o GlobalMonsterMap
para navegar entre pisos:

- Quando um jogador ataca ou segue uma criatura em outro piso,
  Player::navigateTo() calcula a rota via GlobalMonsterMap::findPath()
- O jogador caminha até a escada, teleporta automaticamente e continua
- A navegação é ativada em playerSetAttackedCreature e playerFollowCreature
- O sistema reutiliza a mesma lógica de stair pursuit dos monstros
- Futuramente: clique no minimapa para navegação entre regiões exploradas
