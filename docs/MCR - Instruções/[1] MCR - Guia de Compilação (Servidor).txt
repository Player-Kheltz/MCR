🎯 OBJETIVO DESTE GUIA
Fornecer todas as instruções, dependências, customizações e verificações necessárias para compilar o servidor Canary do Projeto MCR a partir do código‑fonte. Esta versão incorpora todos os métodos Lua exigidos pelo Sistema de Progressão do Aventureiro (SPA v4.7.0), incluindo:

Gerenciamento de afinidades em lote, histórico de ações, knockback, modificadores de velocidade e stealth.

getAmmo para projéteis dinâmicos.

getVocationBaseMaxHealth, getVocationBaseMaxMana, getNaturalSpeed e setMovementSpeed para atributos base sem acumulação.

setCombatModifier para modificadores de combate dinâmicos.

Métodos para o sistema de invocação e inventário remoto (openRemoteContainer, closeRemoteContainer, Monster.createInventory, Monster.addItemToInventory, Monster.getInventoryContainer, Monster.setCustomName, Monster.getCustomName).

Novos métodos e estruturas para perseguição multi‑piso nativa: Creature::onFloorChange, Player::storeFloorChange, Player::hasPendingFloorChange, Player::getNextFloorChange, Player::clearNextFloorChange, Monster::followTargetAcrossFloor, Monster::findFreeTileNear, e a captura automática de transições de piso em Game::internalTeleport e Map::moveCreature.

Estrutura modular de scripts em data-canary/scripts/MCR/SPA/core/ e subpastas, incluindo os novos módulos de pets e summons.

O assistente DEVE seguir estas directrizes ao modificar o código C++.

⚠️ REGRAS DE REVISÃO PARA O ASSISTENTE (C++ – SERVIDOR CANARY)
Ao modificar QUALQUER arquivo .cpp/.h no diretório Canary/src/, o assistente DEVE:

Garantir que TODAS as strings visíveis ao jogador (sendTextMessage, sendCancelMessage, logs de sistema) estejam em PT‑BR, com acentuação correcta. Os caracteres acentuados DEVEM ser escritos OBRIGATORIAMENTE como escapes octais Latin‑1 (ex.: "N\355vel", "Voca\347\343o"). NÃO é permitido usar UTF‑8 directamente nas strings; usar escapes octais garante portabilidade e evita problemas de compilador.

Manter comentários em PT‑BR, excepto em nomes de funções/variáveis que permanecem em inglês (camelCase ou snake_case).

Verificar se a alteração mantém compatibilidade com o protocolo 15.00 e não quebra bindings Lua existentes (registerMethod, registerClass).

Se adicionar novos métodos expostos ao Lua, seguir a convenção de nomenclatura minúsculas_com_underscore e incluir validação de parâmetros (número e tipo). Documentar cada novo método neste guia.

Se alterar comunicação de rede (ProtocolGame), garantir que opcodes estendidos estejam na faixa reservada (160–255) e sejam documentados.

Toda nova consulta SQL DEVE usar placeholders parametrizados (evitar concatenação).

Após a edição, listar explicitamente os arquivos modificados para compilação selectiva.

O sistema de progressão usa Revscript: os scripts Lua ficam na pasta data-canary/scripts/MCR/SPA/. O motor carrega‑os automaticamente; não é necessário alterar o C++ para registar scripts.

🔨 PROCEDIMENTO DE COMPILAÇÃO (PADRÃO)
Clone o repositório MCR (já contém os fontes customizados):

bash
git clone https://github.com/Player-Kheltz/ProjetoMCR.git
cd ProjetoMCR\Canary
Configure o CMake (substitua o caminho do vcpkg pelo seu):

bash
mkdir build
cd build
cmake .. -DCMAKE_TOOLCHAIN_FILE=C:/vcpkg/scripts/buildsystems/vcpkg.cmake -DVCPKG_TARGET_TRIPLET=x64-windows
Compile:

bash
cmake --build . --config Release
O executável será gerado em build\Release\canary.exe.

Copie o executável para a raiz do servidor (onde estão data/, config.lua) ou configure o output directamente com cmake --install . (se configurado).

🧬 CUSTOMIZAÇÕES MCR JÁ APLICADAS AO CÓDIGO‑FONTE
As seguintes modificações já estão presentes nos fontes do repositório ProjetoMCR/Canary. O assistente NÃO precisa reaplicá‑las, mas deve conhecê‑las para manutenção e depuração.

Console UTF‑8 (main.cpp): SetConsoleOutputCP(CP_UTF8) e SetConsoleCP(CP_UTF8).

Bindings Lua Preservados: nenhuma remoção de métodos. Apenas adições documentadas.

Remoção de restrições de vocação: vocation 0 permitido em itens e magias (item.cpp, player.cpp, spells.cpp).

Protocolo de Jogo (protocolgame.cpp): substituições de creature->getName() por creature->getCustomName() nas funções sendCreatureSay, sendToChannel e AddCreature, garantindo que o nome customizado do escudeiro seja exibido no cliente.

Perseguição Multi‑Piso Nativa (monster.cpp, player.cpp, creature.cpp, game.cpp, map.cpp):

Player::storeFloorChange, hasPendingFloorChange, getNextFloorChange, clearNextFloorChange: fila de transições de piso do jogador, usada pelos summons para seguir o mestre.

Monster::followTargetAcrossFloor: lógica de perseguição multi‑piso que processa a fila do jogador, movendo incrementalmente a criatura até ao ponto de origem e teleportando‑a para o destino.

Monster::findFreeTileNear: utilitário para encontrar um tile livre próximo a uma posição.

Captura automática em Game::internalTeleport e Map::moveCreature via onFloorChange.

Novos Registos Lua (spa_functions.cpp, monster_functions.cpp): registados os métodos Player.openRemoteContainer, Player.closeRemoteContainer, Monster.createInventory, Monster.addItemToInventory, Monster.getInventoryContainer, Monster.setCustomName, Monster.getCustomName, Monster.followTargetAcrossFloor.

SPA Manager (spa_manager.cpp / spa_manager.hpp): motor de eventos genérico (SPA.onEvent, SPA.dispatchEvent) e registo de domínios em tempo real via Lua.

⚠️ ARMADILHAS E LIÇÕES APRENDIDAS
Codificação de console: SetConsoleOutputCP(CP_UTF8) é imprescindível para logs em PT‑BR.

Tradução de logs: use SEMPRE escapes octais para todos os acentos nas strings C++. Não confie em UTF‑8 directo.

Conexão com MariaDB: verificar senha em config.lua e se o serviço está rodando.

Compilação com vcpkg: mantenha o vcpkg atualizado.

Escapes octais vs UTF‑8: use sempre escapes octais para portabilidade. Esta é uma regra absoluta no MCR.

Coluna lastlogin: EXISTE na tabela players do schema actual do Canary (confirmado no banco de dados do projecto). Pode ser usada para lógica de inicialização/migração.

Buffer de afinidade: o método addDominioAfinidadeBatch deve ser eficiente para evitar lag.

Estados de alma: use player:getAllActiveStates(); não itere manualmente sobre todos os domínios.

Cores: as definidas em texto_colorido.lua (v10.0) devem ser mantidas consistentes.

Projéteis dinâmicos: player:getAmmo() é essencial e deve ser preservado.

Modificadores de combate: setCombatModifier e getCombatModifier suportam a Sinergia Elemental.

Atributos base: os novos métodos getVocationBaseMaxHealth, getVocationBaseMaxMana, getNaturalSpeed e setMovementSpeed são utilizados pelas funções centrais recalcularVidaMana e recalcularVelocidade e NÃO devem ser alterados, pois garantem a estabilidade dos valores sem acumulação.

Inventário remoto: os métodos openRemoteContainer, closeRemoteContainer, createInventory, etc., devem ser usados apenas no contexto do escudeiro/pets. O closeRemoteContainer deve ser chamado quando a criatura morre para evitar janelas órfãs.

Nome customizado: setCustomName/getCustomName substituem setName/getName para criaturas SPA, garantindo que o nome exibido no cliente seja o nome do escudeiro e não o nome do arquivo do monstro.

Perseguição multi‑piso: a fila de transições é capturada automaticamente em Game::internalTeleport e Map::moveCreature. O método followTargetAcrossFloor deve ser chamado apenas no onThink_async do monstro. A fila é automaticamente limpa quando o monstro alcança o mestre no mesmo piso.
