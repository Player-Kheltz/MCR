⚠️ REGRAS DE REVISÃO PARA O ASSISTENTE (C++ – SERVIDOR CANARY)
Ao modificar QUALQUER arquivo .cpp/.h no directório Canary/src/, o assistente DEVE:

Garantir que TODAS as strings visíveis ao jogador (sendTextMessage, sendCancelMessage, logs de sistema) estejam em PT‑BR, com acentuação correcta. Os caracteres acentuados DEVEM ser escritos OBRIGATORIAMENTE como escapes octais Latin‑1 (ex.: "N\355vel", "Voca\347\343o"). NÃO é permitido usar UTF‑8 directamente nas strings; usar escapes octais garante portabilidade e evita problemas de compilador.

Manter comentários em PT‑BR, excepto em nomes de funções/variáveis que permanecem em inglês (camelCase ou snake_case).

Verificar se a alteração mantém compatibilidade com o protocolo 15.00 e não quebra bindings Lua existentes (registerMethod, registerClass).

Se adicionar novos métodos expostos ao Lua, seguir a convenção de nomenclatura minúsculas_com_underscore e incluir validação de parâmetros (número e tipo). Documentar cada novo método neste guia.

Se alterar comunicação de rede (ProtocolGame), garantir que opcodes estendidos estejam na faixa reservada (160–255) e sejam documentados.

Toda nova consulta SQL DEVE usar placeholders parametrizados (evitar concatenação).

Após a edição, listar explicitamente os arquivos modificados para compilação selectiva.

O sistema de progressão usa Revscript: os scripts Lua ficam na pasta data-canary/scripts/MCR/SPA/. O motor carrega‑os automaticamente; não é necessário alterar o C++ para registar scripts.

🔨 PROCEDIMENTO DE COMPILAÇÃO (PADRÃO)
Clone o repositório MCR (já contém os fontes customizados):

bash
git clone https://github.com/Player-Kheltz/ProjetoMCR.git
cd ProjetoMCR\Canary
Configure o CMake (substitua o caminho do vcpkg pelo seu):

bash
mkdir build
cd build
cmake .. -DCMAKE_TOOLCHAIN_FILE=C:/vcpkg/scripts/buildsystems/vcpkg.cmake -DVCPKG_TARGET_TRIPLET=x64-windows
Compile:

bash
cmake --build . --config Release
O executável será gerado em build\Release\canary.exe.

Copie o executável para a raiz do servidor (onde estão data/, config.lua) ou configure o output directamente com cmake --install . (se configurado).

📌 GlobalMonsterMap (v8.0)
Classe singleton (global_monster_map.hpp / .cpp) que mantém:

Grafo de tiles caminháveis (std::unordered_map<Position, TileInfo> m_tiles) com flags walkable e dynamic.

Transições de piso (std::unordered_map<Position, std::vector<Position>> m_transitions).

Obstáculos temporários (std::unordered_map<Position, TemporaryObstacle> m_temporaryObstacles).

std::shared_mutex m_mutex para acesso thread‑safe (leituras paralelas, escrita exclusiva).

Métodos principais:

addTile(pos, walkable, dynamic) – adiciona/actualiza um tile no grafo.

isWalkable(pos) – retorna true se o tile é caminhável (com validação preguiçosa para tiles dinâmicos).

scanArea(origin, radius, maxZUp, maxZDown, maxTiles, playerStairs, force) – flood‑fill multi‑piso que adiciona tiles caminháveis e transições. Se force=true, ignora o limite MAX_ACTIVE_SCANS (actualmente 20).

findPath(start, goal) – BFS multi‑piso (2000 nós máx.) que encontra rota entre dois pontos, usando transições de piso e expansão ao vivo (registerDiscoveredTileUnlocked). Suporta BFS bidireccional como fallback e bridgeCorridorUnlocked para conectar grafos no mesmo piso.

findNearestApproachTo(start, goal) – BFS que encontra o tile mais próximo do objectivo, útil quando o destino exacto está bloqueado.

addStairTransition(from, to) – regista uma transição de piso (escada/teleporte).

getDestinations(pos) – retorna destinos de transições a partir de uma posição.

getAllTransitions() – retorna todas as arestas (origem, destino) do grafo. Usado pela procura global de escadas.

addTemporaryObstacle(pos, durationMs) – marca um tile como bloqueado temporariamente (TTL).

cleanupTemporaryObstacles() – remove obstáculos expirados, revalidando o estado real do tile.

markTilePermanentlyBlocked(pos) – marca um tile como permanentemente bloqueado (ex.: parede).

validateTransitions() – remove transições inválidas (destinos que são paredes sólidas).

saveToFile(filename) – persistência assíncrona (grava apenas tiles estáticos e transições).

loadFromFile(filename) – carregamento no arranque do servidor.

scanPlayerTrail(center, radius) – mini‑scan (forçado) para alimentar o grafo com o rasto do jogador.

Alimentação do grafo:

Transições do jogador: injectadas directamente via addStairTransition quando o monstro consulta a fila do líder.

Scans periódicos: scanArea faz flood‑fill limitado, adicionando tiles caminháveis e transições. Descoberta automática de escadas: qualquer tile com TILESTATE_FLOORCHANGE gera transições para z±1 (ou para o destino exacto via Tile::getDestination). Os 8 vizinhos do destino são também mapeados.

Alimentação contínua: Monster::onWalkComplete adiciona o tile actual ao grafo.

Invalidação reactiva: invalidateTileBlocked actualiza o grafo, distinguindo bloqueios permanentes (BLOCKSOLID) de temporários (BLOCKPATH sem BLOCKSOLID).

Exploração passiva: tryExploreArea (chamada por monstros hostis) faz scans de raio 30, profundidade Z=3, a cada 30 segundos se a cobertura for inferior a 60%.

BFS Multi‑Piso (findPath):

Expande 8 direcções com validação de diagonais (evita cortar paredes).

Inclui transições de piso como arestas.

Permite alcançar o objectivo mesmo que bloqueado, se for uma escada (goalIsStair).

Regista automaticamente tiles desconhecidos via registerDiscoveredTileUnlocked (expansão ao vivo).

Se o BFS principal falhar, tenta BFS bidireccional (se existirem transições).

Se ainda falhar e start e goal estiverem no mesmo piso, tenta bridgeCorridorUnlocked (até 64 passos) para reconectar o grafo.

Se tudo falhar, retorna um caminho vazio e regista logs detalhados.

📌 Comportamento (v8.0)
Máquina de estados unificada (Monster::onThinkMCR) com prioridades:

Combate – se o líder está visível e atacável, cancela tudo, faz follow e ataca.

Retorno ao spawn/casa – se m_returningToSpawn ou m_returningHome activo, processa retorno (caminhando, nunca teleporta).

Seguir rota global (BFS) – se m_usingGlobalPath activo, segue o caminho planeado; cancela se o líder ficar acessível.

Perseguição activa por transições – tenta alcançar m_stairOrigin. Usa BFS reativo, atalho cognitivo periódico, cercamento (tryFlankingRoute) e fallback para waypoints de superfície.

Procurar próxima transição – checkStairPursuit(): primeiro tenta atalho pelo mapa cognitivo, depois procura global de escadas no grafo (getAllTransitions), e por último escolhe a melhor escada da fila do jogador.

Perseguição proactiva + cercamento – tryFlankingRoute (cercamento multi‑piso), tryAlternativeRouteToLeader, tryProactiveGlobalPursuit.

Ocioso / deambulação – se sem alvo, tenta seguir ou andar aleatoriamente perto do spawn.

Sistema de engajamento e deadline adaptativo:

m_engagement (0‑100) sobe: +2 ao causar ou receber dano do jogador; +1 por segundo com o jogador visível. Desce: -3 por segundo sem ver o jogador.

deadline = currentTime + (25000 + engagement × 350) ms (mín. 10s, máx. 45s). Enquanto o líder está visível, o deadline é constantemente renovado.

Timeout por perda de visão: 5 segundos consecutivos sem ver o líder → desistência e retorno ao spawn.

Anti‑ping‑pong:

Se a mesma escada for usada 3 vezes em menos de 5 segundos, é aplicado um cooldown longo de 45 segundos (FAILED_STAIR_COOLDOWN = 30000 ms), forçando o sistema a explorar outras alternativas.

Retorno sem teleportes:

Com escadas percorridas (m_returnPath): processarRetornoSpawn() percorre o caminho inverso, teleportando apenas nos tiles de escada.

Sem escadas: processarRetornoSimples() caminha directamente para m_homePosition.

Timeout de caminhada de 5 segundos – se a caminhada não progredir, cancela e avança.

Após 15 falhas totais no retorno por escadas, reinicia o percurso.

Após 10 falhas totais no retorno simples, desiste e deambula (tryIdleWander).

Cancelamento do retorno: se o líder ficar visível (≤ 25 tiles), o retorno é cancelado e o monstro reengaja.

Cancelamento imediato por dano:

Monster::onAttackedByPlayer limpa m_pursuingStairs, m_usingGlobalPath, m_returningToSpawn, m_returningHome, m_returnPath, e define m_defensiveRetaliation = true por 5 segundos.

Throttles independentes:

Procura de escadas (checkStairPursuit): 1500 ms.

BFS reativo (tryUseGlobalMap): 1000 ms (primeira tentativa), segunda tentativa com scan expandido a cada 10000 ms.

Cercamento (tryFlankingRoute): 600 ms (líder visível) / 1000 ms (líder invisível).

Perseguição proactiva (tryProactiveGlobalPursuit): 5000 ms (parado) / 10000 ms (perseguindo).

Exploração passiva (tryExploreArea): 30000 ms.

Idle wander (tryIdleWander): 3000 ms.

⚠️ ARMADILHAS E LIÇÕES APRENDIDAS
Codificação de console: SetConsoleOutputCP(CP_UTF8) é imprescindível para logs em PT‑BR.

Tradução de logs: use SEMPRE escapes octais para todos os acentos nas strings C++. Não confie em UTF‑8 directo.

Conexão com MariaDB: verificar senha em config.lua e se o serviço está rodando.

Compilação com vcpkg: mantenha o vcpkg actualizado.

Escapes octais vs UTF‑8: use sempre escapes octais para portabilidade. Esta é uma regra absoluta no MCR.

Coluna lastlogin: EXISTE na tabela players do schema actual do Canary (confirmado no banco de dados do projecto). Pode ser usada para lógica de inicialização/migração.

Buffer de afinidade: o método addDominioAfinidadeBatch deve ser eficiente para evitar lag.

Estados de alma: use player:getAllActiveStates(); não itere manualmente sobre todos os domínios.

Cores: as definidas em texto_colorido.lua (v10.0) devem ser mantidas consistentes.

Projécteis dinâmicos: player:getAmmo() é essencial e deve ser preservado.

Modificadores de combate: setCombatModifier e getCombatModifier suportam a Sinergia Elemental.

Atributos base: os métodos getVocationBaseMaxHealth, getVocationBaseMaxMana, getNaturalSpeed e setMovementSpeed são utilizados pelas funções centrais recalcularVidaMana e recalcularVelocidade e NÃO devem ser alterados, pois garantem a estabilidade dos valores sem acumulação.

Inventário remoto: os métodos openRemoteContainer, closeRemoteContainer, createInventoryContainer, etc., devem ser usados apenas no contexto do escudeiro/pets. O closeRemoteContainer deve ser chamado quando a criatura morre para evitar janelas órfãs – isso é feito em Monster::death.

Nome customizado: setCustomName/getCustomName substituem setName/getName para criaturas SPA, garantindo que o nome exibido no cliente seja o nome do escudeiro e não o nome do arquivo do monstro.

Perseguição multi‑piso (v8.0):

A transição do jogador é capturada automaticamente em Player::onFloorChange chamado por Game::internalTeleport e Map::moveCreature.

O sistema usa uma fila de transições (std::deque<StairTransition>) consumida de forma independente por cada perseguidor.

Summons teleportam para o mestre se a fila estiver vazia e estiverem noutro piso; hostis nunca teleportam durante a perseguição (excepto para usar escadas activas).

O Tile::queryAdd tem excepções para isPursuingStairs e getStairOrigin, permitindo que o monstro pise o tile de escada.

O clearStairPursuit restaura o follow original usando m_savedFollowId.

Transições expiradas (>30s) são removidas da fila.

Waypoints de superfície: o jogador DEVE chamar addSurfaceWaypoint(getPosition()) durante o movimento (ex.: no onWalkComplete) para que as "migalhas" sejam geradas. Sem isso, os monstros não terão referências para navegação em terrenos complexos.

Timeout de progresso: hostis abandonam uma escada se ficarem 1,5 segundos sem reduzir a distância ao waypoint e sem estar a caminhar activamente.

Movimento forçado: apenas para summons; hostis utilizam waypoints de superfície e tiles adjacentes como alternativas.

GlobalMonsterMap (v8.0):

O grafo é partilhado entre todos os monstros; usar std::shared_mutex evita gargalos.

MAX_ACTIVE_SCANS = 20 impede picos de CPU por scans simultâneos (valor muito mais alto que antes, permitindo scans concorrentes leves).

Obstáculos temporários (Magic Walls, campos) são geridos com TTL de 30s; isWalkableUnlocked faz validação preguiçosa, garantindo reacção imediata.

Apenas tiles estáticos são salvos em disco; a persistência é assíncrona e não bloqueia o jogo.

O Tile::getDestination() é usado para obter destinos reais de teleportes; se não disponível, assume z±1 como fallback.

getAllTransitions() permite a procura global de escadas no checkStairPursuit e tryProactiveGlobalPursuit.

Mapeamento de 8 vizinhos (raio 1) nos destinos das escadas durante o scanArea.

bridgeCorridorUnlocked reconecta grafos desconectados no mesmo piso (até 64 passos).

BFS bidireccional como fallback melhora a descoberta de caminhos em grafos grandes.

Máquina de estados: a prioridade máxima é sempre o combate; qualquer rota global ou perseguição é cancelada se o jogador ficar atacável.

Engajamento: o deadline é renovado enquanto o líder está visível; o timeout por perda de visão (5s) impede perseguições eternas a jogadores escondidos.

Cercamento: tryFlankingRoute pode fazer o monstro sair do piso actual e regressar por outra escada; throttle de 600/1000 ms evita abusos.

SPAManager: os pets são armazenados em memória; a persistência (guarda/load) ainda não está implementada – é um ponto de melhoria futuro.
