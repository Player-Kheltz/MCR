⚠️ REGRAS DE REVISÃO PARA O ASSISTENTE (C++ – SERVIDOR CANARY)
Ao modificar QUALQUER arquivo .cpp/.h no directório Canary/src/, o assistente DEVE:

Garantir que TODAS as strings visíveis ao jogador (sendTextMessage, sendCancelMessage, logs de sistema) estejam em PT‑BR, com acentuação correcta.

Os caracteres acentuados PODEM ser escritos como UTF‑8 literal (ex.: "Não", "Você") desde que:
- O compilador MSVC tenha a flag `/utf-8` activa (verificada nas 3 configurações do .vcxproj)
- A string passe por `toLatin1()` antes de ser enviada ao protocolo (`sendCancelMessage` e `sendTextMessage` já fazem isto automaticamente)

Escapes octais (ex.: `\343` para `ã`) continuam a funcionar e são uma alternativa segura, mas não são obrigatórios.

Manter comentários em PT‑BR, excepto em nomes de funções/variáveis que permanecem em inglês (camelCase ou snake_case).

Verificar se a alteração mantém compatibilidade com o protocolo 15.00 e não quebra bindings Lua existentes (registerMethod, registerClass).

Se adicionar novos métodos expostos ao Lua, seguir a convenção de nomenclatura minúsculas_com_underscore e incluir validação de parâmetros (número e tipo). Documentar cada novo método neste guia.

Se alterar comunicação de rede (ProtocolGame), garantir que opcodes estendidos estejam na faixa reservada (160–255) e sejam documentados.

Toda nova consulta SQL DEVE usar placeholders parametrizados (evitar concatenação).

Após a edição, listar explicitamente os arquivos modificados para compilação selectiva.

O sistema de progressão usa Revscript: os scripts Lua ficam na pasta data-canary/scripts/MCR/SPA/. O motor carrega‑os automaticamente; não é necessário alterar o C++ para registar scripts.

🔨 PROCEDIMENTO DE COMPILAÇÃO (PADRÃO)
Clone o repositório MCR (já contém os fontes customizados):

bash
git clone https://github.com/Player-Kheltz/ProjetoMCR.git
cd ProjetoMCR\Canary
Configure o CMake (substitua o caminho do vcpkg pelo seu):

bash
mkdir build
cd build
cmake .. -DCMAKE_TOOLCHAIN_FILE=C:/vcpkg/scripts/buildsystems/vcpkg.cmake -DVCPKG_TARGET_TRIPLET=x64-windows
Compile:

bash
cmake --build . --config Release
O executável será gerado em build\Release\canary.exe.

Copie o executável para a raiz do servidor (onde estão data/, config.lua) ou configure o output directamente com cmake --install . (se configurado).

📌 Arquitectura da Máquina de Estados (v3.2.0)
A máquina de estados é executada integralmente dentro do ciclo Monster::onThinkMCR(), com apenas 5 estados processados por ciclo, pela seguinte ordem de prioridade:

text
┌──────────────────────────────────────────────┐
│ 1. COMBATE (prioridade máxima)                │
│    Se pode atacar → cancela tudo, ataca       │
├──────────────────────────────────────────────┤
│ 2. PERSEGUIÇÃO MULTI‑PISO (m_pursuingStairs)  │
│    Tenta alcançar m_stairOrigin               │
│    Gere travamentos, tenta movimento forçado   │
│    Monitoriza progresso, teleporta para escada │
├──────────────────────────────────────────────┤
│ 3. PLANEAMENTO DE ESCADA                      │
│    Se líder noutro piso → planStairTrajectory │
│    Se líder no mesmo piso e inacessível →     │
│      Avalia atalho multi‑piso vs rota alt.    │
├──────────────────────────────────────────────┤
│ 4. RETORNO AO SPAWN / CASA                   │
│    (Apenas monstros normais)                  │
│    processarRetornoSpawn / processarRetornoSimples │
│    Summons NUNCA entram neste estado           │
├──────────────────────────────────────────────┤
│ 5. IDLE                                       │
│    Se sem alvo → tenta follow, deambula ou    │
│    retorna ao spawn (só normais)              │

📌 Comportamento (v3.2.0)
Estado 1 – COMBATE: Se o monstro tem um alvo atacável, inicia follow direto.

Estado 2 – PERSEGUIÇÃO MULTI‑PISO:

Cancela perseguição se o líder voltou ao mesmo piso e a escada não é um atalho (m_multiFloorShortcut == false).

Movimento para a escada: distância 1 → movimento direto (internalMoveCreature); se falhar 3×, teleporta para o tile. Distância > 1 e não está a caminhar → tenta A* para a escada; se falhar, procura tile adjacente; se tudo falhar, movimento forçado na direção da escada.

Monitorização de progresso: se isWalkingToPosition(), reseta contador de stuck; senão, se a distância diminuiu, reseta; caso contrário, incrementa m_stairStuckCount. Após 5 ciclos preso, abandona a escada.

Chegada à escada: se myPos == m_stairOrigin, efetua a transição (nativa ou ativa). Após transição, verifica se o líder ficou acessível; se sim, inicia follow; senão, limpa perseguição e permite reavaliação.

Estado 3 – PLANEAMENTO DE ESCADA:

Se líder noutro piso → planStairTrajectory (throttle 2500ms).

Se líder no mesmo piso mas inacessível via A*:

Calcula multiFloorCost = evaluateMultiFloorCost(leaderPos).
Calcula alternativeCost = findNearestApproachTo(myPos, leaderPos).size().
Se multiFloorCost < alternativeCost e multiFloorCost < INT_MAX → inicia planStairTrajectory imediatamente (atalho multi‑piso).
Se alternativeCost < INT_MAX → tenta tryAlternativeRouteToLeader (throttle 5000ms).
Se a rota alternativa já estava ativa mas m_waypointStuckCycles >= 3 → força o atalho (limpa waypoints e chama planStairTrajectory).
Estado 4 – RETORNO AO SPAWN (apenas hostis): Summons nunca entram neste estado. Hostis regressam ao spawn usando processarRetornoSimples (caminhada A* até m_homePosition) ou processarRetornoSpawn (percorre m_returnPath no sentido inverso, usando as escadas de volta).

Estado 5 – IDLE:

Se m_followingWaypoints ativo, chama walkToWaypoint para o waypoint atual.

Destravamento silencioso para summons: se a summon estiver a seguir waypoints e m_waypointStuckCycles >= 3, teleporta silenciosamente para o primeiro tile do caminho calculado pelo findNearestApproachTo e reinicia os waypoints.

Caso contrário, tenta follow normal, deambulação ou retorno ao spawn (apenas normais, se distToSpawn > 10).

Tratamento de Summons vs Hostis:

Funcionalidade	Summons	Hostis
Perda de líder (5 ticks)	Continua perseguição normal (nunca desiste)	Inicia retorno ao spawn
Deadline expirado	Continua perseguição normal	Inicia retorno ao spawn
Retorno ao spawn	Nunca – continua perseguição	Sim, caminha de volta
Destravamento silencioso	Sim, se presa ≥ 3 ciclos	Não
Seguimento imediato da escada do líder: No início do planStairTrajectory, se existir uma transição do jogador com origem no piso do monstro, destino no piso do líder, distância ≤ 3 tiles e válida → perseguição imediata.

Anti‑ping‑pong: O flag m_multiFloorShortcut é ativado quando leaderPos.z == myPos.z no momento do planeamento, protegendo perseguições de atalho. Escadas falhadas têm cooldown de 10 segundos em m_recentlyFailedStairs.

Avaliação de custo total: custo = distância A* até à escada + 1 (transição) + distância estimada do destino ao líder, usando findNearestApproachTo para a estimativa.

Integração com Tile::queryAdd: Corrigidas as verificações de TILESTATE_FLOORCHANGE e TILESTATE_TELEPORT para usar || em vez de |. O pathfinding (A*) só considera escadas/teleportes se o monstro estiver em perseguição. O movimento real permite a entrada apenas quando o tile é a escada planeada.

Sistema de engajamento e deadline:

m_engagement (0–100): sobe +2 ao causar/receber dano, +1/s com líder visível; desce -3/s sem ver o líder.

deadline = currentTime + min(60000, 30000 + engagement * 500) ms. Renovado enquanto o líder está visível.

⚠️ ARMADILHAS E LIÇÕES APRENDIDAS
Gerais
Codificação de console: SetConsoleOutputCP(CP_UTF8) é imprescindível para logs em PT‑BR.

Tradução de logs: use SEMPRE escapes octais para todos os acentos nas strings C++. Não confie em UTF‑8 directo.

Conexão com MariaDB: verificar senha em config.lua e se o serviço está rodando.

Compilação com vcpkg: mantenha o vcpkg actualizado.

Escapes octais vs UTF‑8: use sempre escapes octais para portabilidade. Esta é uma regra absoluta no MCR.

Coluna lastlogin: EXISTE na tabela players do schema actual do Canary (confirmado no banco de dados do projecto). Pode ser usada para lógica de inicialização/migração.

Buffer de afinidade: o método addDominioAfinidadeBatch deve ser eficiente para evitar lag.

Estados de alma: use player:getAllActiveStates(); não itere manualmente sobre todos os domínios.

Cores: as definidas em texto_colorido.lua (v10.0) devem ser mantidas consistentes.

Projécteis dinâmicos: player:getAmmo() é essencial e deve ser preservado.

Modificadores de combate: setCombatModifier e getCombatModifier suportam a Sinergia Elemental.

Atributos base: os métodos getVocationBaseMaxHealth, getVocationBaseMaxMana, getNaturalSpeed e setMovementSpeed são utilizados pelas funções centrais recalcularVidaMana e recalcularVelocidade e NÃO devem ser alterados, pois garantem a estabilidade dos valores sem acumulação.

Inventário remoto: os métodos openRemoteContainer, closeRemoteContainer, createInventoryContainer, etc., devem ser usados apenas no contexto do escudeiro/pets. O closeRemoteContainer deve ser chamado quando a criatura morre para evitar janelas órfãs – isso é feito em Monster::death.

Nome customizado: setCustomName/getCustomName substituem setName/getName para criaturas SPA, garantindo que o nome exibido no cliente seja o nome do escudeiro e não o nome do arquivo do monstro.

Perseguição Multi‑Piso (v3.2.0)
A transição do jogador é capturada automaticamente em Player::onFloorChange chamado por Game::internalTeleport e Map::moveCreature.

O sistema usa uma fila de transições (std::deque<StairTransition>) consumida de forma independente por cada perseguidor.

Summons nunca teleportam para o mestre durante a perseguição normal; se ficarem completamente bloqueadas, o destravamento silencioso (teleporte invisível para o primeiro waypoint) as recoloca em movimento.

O Tile::queryAdd foi corrigido para usar || em vez de | nas verificações de flags, e tem excepções para isPursuingStairs e getStairOrigin, permitindo que o monstro pise o tile de escada.

O clearStairPursuit restaura o follow original usando m_savedFollowId. Não zera m_pursuitDeadline.

O flag m_multiFloorShortcut é ativado quando leaderPos.z == myPos.z no momento do planeamento, protegendo perseguições de atalho.

O findStairNearReal foi corrigido para usar deltaZ = 1 para FLOORCHANGE_DOWN (desce) e deltaZ = -1 para flags de subida.

O destravamento silencioso de summons usa internalTeleport sem efeitos visuais, mantendo a imersão.

GlobalMonsterMap (v3.2.0)
O grafo é partilhado entre todos os monstros; usar std::shared_mutex evita gargalos.

MAX_ACTIVE_SCANS = 20 impede picos de CPU por scans simultâneos.

Obstáculos temporários (Magic Walls, campos) são geridos com TTL de 30s; isWalkableUnlocked faz validação preguiçosa, garantindo reacção imediata.

Apenas tiles estáticos são salvos em disco; a persistência é assíncrona e não bloqueia o jogo.

O Tile::getDestination() é usado para obter destinos reais de teleportes; se não disponível, assume z±1 como fallback.

getAllTransitions() é usado pelo planStairTrajectory para obter escadas candidatas.

Mapeamento de 8 vizinhos (raio 1) nos destinos das escadas durante o scanArea.

bridgeCorridorUnlocked reconecta grafos desconectados no mesmo piso (até 64 passos).

BFS bidireccional como fallback melhora a descoberta de caminhos em grafos grandes.

Máquina de estados (v3.2.0)
A prioridade máxima é sempre o combate; qualquer rota ou perseguição é cancelada se o jogador ficar atacável.

O Bloco 3 avalia simultaneamente o atalho multi‑piso e a rota alternativa, escolhendo o de menor custo total.

O Bloco 5 inclui o destravamento silencioso para summons, que só é activado se a summon estiver parada há ≥ 3 ciclos.

SPAManager
Os pets são armazenados em memória; a persistência (guarda/load) ainda não está implementada – é um ponto de melhoria futuro.
