⚠️ REGRAS DE REVISÃO PARA O ASSISTENTE (C++ – SERVIDOR CANARY)

Ao modificar QUALQUER arquivo .cpp/.h no diretório Canary/src/, o assistente DEVE:

- Garantir que TODAS as strings visíveis ao jogador (sendTextMessage, sendCancelMessage, logs de sistema) estejam em PT‑BR, com acentuação correcta. Os caracteres acentuados DEVEM ser escritos OBRIGATORIAMENTE como escapes octais Latin‑1 (ex.: "N\355vel", "Voca\347\343o"). NÃO é permitido usar UTF‑8 diretamente nas strings; usar escapes octais garante portabilidade e evita problemas de compilador.
- Manter comentários em PT‑BR, excepto em nomes de funções/variáveis que permanecem em inglês (camelCase ou snake_case).
- Verificar se a alteração mantém compatibilidade com o protocolo 15.00 e não quebra bindings Lua existentes (registerMethod, registerClass).
- Se adicionar novos métodos expostos ao Lua, seguir a convenção de nomenclatura minúsculas_com_underscore e incluir validação de parâmetros (número e tipo). Documentar cada novo método neste guia.
- Se alterar comunicação de rede (ProtocolGame), garantir que opcodes estendidos estejam na faixa reservada (160–255) e sejam documentados.
- Toda nova consulta SQL DEVE usar placeholders parametrizados (evitar concatenação).
- Após a edição, listar explicitamente os arquivos modificados para compilação selectiva.
- O sistema de progressão usa Revscript: os scripts Lua ficam na pasta `data-canary/scripts/MCR/Dominio/`. O motor carrega‑os automaticamente; não é necessário alterar o C++ para registar scripts.

📦 DEPENDÊNCIAS (AMBIENTE WINDOWS)

Ferramentas Obrigatórias:

- Visual Studio 2022 (Community ou superior) com carga de trabalho "Desenvolvimento para desktop com C++".
- CMake 3.20 ou superior (adicionar ao PATH).
- Git for Windows.
- vcpkg (gerenciador de bibliotecas). Instale em C:\vcpkg (ou caminho sem espaços).

Comandos de instalação do vcpkg:
git clone https://github.com/Microsoft/vcpkg.git C:\vcpkg
cd C:\vcpkg
.\bootstrap-vcpkg.bat
.\vcpkg integrate install

Bibliotecas Requeridas (instalar via vcpkg, para x64-windows):
.\vcpkg install boost-beast:x64-windows boost-asio:x64-windows
.\vcpkg install boost-filesystem:x64-windows boost-system:x64-windows
.\vcpkg install boost-variant:x64-windows libmariadb:x64-windows
.\vcpkg install lua:x64-windows pugixml:x64-windows cryptopp:x64-windows
.\vcpkg install fmt:x64-windows spdlog:x64-windows nlohmann-json:x64-windows
.\vcpkg install gtest:x64-windows (opcional, para testes)

Observação: os pacotes podem mudar de nome; consulte o README.md do repositório canary para obter a lista exacta para a versão actual.

🔨 PROCEDIMENTO DE COMPILAÇÃO (PADRÃO)

Clone o repositório MCR (já contém os fontes customizados):
git clone https://github.com/Player-Kheltz/ProjetoMCR.git
cd ProjetoMCR\Canary

Configure o CMake (substitua o caminho do vcpkg pelo seu):
mkdir build
cd build
cmake .. -DCMAKE_TOOLCHAIN_FILE=C:/vcpkg/scripts/buildsystems/vcpkg.cmake -DVCPKG_TARGET_TRIPLET=x64-windows

Compile:
cmake --build . --config Release

O executável será gerado em build\Release\canary.exe.

Copie o executável para a raiz do servidor (onde estão data/, config.lua) ou configure o output directamente com cmake --install . (se configurado).

🚀 Novos Métodos Lua para o Sistema de Progressão v2.1.1 (A IMPLEMENTAR)

Estes métodos devem ser adicionados a player.h/cpp e expostos em luascript.cpp. São essenciais para o funcionamento dos Estados de Alma, progressão multi‑domínio e Posturas de Combate. ⚠️ NENHUM destes métodos existe ainda — a implementação é o objectivo da Fase 2.

- player:getDominioAfinidade(dominioId) – retorna a afinidade acumulada no Domínio.
- player:addDominioAfinidade(dominioId, valor) – adiciona afinidade (gerida pelos scripts).
- player:setDominioDespertado(dominioId, bool) – marca o Domínio como despertado.
- player:hasHabilidade(habilidadeId) – verifica se o jogador conhece a habilidade.
- player:learnHabilidade(habilidadeId) – concede uma habilidade permanentemente.
- player:getEstadoAlma(dominioId) – retorna "lampejo", "vinculo", "maestria" ou nil.
- player:registerDominioAction(dominioId) – regista uma acção no deque de histórico para cálculo dos Estados de Alma.
- player:getDominioNivel(dominioId) — retorna o nível actual do Domínio. 

Lógica interna necessária em player.cpp:

- std::deque<ActionRecord> onde cada registo contém timestamp e lista de domínios envolvidos.
- Método calculateEstadosAlma() chamado por registerDominioAction (ou periodicamente):
  - Lampejo: 3 acções consecutivas da mesma especialização em 2 minutos.
  - Vínculo: >60% das acções nos últimos 15 minutos associadas a uma especialização.
- Cache dos estados para evitar recálculo excessivo.

📅 EVENTO onHealthChange (FUTURO)

Actualmente o Canary não possui um evento onHealthChange nativo. Este evento será adicionado no futuro para permitir a progressão contínua por dano. O planeamento é:

- Disparar o evento Lua em Game::combatChangeHealth (ou equivalente), antes ou depois do dano.
- Assinatura Lua: onHealthChange(creature, attacker, primaryDamage, primaryType, secondaryDamage, secondaryType, origin).
- O script `dominio_healthchange.lua` (já esboçado) usará este evento para distribuir afinidade com base no dano causado.

Enquanto este evento não existir, a progressão por dano NÃO estará activa. A progressão por morte (onKill) e por uso de ferramentas continuará a funcionar.

🛠️ PIPELINE DE TRADUÇÃO AUTOMÁTICA

A partir da versão actual, TODO o código C++ do servidor foi traduzido para PT‑BR através de uma pipeline automática supervisionada. As directrizes abaixo DEVEM ser mantidas em qualquer modificação futura:

- Strings de interface e logs: PT‑BR com escapes octais Latin‑1 (OBRIGATÓRIO). NUNCA colocar texto acentuado em UTF‑8 diretamente nas strings C++. Exemplo correcto: "N\355vel de magia \351 alto." Exemplo INCORRECTO: "Nível de magia é alto."
- Bindings Lua intactos: nomes de funções registadas NUNCA são alterados.
- Cabeçalhos (.h/.hpp): comentários traduzidos, definições de classes/macros intactas.

Consulte a lista completa de scripts da pipeline no Índice Geral (Ferramentas de Automação).

🗂️ ESTRUTURA DE PASTAS REVSCRIPT PARA O SISTEMA DE PROGRESSÃO

Os scripts do Sistema de Progressão do Aventureiro residem em:

`data-canary/scripts/MCR/Dominio/`

O Canary, com suporte a Revscript, carrega automaticamente todos os ficheiros .lua nessa pasta e subpastas. Não é necessário registá‑los manualmente no config.lua ou em qualquer outro lugar. A estrutura recomendada é:
data-canary/scripts/MCR/Dominio/
├── config_dominios.lua
├── dominio_utils.lua
├── evento_login.lua
├── evento_kill.lua
├── evento_postura.lua
├── evento_ferramentas.lua
├── evento_healthchange.lua (futuro)
└── ...

text

Scripts genéricos e utilitários partilhados (como `pronouns.lua`, `npc_utils.lua`) ficam em `data-canary/scripts/MCR/Geral/` e também são carregados automaticamente. O uso de `dofile` para estes módulos é desnecessário se eles estiverem na estrutura Revscript — as funções que expõem tornam-se globais. Contudo, se um script estiver fora da árvore Revscript, `dofile` com caminho relativo a `data-canary/` ainda pode ser usado.

⚠️ ARMADILHAS E LIÇÕES APRENDIDAS

- Codificação de console: SetConsoleOutputCP(CP_UTF8) é imprescindível para logs em PT‑BR.
- Tradução de logs: use SEMPRE escapes octais para todos os acentos nas strings C++. Não confie em UTF‑8 direto.
- Conexão com MariaDB: verificar senha em config.lua e se o serviço está rodando.
- Compilação com vcpkg: mantenha o vcpkg atualizado.
- Escapes octais vs UTF‑8: use sempre escapes octais para portabilidade. Esta é uma regra absoluta no MCR.
- A coluna `lastlogin` EXISTE na tabela `players` do schema actual do Canary (verificado no banco de dados do projecto). Pode ser usada para lógica de inicialização/migração de personagens no evento_login.lua.
