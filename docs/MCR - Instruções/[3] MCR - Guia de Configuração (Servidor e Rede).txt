🎯 OBJETIVO DESTE GUIA
Este documento reúne TODAS as configurações necessárias para que o servidor Canary, o Login Server e o ambiente de rede do Projeto MCR funcionem em harmonia. Ele cobre:

- O arquivo config.lua do Canary (com cada parâmetro explicado).
- A tabela obrigatória OTCFEATURES.
- O arquivo .env do Login Server (variáveis de ambiente).
- Portas, firewall e conectividade.
- Configuração do banco de dados (acesso e charset).
- Definição dos pronomes do personagem (coluna nativa players.pronoun) e do pronome da conta (accounts.player_pronoun).
- Faixas de storages reservados para sistemas globais.
- Estrutura de pastas Revscript para os scripts do Sistema de Progressão.
- Referência às ferramentas de automação (pipeline de tradução).
- Versionamento com Git (boas práticas).

Toda vez que for necessário ajustar uma porta, adicionar uma flag ou depurar um erro de conexão, este guia é a referência primária. A versão 3.4 atualiza as faixas de opcodes e storages para acomodar o Sistema de Progressão v2.1.1 (Estados de Alma, Posturas, progressão multi‑domínio) e confirma a compatibilidade total com as novas mecânicas.

- Garantir que as portas mencionadas são EXACTAMENTE:

  - 7172 (login – ProtocolLogin)
  - 7173 (game – ProtocolGame)
  - 8080 (HTTP – Login Server)
  - 3306 (MySQL/MariaDB – somente interno)
- Verificar se as flags em OTCFEATURES no config.lua estão sincronizadas com o que o OTClient realmente espera (conforme lista abaixo).
- Sempre que adicionar uma nova variável ao .env, incluir a documentação correspondente e atualizar o arquivo .env.example (para manter o template sem segredos).
- Nas instruções de versionamento, lembrar que o PowerShell requer aspas duplas em caminhos com espaços. Exemplo: git add "docs/Projeto MCR.txt"
- Ao configurar charset do banco, sempre forçar utf8mb4 (COLLATE utf8mb4_unicode_ci).
- O pronome do personagem é armazenado na coluna nativa players.pronoun. O storage 100001 NÃO é mais utilizado para essa finalidade.
- O pronome da conta (jogador/jogadora) é armazenado em accounts.player_pronoun.
- As faixas de storages reservados para sistemas globais (100010+ para intimidade, 50000+ para missões) continuam válidas. Os storages para o Sistema de Progressão são mínimos, pois a progressão usa principalmente as tabelas player_dominios e player_habilidades.
- Os novos opcodes (182, 183, 184) usados pelo sistema de progressão pertencem à faixa 180‑199, já reservada para o SQH. Não há conflito, mas o assistente deve confirmar que os números exactos estão alinhados com os módulos do cliente.
- As Posturas de Combate (Ímpeto, Equilíbrio, Guarda) reutilizam os modos de luta nativos (fightMode) do Tibia. Nenhuma configuração extra é necessária no servidor ou cliente para suportar essa mecânica além da interpretação via script.
- Os scripts do Sistema de Progressão são carregados automaticamente pelo Revscript a partir da pasta `data-canary/scripts/MCR/Dominio/`. O desenvolvedor deve garantir que essa pasta existe e contém os scripts .lua necessários. Não é preciso alterar config.lua para ativá‑los, mas o diretório `data-canary` deve estar configurado como a raiz de dados do servidor.

📄 1. CONFIGURAÇÃO DO CANARY (config.lua)

Este arquivo está localizado na raiz do servidor Canary (ProjetoMCR/Canary/config.lua). Abaixo, cada parâmetro obrigatório e seu respectivo valor no Projeto MCR.

```lua
-- Conexão com o Banco de Dados
mysqlHost = "127.0.0.1"
mysqlUser = "root"
mysqlPass = "<Senha>" -- ⚠️ usar placeholder, senha real nunca em documentação pública
mysqlDatabase = "BancoServer"
mysqlPort = 3306
mysqlInitCommand = "SET NAMES 'utf8mb4'" -- Garante UTF-8 completo na sessão

-- Autenticação e Criptografia
passwordType = "sha1" -- Hash de senhas (SHA1 compatível com Login Server)
encryptionType = "aes" -- Criptografia de rede (protocolo 15.00)

-- Rede e Portas
ip = "0.0.0.0" -- Essencial para aceitar conexões externas (inclusive localhost)
statusPort = 7171 -- Porta de status (opcional, mas configurada)
loginPort = 7172 -- Porta de login (usada pelo Login Server internamente)
gamePort = 7173 -- Porta do jogo (players se conectam a esta)

-- Identificação do Servidor
serverName = "Projeto MCR" -- Nome exibido no cliente
ownerEmail = "admin@projetomcr.com"
url = "http://projetomcr.com/"
url_shop = "http://loja.projetomcr.com/"
motd = "Bem-vindo ao Projeto MCR! Uma aventura fantasiosa te aguarda."

-- Limites
maxPlayers = 1000
maxMessageBuffer = 5000

-- Rates (multiplicadores de experiência, skill, etc.)
rateExp = 5.0
rateSkill = 3.0
rateLoot = 2.0
rateMagic = 3.0
rateSpawn = 2.0

-- PvP / Frag
useFragHandler = true

-- Recursos de Cliente (OTCFeatures) – ver secção 2 abaixo

-- Outras
enableLiveCasting = false
showPlayerInfo = true
sqlType = "mysql"
⚠️ NOTA CRÍTICA: ip = "0.0.0.0" é necessário para evitar erro 10060 no cliente mesmo em testes locais (Fase 18). O default 127.0.0.1 não permite que o OTClient se conecte ao ProtocolGame em certas condições.

📋 2. TABELA OTCFEATURES (CONFIG.LUA)

Esta tabela informa ao OTClient quais recursos estendidos o servidor suporta. DEVE ser mantida exatamente como abaixo. Qualquer alteração aqui precisa ser refletida nos módulos do OTClient.

lua
OTCFEATURES = {
    modalWindows = true,          -- Janelas modais (Banco, Loja, confirmações críticas)
    extendedOpcode = true,        -- Comunicação por opcodes customizados (essencial para SQH e progressão)
    changeMapAwareRange = true,
    newWalking = false,           -- ⚠️ MANTENHA FALSE, a menos que o servidor tenha suporte compilado
    newCreatureIcons = true,
    newPartyList = true,
    newBattlList = true,          -- Lista de batalha otimizada (dispensa game_battle)
    shopModule = true,            -- Loja in-game
    playerMounts = true,
    playerAddons = true,
    expertMode = true,
    dailyReward = true,
    imbuementSystem = true,
    analytics = false             -- Sempre desabilitado
}
Os opcodes estendidos usados pelo Sistema de Progressão (182, 183, 184) dependem de extendedOpcode = true. Esta flag já está activa e garante a comunicação bidirecional.

🌐 3. LOGIN SERVER – ARQUIVO .env

Local: ProjetoMCR/LoginServer/.env

Este arquivo contém variáveis de ambiente sensíveis. NÃO é versionado (está no .gitignore). Um template público está em LoginServer/.env.example.

dotenv
# Banco de dados (mesmo do Canary)
MYSQL_DBNAME=BancoServer
MYSQL_HOST=127.0.0.1
MYSQL_PORT=3306
MYSQL_USER=root
MYSQL_PASS=<Senha>

# Login Server
LOGIN_IP=0.0.0.0
LOGIN_HTTP_PORT=8080

# Servidor de jogo (Canary) – informações enviadas ao cliente após autenticação
SERVER_IP=127.0.0.1
SERVER_NAME=ProjetoMCR
SERVER_PORT=7173          # ⚠️ DEVE SER 7173 (gamePort), NUNCA 7172. Erro 10060 se errado.
SERVER_LOCATION=Brasil
VOCATIONS=0               # Apenas Aventureiro (vocation 0). O projeto MCR não utiliza vocações tradicionais.
⚠️ LIÇÃO CRÍTICA: O campo SERVER_PORT é a porta do ProtocolGame (7173). Na Fase 18, um valor errado (7172) causou timeout 10060. Verifique SEMPRE essa variável após alterações no config.lua.

⚠️ ATUALIZAÇÃO v2.1.1: A variável VOCATIONS foi ajustada para "0". O MCR não tem múltiplas vocações; todos os personagens são Aventureiros. Se o Login Server exigir uma lista, mantenha apenas {0}. O código do cliente deve estar preparado para ignorar esse campo ou utilizá‑lo apenas para validação de protocolo.

🔥 4. FIREWALL E CONECTIVIDADE

Para testes locais (tudo em 127.0.0.1), o firewall pode não bloquear, mas em produção ou rede local, as seguintes regras DEVEM ser aplicadas:

Porta 8080 (TCP): Acesso público – Login Server HTTP. Necessário para que o cliente faça login e crie contas.

Porta 7172 (TCP): Comunicação interna entre Login Server e Canary (ProtocolLogin). NÃO deve ser exposta ao público externo.

Porta 7173 (TCP): Acesso público – ProtocolGame. Players se conectam directamente.

Porta 3306 (TCP): MySQL/MariaDB. JAMAIS exponha à internet.

A porta 8080 também é usada pelo cliente para chamar /guest_login (fluxo do Salão dos Destinos).

No Windows, criar regra via PowerShell (Admin):

powershell
New-NetFirewallRule -DisplayName "MCR Game 7173" -Direction Inbound -LocalPort 7173 -Protocol TCP -Action Allow
New-NetFirewallRule -DisplayName "MCR HTTP 8080" -Direction Inbound -LocalPort 8080 -Protocol TCP -Action Allow
Testar conectividade:
Test-NetConnection 127.0.0.1 -Port 7173

Se falhar, verificar se o Canary está rodando e se ip = "0.0.0.0" está configurado.

🗂️ 7. PRONOMES DO PERSONAGEM E DA CONTA

IMPORTANTE: O Projeto MCR distingue dois conceitos de pronome:

Pronome da conta: como o Oráculo se dirige ao JOGADOR. Armazenado em accounts.player_pronoun (0 = não definido, 1 = jogador, 2 = jogadora).

Pronome do personagem: como os NPCs e o mundo se referem ao avatar. Armazenado na coluna nativa players.pronoun (0 = usa o sexo, 2 = ela, 3 = ele...).

O storage 100001 NÃO é mais usado para pronome do personagem. Ele foi removido dessa função e está livre para outros usos (se necessário).

Para consultar o pronome do personagem em scripts Lua, utilize o módulo pronouns.lua localizado em data-canary/scripts/MCR/Geral/pronouns.lua. Este módulo é carregado automaticamente pelo Revscript, portanto NÃO é necessário usar dofile para carregá‑lo. Basta chamar:

lua
local subject = Pronouns.getSubject(player)  -- "he", "she" ou "they"
Este módulo consulta automaticamente player:getPronoun() e faz fallback para o sexo do personagem quando o valor é 0.

🗂️ 8. STORAGES E OPCODES RESERVADOS PARA O SISTEMA DE PROGRESSÃO

A fim de evitar conflitos, as seguintes faixas estão reservadas:

Faixa / ID	Sistema	Descrição
100010–109999	Intimidade com NPCs	Nível de intimidade: 0=desconhecido, 1=conhecido, 2=amigo, 3=confidente.
50000–50999	Missões (progresso)	Storages de progresso de missões.
90001	Temporário (accountId)	Usado pelo Oráculo durante gerenciamento de contas.
Opcodes 180‑199	SQH e Progressão	HUD de missões (180), toasts (181), ícone de Vínculo (182), partículas de Estados de Alma (183), janela de conhecimento (184).
Opcode 200	Migração de conta	Usado pelo Oráculo para transferir o jogador da Alma para a conta permanente.
Os opcodes 182, 183 e 184 são dedicados ao feedback visual do Sistema de Progressão. O desenvolvedor deve garantir que os listeners no cliente estejam sincronizados com esses números.

As Posturas de Combate NÃO usam opcodes; são baseadas no fightMode enviado pelo cliente (já nativo) e interpretado pelo script dominio_postura.lua do lado do servidor.

O progresso de Afinidade, níveis e Estados de Alma são armazenados nas tabelas player_dominios e player_habilidades do banco de dados. Storages adicionais podem ser usados temporariamente para migração (ex.: storage 90000 para flag de migração concluída), mas devem ser documentados.

🗂️ 9. ESTRUTURA DE PASTAS REVSCRIPT PARA O SISTEMA DE PROGRESSÃO

Os scripts do Sistema de Progressão do Aventureiro residem em:

data-canary/scripts/MCR/Dominio/

Scripts genéricos e utilitários partilhados (como pronouns.lua, npc_utils.lua) ficam em:

data-canary/scripts/MCR/Geral/

O Canary, com suporte a Revscript, carrega automaticamente todos os ficheiros .lua nessas pastas e subpastas. Não é necessário registá‑los manualmente no config.lua ou em qualquer outro lugar. A estrutura recomendada é:

text
data-canary/scripts/MCR/
├── Dominio/
│   ├── config_dominios.lua
│   ├── dominio_utils.lua
│   ├── evento_login.lua
│   ├── evento_kill.lua
│   ├── evento_postura.lua
│   ├── evento_ferramentas.lua
│   ├── evento_healthchange.lua (futuro)
│   └── ...
├── Geral/
│   ├── npc_utils.lua
│   ├── pronouns.lua
│   └── ...
├── Criacao/
│   ├── oraculo.lua
│   └── ...
├── Tutorial/
│   ├── eldrin.lua
│   ├── thorne.lua
│   └── ...
Cada script pode conter múltiplos eventos (onLogin, onKill, onFightModeChange, etc.) e é completamente modular. Funções definidas nos módulos em Geral/ tornam‑se globais automaticamente e podem ser chamadas por qualquer outro script sem necessidade de dofile.

🗺️ 10. CONFIGURAÇÃO DO MAP EDITOR (RME)

O RME modificado importa monstros e NPCs diretamente dos diretórios Lua do Canary. As configurações relevantes estão em:

Preferências → Monstros: Diretório Lua (ex: data/monster)

Preferências → NPCs: Diretório Lua (ex: data/npc)

Estes caminhos devem apontar para os diretórios do servidor Canary MCR. A conversão ISO‑8859‑1 → UTF‑8 já está incorporada no RME.

⚠️ ARMADILHAS E LIÇÕES APRENDIDAS

SERVER_PORT errado: Usar 7172 no .env causa erro 10060 (timeout) no cliente. O campo externalport na resposta de login deve ser 7173.

ip = "127.0.0.1": Pode impedir conexões do OTClient mesmo no mesmo PC. Sempre use ip = "0.0.0.0".

Senha do MariaDB: use SET PASSWORD FOR, não ALTER USER.

phpMyAdmin sem senha: configurar config.inc.php correctamente.

Cache do OTClient: manter cache local (cache/).

OTCFeatures.newWalking: false evita problemas.

Firewall no Windows: sempre verificar com Test-NetConnection.

Botão "Criar Conta" removido: agora é "Salão dos Destinos".

storage 100001 abolido para pronome; usar players.pronoun.

A coluna lastlogin EXISTE na tabela players (confirmado no schema do projecto). Pode ser usada em scripts de inicialização.

Os scripts da pasta MCR/Dominio são carregados automaticamente (Revscript). Se um evento não dispara, verifique se o ficheiro está na pasta correcta e se não há erros de sintaxe.

O módulo pronouns.lua está em data-canary/scripts/MCR/Geral/ e NÃO requer dofile para ser usado.
