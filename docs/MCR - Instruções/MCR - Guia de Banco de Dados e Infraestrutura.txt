================================================================================
⚠️ REGRAS DE REVISÃO PARA O ASSISTENTE (BANCO DE DADOS E INFRA)
================================================================================

Ao sugerir QUALQUER alteração no banco de dados ou na infraestrutura, o assistente DEVE:

JAMAIS incluir senhas reais em texto. Usar SEMPRE o placeholder <Senha>.

Garantir que toda nova tabela, coluna ou banco seja criado com:
CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci.

Lembrar que o XAMPP atual utiliza MariaDB, e a sintaxe de alguns comandos
difere do MySQL puro (ex.: SET PASSWORD FOR... em vez de ALTER USER).

Incluir o comando SQL exato para qualquer modificação estrutural.

Recomendar backup ANTES de qualquer operação destrutiva (DROP, DELETE sem WHERE).

Manter a compatibilidade com o schema do Canary 3.4.1 (tabelas accounts, players,
etc.) e com as consultas do Login Server.

Documentar novos campos ou tabelas para que o assistente saiba da sua existência
em interações futuras.

**NOVO**: A coluna `created_at` (INT, default 0) na tabela `accounts` é essencial
para a limpeza de contas guest. Não a remova ou altere sem actualizar o Login Server.

**NOVO**: A coluna `player_pronoun` (TINYINT, default 0) na tabela `accounts`
armazena o pronome do JOGADOR (0 = não definido, 1 = jogador, 2 = jogadora).
É utilizada pelo Oráculo ao se dirigir ao dono da conta.

**NOVO**: A coluna `pronoun` na tabela `players` é NATIVA do Canary e armazena o
pronome do PERSONAGEM (0 = usa o sexo, 2 = ela, 3 = ele, etc.). NÃO utilize
storages manuais para este fim.

================================================================================
🗄️ 1. AMBIENTE: XAMPP (MARIADB)
================================================================================

O Projeto MCR utiliza o XAMPP como stack de desenvolvimento local, que fornece
o MariaDB (fork do MySQL) e o phpMyAdmin.

Instalação:

Versão: XAMPP 8.2.12 (ou superior compatível).

Local: D:\Programas\Xampp (ajustar conforme o ambiente).

Componentes instalados: MySQL e phpMyAdmin (Apache opcional para o MyAAC).

Configuração do MariaDB (my.ini):

Local do arquivo: D:\Programas\Xampp\mysql\bin\my.ini

Parâmetros obrigatórios:
[client]
default-character-set = utf8mb4

[mysql]
default-character-set = utf8mb4

[mysqld]
character-set-server = utf8mb4
collation-server = utf8mb4_unicode_ci
skip-character-set-client-handshake
init-connect = 'SET NAMES utf8mb4 COLLATE utf8mb4_unicode_ci'

Essas configurações garantem que todas as conexões usem UTF-8 completo, sem
depender de configuração manual em cada sessão.

Definição da Senha do Root (MariaDB):
SET PASSWORD FOR 'root'@'localhost' = PASSWORD('<Senha>');
⚠️ DIFERENÇA CRÍTICA: O comando ALTER USER ... IDENTIFIED WITH mysql_native_password
NÃO funciona no MariaDB. Use SEMPRE SET PASSWORD FOR... (Fase 33)

================================================================================
🛠️ 2. CRIAÇÃO DO BANCO DE DADOS
================================================================================

O banco de dados oficial do Projeto MCR se chama "BancoServer".

Comando de criação:
CREATE DATABASE BancoServer CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

Por que utf8mb4_unicode_ci e não utf8mb4_general_ci?

unicode_ci oferece suporte superior a caracteres acentuados do português (ç, ã, õ),
tratando 'c' e 'ç' como letras distintas e ordenando corretamente.

É o padrão recomendado para aplicações multilíngues e localizadas.

================================================================================
📜 3. SCHEMA DO BANCO DE DADOS
================================================================================

O schema principal está no arquivo schema.sql, fornecido com o Canary v3.4.1.
Local: ProjetoMCR/Canary/data/schema.sql

Principais tabelas e suas funções:

accounts: armazena contas de jogador. Colunas: id, name, password (hash SHA1), email (vazio), **created_at (INT)**, **player_pronoun (TINYINT)**.

players: armazena personagens. Colunas: id, name, account_id, level, vocation, sex, posx, posy, posz, town_id, **pronoun (TINYINT)**, etc.

É complementado por dezenas de outras tabelas para houses, guilds, store, daily rewards, etc.

O Login Server acessa as mesmas tabelas accounts e players. Qualquer alteração
nessas tabelas DEVE ser sincronizada com o código do Login Server.

**Adições MCR ao schema:**

- **accounts.created_at**: Coluna INT (timestamp Unix). Usada para controlar a criação
  de contas guest e permitir a sua remoção automática após 30 minutos de inatividade.
  Contas permanentes podem ter o valor 0 ou o timestamp de criação.

- **accounts.player_pronoun**: Coluna TINYINT. Define como o Oráculo deve se dirigir
  ao JOGADOR (dono da conta). Valores:
    0 = não definido (perguntará novamente)
    1 = jogador (tratamento masculino)
    2 = jogadora (tratamento feminino)

- **players.pronoun**: Coluna TINYINT nativa do Canary. Define o pronome do PERSONAGEM
  (como o mundo deve se referir a ele). Valores:
    0 = não definido (usa o sexo do personagem como fallback)
    1 = they (neutro) – disponível, mas não exposto pelo Oráculo por enquanto
    2 = she (ela)
    3 = he (ele)
    4 = ze
    5 = usar o nome (name)
  O Oráculo grava diretamente 2 ou 3 durante a criação/gerenciamento, e o servidor
  automaticamente trata pronome = 0 como fallback para o sexo.

Não é mais necessário o uso de storage 100001 para pronome do personagem.

- **Rotina de limpeza de contas guest** (implementada no Login Server):
  Remove contas com prefixo `guest_` e `created_at` anterior a 30 minutos, assim como
  os respectivos personagens (`Guest_*`).
  Comandos SQL executados periodicamente:
  ```sql
  DELETE FROM players WHERE name LIKE 'Guest\_%' AND account_id IN (
      SELECT id FROM accounts WHERE name LIKE 'guest\_%' AND created_at < UNIX_TIMESTAMP() - 1800
  );
  DELETE FROM accounts WHERE name LIKE 'guest\_%' AND created_at < UNIX_TIMESTAMP() - 1800;
Verificação de compatibilidade:

O Canary 3.4.1 NÃO possui a coluna "lastlogin" na tabela players. O Login Server
foi ajustado para não referenciar essa coluna. Se o schema for atualizado, verificar
novamente. (Fase 10)

O campo "email" na tabela accounts é mantido como string vazia, pois o MCR não
utiliza e-mail para autenticação.

A coluna created_at deve existir na tabela accounts. Se estiver ausente, adicione-a com:

sql
ALTER TABLE accounts ADD COLUMN created_at INT NOT NULL DEFAULT 0;

A coluna player_pronoun deve ser adicionada manualmente (ainda não faz parte do schema.sql original):
ALTER TABLE accounts ADD COLUMN player_pronoun TINYINT NOT NULL DEFAULT 0;

A coluna pronoun já existe na tabela players (nativa do Canary). Não precisa ser criada.

🔐 4. phpMyAdmin
================================================================================

URL: http://localhost/phpmyadmin (se o Apache estiver rodando).

Configuração do acesso (config.inc.php):
Local: D:\Programas\Xampp\phpMyAdmin\config.inc.php

Adicionar/alterar as linhas:
$cfg['Servers'][$i]['auth_type'] = 'config';
$cfg['Servers'][$i]['user'] = 'root';
$cfg['Servers'][$i]['password'] = '<Senha>';

⚠️ Se auth_type = 'config' e a senha NÃO for configurada, o phpMyAdmin tentará
conectar sem senha e falhará com "using password: NO". (Fase 33)

================================================================================
💾 5. BACKUP E RESTAURAÇÃO
================================================================================

Backup (exportar):
mysqldump -u root -p --single-transaction --routines --triggers --events --default-character-set=utf8mb4 BancoServer > backup_BancoServer_YYYYMMDD.sql

Restauração (importar):
mysql -u root -p --default-character-set=utf8mb4 BancoServer < backup_BancoServer_YYYYMMDD.sql

Procedimento recomendado:

Fazer backup ANTES de qualquer alteração no schema.

Manter backups datados em local seguro.

Testar a restauração em um banco secundário antes de aplicar em produção.

================================================================================
🔗 6. INTEGRAÇÃO COM CANARY E LOGIN SERVER
================================================================================

O Canary e o Login Server compartilham o mesmo banco de dados (BancoServer).

Configuração no Canary (config.lua):
mysqlHost = "127.0.0.1"
mysqlUser = "root"
mysqlPass = "<Senha>"
mysqlDatabase = "BancoServer"
mysqlPort = 3306
mysqlInitCommand = "SET NAMES 'utf8mb4'"

Configuração no Login Server (.env):
MYSQL_DBNAME=BancoServer
MYSQL_HOST=127.0.0.1
MYSQL_PORT=3306
MYSQL_USER=root
MYSQL_PASS=<Senha>

Ambos os sistemas precisam estar rodando para o fluxo completo de autenticação
e jogo funcionar.

================================================================================
⚠️ ARMADILHAS E LIÇÕES APRENDIDAS (EXTRAÍDAS DO DEVLOG)
================================================================================

MariaDB vs MySQL: O XAMPP atual usa MariaDB. O comando ALTER USER não funciona.
Use SET PASSWORD FOR 'root'@'localhost' = PASSWORD('senha'); (Fase 33)

Acesso negado (Access denied): Senha incorreta no config.lua ou .env.
Verificar se a senha foi definida no MariaDB e se os serviços estão rodando. (Fase 33)

Erro 10061 (Can't connect): O serviço MySQL/MariaDB não está rodando.
Iniciar pelo XAMPP Control Panel. (Fase 33)

Coluna lastlogin inexistente: O schema do Canary 3.4.1 não possui essa coluna.
O Login Server foi ajustado para removê-la da consulta. Ao atualizar o schema,
verificar novamente. (Fase 10)

Charset incorreto: Dados salvos com charset errado (latin1) podem corromper
caracteres acentuados. Sempre force utf8mb4 em todas as camadas. (Fase 32)

phpMyAdmin sem senha configurada: Mesmo com senha no MySQL, o phpMyAdmin
precisa da senha no config.inc.php. (Fase 33)

Personagens de teste: Para testar o fluxo de criação de personagem, é útil
criar personagens manualmente via SQL:
INSERT INTO players (name, account_id, level, vocation, sex, posx, posy, posz, town_id, ...) VALUES (...);

Coluna created_at: Necessária para expiração de contas guest. Deve ser adicionada
manualmente caso o schema não a possua. A rotina de limpeza do Login Server depende
deste campo para remover contas guest abandonadas.

Coluna player_pronoun: Armazena o pronome do JOGADOR (dono da conta).
Deve ser adicionada com ALTER TABLE accounts ADD COLUMN player_pronoun TINYINT NOT NULL DEFAULT 0;

Coluna pronoun: NATIVA do Canary. Armazena o pronome do PERSONAGEM.
NÃO utilize o storage 100001 para essa finalidade.
