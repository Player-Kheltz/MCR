Fornecer um roteiro único, ordenado e atualizado para implementar simultaneamente:

⚠️ REGRAS DE REVISÃO PARA O ASSISTENTE
- Seguir rigorosamente a ordem das fases; desvios só com autorização.
- Respeitar todos os guias temáticos aplicáveis.
- Strings em C++ com escapes octais Latin‑1 (OBRIGATÓRIO); scripts Lua em ISO‑8859‑1.
- Novas tabelas/colunas com charset utf8mb4 COLLATE utf8mb4_unicode_ci.
- Incluir logs DEBUG em todos os novos scripts Lua.
- Manter a máscara de Alma e conversão UTF‑8→Latin‑1 no cliente.
- NUNCA usar }? junto; sempre } ?.
- O Canary utiliza Revscript: os scripts .lua devem ser colocados nas subpastas apropriadas dentro de `data-canary/scripts/MCR/` (ex.: `Dominio/` para progressão, `Tutorial/` para o conteúdo da ilha, `Criacao/` para Oráculo). Cada subpasta é auto‑suficiente; os scripts são carregados automaticamente e podem registar os seus próprios eventos sem dependência de um ficheiro central.
- Os módulos partilhados (pronouns.lua, npc_utils.lua) estão em `MCR/Geral/` e são carregados automaticamente; não usar `dofile` para eles.
- A coluna `lastlogin` existe na tabela `players` e pode ser usada para verificar se é o primeiro login do personagem.

1.1 CRIAR TABELA dominios_def (com suporte a especializações)

sql
CREATE TABLE dominios_def (
    id SMALLINT AUTO_INCREMENT PRIMARY KEY,
    nome VARCHAR(50) NOT NULL,
    descricao VARCHAR(255),
    icone VARCHAR(50),
    parent_id SMALLINT DEFAULT NULL,
    FOREIGN KEY (parent_id) REFERENCES dominios_def(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
1.2 CRIAR TABELA habilidades_def

sql
CREATE TABLE habilidades_def (
    id INT AUTO_INCREMENT PRIMARY KEY,
    nome VARCHAR(50) NOT NULL,
    dominio_id SMALLINT NOT NULL,
    tipo ENUM('ativa','passiva') NOT NULL,
    requisito_afinidade INT DEFAULT 0,
    descricao TEXT,
    sinergia_dominio_id SMALLINT DEFAULT NULL,
    sinergia_afinidade INT DEFAULT NULL,
    FOREIGN KEY (dominio_id) REFERENCES dominios_def(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
1.3 CRIAR TABELA player_dominios

sql
CREATE TABLE player_dominios (
    player_id INT NOT NULL,
    dominio_id SMALLINT NOT NULL,
    afinidade INT DEFAULT 0,
    despertado TINYINT DEFAULT 0,
    PRIMARY KEY (player_id, dominio_id),
    FOREIGN KEY (player_id) REFERENCES players(id) ON DELETE CASCADE,
    FOREIGN KEY (dominio_id) REFERENCES dominios_def(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
1.4 CRIAR TABELA player_habilidades

sql
CREATE TABLE player_habilidades (
    player_id INT NOT NULL,
    habilidade_id INT NOT NULL,
    PRIMARY KEY (player_id, habilidade_id),
    FOREIGN KEY (player_id) REFERENCES players(id) ON DELETE CASCADE,
    FOREIGN KEY (habilidade_id) REFERENCES habilidades_def(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

2.2 ADICIONAR NOVOS MÉTODOS LUA DE DOMÍNIO, ESTADOS DE ALMA E POSTURAS
Em player.h/cpp e luascript.cpp, implementar:

player:getDominioAfinidade(dominioId) – retorna afinidade acumulada.

2.3 IMPLEMENTAR SISTEMA DE ESTADOS DE ALMA (LÓGICA EM C++)

calculateEstadosAlma() percorre o deque e:

Define Lampejo para uma especialização se as últimas 3 acções consecutivas (em 2 min) a incluem.

Define Vínculo se >60% das acções dos últimos 15 minutos incluem a especialização.

Cachear os estados e reavaliar periodicamente ou a cada registo.

Expor estados via getEstadoAlma.
