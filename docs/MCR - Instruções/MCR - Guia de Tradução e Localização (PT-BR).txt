- Pipeline de tradução automática supervisionada para C++.

  compatibilidade Latin-1 em todos os compiladores e sistemas.

- Ajuste do comando `look` para respeitar o pronome do personagem.
- Reforço da máscara de Alma em mensagens do servidor.

⚠️ REGRAS DE REVISÃO PARA O ASSISTENTE (TRADUÇÃO E LOCALIZAÇÃO)
================================================================================

Ao gerar QUALQUER texto, código, comentário ou mensagem para o Projeto MCR, o
assistente DEVE:

1. Escrever em português do Brasil CORRETO (ortografia, acentuação, pontuação,
   concordância nominal e verbal).
2. Identificar a camada em que o texto será usado e aplicar a codificação correta
   (ver seção 2 – Codificação por Tipo de Arquivo).
3. Nunca usar traduções automáticas grosseiras. Preferir linguagem natural,
   coerente com a ambientação fantasiosa.
4. Manter a consistência terminológica: usar os mesmos termos para os mesmos
   conceitos em todo o projeto (ex.: "vocação" não "classe", "personagem" não "boneco").
5. Respeitar a personalidade do NPC ao redigir diálogos (ver Guia de Narrativa).
6. Na comunicação backend→frontend, JAMAIS enviar strings acentuadas em JSON.
   Usar o padrão error_code (ver seção 4).
7. No chat do jogo, destacar palavras importantes SOMENTE com {chaves}. NÃO usar
   tags como `{{#RRGGBB}}` ou HTML.
8. Nunca usar "}?" junto; sempre "} ?" para evitar erro visual no cliente.
9. **ATUALIZADO**: Em todas as mensagens de sistema que mencionem o personagem convidado,
   aplicar a máscara de Alma (substituir "Guest_*" por "Alma").
10. **ATUALIZADO**: Em todas as mensagens dirigidas ao jogador, utilizar o **pronome do personagem**
    obtido via módulo pronouns.lua (Pronouns.getSubject, etc.). NÃO usar storage 100001.
11. Ao traduzir código C++ (servidor ou cliente), utilizar SEMPRE a pipeline
    automática supervisionada: ferramentas de tradução automática seguidas de
    revisão humana. Apenas strings de interface e comentários são traduzidos;
    nomes de variáveis, funções e bindings Lua permanecem em inglês.
12. Nos ficheiros .cpp/.h, representar caracteres acentuados através de escapes
    octais (`\xyz`) com os códigos da tabela Latin-1 (ISO-8859-1). Isto evita
    problemas de BOM, compilador e encoding do sistema de ficheiros.
    Exemplo: "Nível" → "N\355vel"; "Vocação" → "Voca\347\343o".
13. Jamais alterar assinaturas de funções expostas ao Lua (`registerMethod`,
    `registerClass`). A tradução de strings internas não pode quebrar os nomes
    dos métodos visíveis nos scripts.
14. Proteger ficheiros de cabeçalho (.h/.hpp): comentários podem ser traduzidos,
    mas macros, nomes de classes, enums e inclusões permanecem intactos.

================================================================================
📋 1. ESCOPO DE APLICAÇÃO DO PT-BR
================================================================================

Devem estar OBRIGATORIAMENTE em português:

- Diálogos de NPC (npc:say, sendTextMessage para o jogador).
- Descrições de itens (name, description).
- Mensagens de sistema (broadcasts, MOTD, mensagens de login/logout).
- Textos de janelas modais (títulos, botões, corpo do texto).
- Labels em arquivos .otui (interface do cliente).
- Notificações toast e HUD de missões.
- Comentários em TODOS os arquivos de código (C++, Lua servidor, Lua cliente, Go).
- Logs do servidor e do Login Server (mensagens de inicialização, erros).
- Nomes de NPCs, monstros, locais e itens no mapa/XML.
- Documentação do projeto (guias, DevLog, instruções).
- Mensagens de "look" (examinar) para criaturas e itens.

Podem permanecer em INGLÊS:

- Nomes de variáveis, funções, classes, enums, namespaces (ex.: getPlayer, onLogin).
- Palavras-chave das linguagens (if, return, function, end, class, package).
- Chaves de configuração em tabelas (ex.: modalWindows = true).
- Nomes de campos em JSON (ex.: "account", "password", "error_code").
- Comandos SQL e nomes de tabelas/colunas (ex.: SELECT * FROM accounts).
- Endpoints de API (/register, /login).

================================================================================
📝 2. CODIFICAÇÃO POR TIPO DE ARQUIVO (REVISADA)
================================================================================

Esta é a seção MAIS CRÍTICA deste guia. A codificação errada é a causa número 1
de bugs visuais (acentos corrompidos) no Projeto MCR.

| Tipo de Arquivo                          | Codificação      | BOM?  | Exemplo                                |
|------------------------------------------|------------------|-------|----------------------------------------|
| Scripts Lua do CLIENTE (dofile, módulos) | ISO-8859-1       | Não   | registro.lua, entergame.lua            |
| Scripts Lua do SERVIDOR (data/npc/, etc.)| ISO-8859-1       | Não   | oraculo.lua, npc_utils.lua, player.lua |
| Código C++ (servidor e cliente)          | UTF-8            | COM   | canary_server.cpp, main.cpp            |
| Código Go (Login Server)                 | UTF-8            | COM   | register.go, login.go                  |
| Arquivos .otui (interface)               | UTF-8            | SEM   | register_window.otui                   |
| Arquivos .xml (spawn, itens)             | UTF-8            | SEM   | MCR-npc.xml, items.xml                 |
| Arquivos de configuração (.lua, .env)    | UTF-8            | SEM   | config.lua, .env                       |
| Documentação (.txt)                      | UTF-8            | COM   | MCR - Guia de *.txt                    |
| Código C++ (servidor e cliente) | UTF-8 COM BOM **ou** ASCII com escapes octais Latin-1 | COM (UTF-8) ou SEM (ASCII) | canary_server.cpp, creature.cpp |

Detalhamento:

⚠️ CÓDIGO C++ (SERVIDOR E CLIENTE):
A partir da versão 5.0, todo o código-fonte C++ do servidor e do cliente foi
traduzido para PT-BR utilizando uma pipeline de tradução automática supervisionada.
Para garantir a máxima compatibilidade e evitar quaisquer artefactos de encoding,
todas as strings que contêm acentos ou caracteres especiais são representadas
exclusivamente através de escapes octais da tabela Latin-1 (ISO-8859-1).
- Exemplo: `"bem-vindo ao Projeto MCR"` pode ser escrito como `"bem-vindo ao Projeto MCR"` (sem acentos) ou, se necessário, `"bem-vindo \340o Projeto MCR"` para "à".
- NÃO se utiliza UTF-8 directamente nas strings; o ficheiro pode ser salvo como
  ASCII puro, sem BOM, pois os escapes garantem a representação correcta.
- Comentários também seguem a mesma regra, embora seja preferível usar apenas
  caracteres ASCII nos comentários para evitar problemas.

⚠️ PROTEÇÃO DE BINDINGS LUA:
Durante a tradução, foram preservados todos os nomes de funções e classes
registados no Lua (ex.: `Player::getPronoun`, `Game::logout`). Nomes de
parâmetros em C++ podem ser traduzidos nos comentários, mas nunca nos
`registerMethod` ou `registerClass`.


⚠️ ISO-8859-1 (SCRIPTS LUA – CLIENTE E SERVIDOR):
Scripts Lua que contenham strings acentuadas e sejam exibidos ao jogador (tanto no
cliente quanto no servidor) DEVEM ser salvos como ISO-8859-1 (Latin-1 / ANSI no Windows).
Isso porque o OTClient interpreta todas as strings recebidas como Latin-1, mesmo
quando originadas de um script do servidor.
- Caracteres como 'ç', 'ã', 'é' são representados por um ÚNICO byte (ex.: 0xE7 = ç).
- Se salvo como UTF-8, aparecerão como "Ã§", "Ã£", "Ã©" na interface.
- Essa regra se aplica a todos os scripts Lua que geram texto visível ao jogador,
  sejam eles do cliente (dofile) ou do servidor (data/creaturescripts/, data/events/, data/npc/).

⚠️ UTF-8 COM BOM (C++, GO):
O compilador C++ e o compilador Go reconhecem o BOM e tratam o arquivo como UTF-8.
Isso garante acentuação correta em logs e mensagens internas.

⚠️ UTF-8 SEM BOM (OTUI, XML):
Arquivos .otui e .xml são processados por parsers que podem rejeitar o BOM.
Salvar sem BOM, mas manter como UTF-8.

================================================================================
🌐 3. COMUNICAÇÃO BACKEND → FRONTEND (EVITAR CORRUPÇÃO)
================================================================================

Devido à incompatibilidade entre o backend Go (UTF-8) e a interpretação do cliente
Lua (ISO-8859-1 para respostas HTTP), strings acentuadas NÃO devem ser enviadas
diretamente no JSON de resposta do Login Server.

PROTOCOLO OBRIGATÓRIO (ERROR_CODE):

1. O backend (Login Server) retorna um código de erro padronizado em INGLÊS,
   SEM acentos, no campo "error_code".
   Exemplo: "account_name_taken", "password_too_weak", "invalid_credentials".

2. O cliente (Lua) possui uma TABELA DE MAPEAMENTO que traduz o error_code para
   a mensagem em PT-BR (com acentos).
   Exemplo:
   lua
   local ERROR_MESSAGES = {
       account_name_taken = "O nome de conta já está em uso.",
       password_too_weak = "A senha deve ter 8+ caracteres com letras e números.",
       invalid_credentials = "Nome de conta ou senha incorretos.",
       registration_success = "Conta criada com sucesso!",
       guest_creation_failed = "Falha ao criar conta convidada.",
       alma_creation_failed = "Falha ao criar personagem convidado.",
   }

3. Esta tabela EXISTE no script Lua do cliente (registro.lua, entergame.lua, etc.),
   que está salvo como ISO-8859-1. Portanto, as strings em PT-BR exibidas ao
   jogador terão os acentos CORRETOS.

4. NOVOS error_codes devem ser adicionados tanto no backend quanto na tabela de
   mapeamento do cliente.

================================================================================
🔑 4. DESTAQUE NO CHAT
================================================================================

O OTClient suporta exclusivamente o destaque de palavras envolvidas por {chaves}.
Esse realce é herdado do protocolo Tibia e não requer nenhuma modificação
no cliente.

Regras:
- Utilize {chaves} para destacar ações ({aceitar}, {entregar}), itens
  ({10 Poções de Vida}), locais ({Floresta Sombria}) e recompensas
  ({500 pontos de experiência}).
- NÃO utilize tags `{{#RRGGBB}}`, HTML, ou qualquer outra sintaxe – elas
  não produzem efeito visual.
- **NUNCA use "}?" junto; sempre "} ?" para evitar corrupção visual no cliente.**

================================================================================
✍️ 5. ESTILO E QUALIDADE DO TEXTO
================================================================================

Além da correção gramatical, o texto do Projeto MCR deve:

- Ser NATURAL: evitar traduções literais do inglês ("você completou a quest" soa
  artificial; prefira "você concluiu a missão").
- Ser COERENTE com o universo: um NPC ancião fala de forma mais formal; um goblin
  mercador fala de forma mais tosca e coloquial.
- Evitar estrangeirismos desnecessários: "missão" em vez de "quest", "elo" em vez
  de "link", "guardião" em vez de "knight" (em contexto narrativo).
- Usar o padrão culto da língua, mas com flexibilidade para personagens caricatos
  (um bêbado pode falar errado; um dragão ancião pode usar arcaísmos).
- Manter consistência: se um termo foi traduzido de uma forma, usar a mesma
  tradução em todo o projeto.

Lista de termos padronizados (atualizada):
- "vocação" (não "classe")
- "personagem" (não "char", "boneco")
- "missão" (não "quest", embora "quest" possa aparecer em comentários de código)
- "servidor" (não "server")
- "conta" (não "account")
- "senha" (não "password")
- "Você vê" (não "You see")
- "Vida" (não "Health")
- "Mana" (não "Mana" – mantém-se igual)
- "Posição" (não "Position")
- "Pronome" (para a identidade do personagem, armazenada em players.pronoun)
- "Sexo" (apenas para a escolha biológica do corpo do personagem)
- "escapar octal" – representação `\ooo` para caracteres Latin‑1.
- "pipeline de tradução" – processo automático + revisão.

🎭 6. MÁSCARA DE ALMA EM MENSAGENS DO SERVIDOR (REVISADA)
================================================================================

O personagem convidado (Guest_*) deve ser sempre apresentado como "Alma" para
o jogador. Esta substituição deve ser aplicada em TODAS as mensagens de sistema,
incluindo:

- Mensagens de "look" (examinar personagem)
- Mensagens de entrada/saída de jogadores
- Qualquer broadcast que mencione o nome do personagem convidado
- Descrições em janelas modais ou tooltips

A máscara é implementada nos scripts do servidor substituindo o nome real
pelo nome "Alma" antes de enviar a mensagem ao jogador:

```lua
local name = creature:getName()
if name:sub(1,6) == "Guest_" then
    name = "Alma"
end
Arquivos que já possuem a máscara:

data/creaturescripts/scripts/player.lua (onLookInBattleList)

data/events/scripts/on_look.lua (handleCreatureDescription)

Novos scripts que gerem mensagens com o nome do personagem devem incluir
a mesma verificação.

📌 7. PRONOME DO PERSONAGEM EM MENSAGENS (ATUALIZADO)
================================================================================

A partir da versão 4.0, todas as mensagens que se dirijam ao personagem devem
respeitar o pronome definido na coluna players.pronoun, obtido preferencialmente
pelo módulo pronouns.lua:

lua
dofile('data/MCR Scripts/pronouns.lua')
local subject = Pronouns.getSubject(player)  -- "he", "she" ou "they"
local object = Pronouns.getObject(player)    -- "him", "her", "them"
Exemplos de uso:

Bem-vindo: "Bem-vindo" (he), "Bem-vinda" (she), "Bem-vinde" (they)

Mensagens de sistema: "Parabéns, você foi promovido/a."

Não definido (0): usar formas neutras ("viajante", "aventureiro(a)") ou perguntar
ao jogador qual o pronome desejado e atualizar players.pronoun.

⚠️ ARMADILHAS E LIÇÕES APRENDIDAS (EXTRAÍDAS DO DEVLOG)
================================================================================

Acentos corrompidos no cliente e servidor: Scripts Lua de ambas as pontas
DEVEM ser salvos como ISO-8859-1 para evitar artefatos. (Fase 22.3)

Mensagens do backend com acentos: Enviar strings acentuadas no JSON do Login
Server resultava em "jÃi estÃi" no cliente. A solução foi o padrão error_code
com mapeamento no cliente. (Fase 22.3)

Console do Canary com "Vers├úo": Os logs em português apareciam corrompidos
porque o console do Windows usava CP 850. A solução foi adicionar
SetConsoleOutputCP(CP_UTF8) no main.cpp. (Fase 40)

Destaque no chat: Apenas {chaves} são reconhecidas; qualquer tentativa de
usar {{#RRGGBB}} falha silenciosamente.

ISO-8859-1 vs ANSI: No Windows, "ANSI" é um apelido para a codificação
Windows-1252, que é um superset do ISO-8859-1. Salvar como "ANSI" no Notepad++
ou VS Code geralmente funciona, desde que não haja caracteres específicos do
Windows-1252 (como €, ‚, ƒ) nos scripts, o que é raro.

Mensagens de look do servidor estavam em inglês: Foram traduzidas ("You see"
→ "Você vê") e a máscara de Alma foi aplicada para consistência total.
