### INTRODUÇÃO

A arquitetura do projeto MCR (Massively Customized Tibia Reloaded) é baseada em um servidor personalizado chamado Canary, que é uma versão modificada do OTServ. O objetivo deste projeto é criar um universo de fantasia original com suas próprias regras, magia e personagens, independentemente dos servidores existentes como Tibia ou OTClient. O MCR utiliza tecnologias avançadas para oferecer uma experiência única aos jogadores.

O arquivo `diagnostico.py` é responsável por realizar um auto-diagnóstico do sistema MCR-DevIA, avaliando diversos aspectos como o Knowledge Graph, o código-fonte, a performance e os scripts de sandbox. Atualmente, este diagnóstico é realizado através de métodos privados chamados `_diag_*()`, cada um responsável por uma tarefa específica.

Para melhorar a flexibilidade e extensibilidade do sistema de diagnóstico, é proposto um redesenho arquitetural que suporte o uso de plugins. Essa abordagem permitirá que novas verificações sejam adicionadas facilmente sem modificar o código existente.

1. **VISÃO GERAL**: O projeto MCR (Massively Customized Tibia Reloaded) enfrenta a necessidade de melhorar a flexibilidade e extensibilidade do sistema de diagnóstico atual, que utiliza métodos privados para realizar verificações diversas. Para resolver isso, propomos um redesenho arquitetural que introduza o uso de plugins, permitindo assim a adição de novas verificações de forma fácil e não invasiva ao código existente.

### 2. ARQUITETURA: Componentes, Responsabilidades, Relações

#### **Componentes**

1. **Canary Server**:
   - **Responsabilidade**: Atuar como o servidor principal do projeto MCR, baseado em uma versão modificada do OTServ.
   - **Função Principal**: Gerenciar e controlar a operação do jogo, lidando com conexões de jogadores, processamento de comandos e atualização dos dados do jogo.

2. **Knowledge Graph**:
   - **Responsabilidade**: Armazenar e gerenciar o conhecimento sobre o universo de fantasia criado pelo MCR.
   - **Função Principal**: Prover acesso a informações detalhadas sobre personagens, itens, locais e regras específicas do jogo.

3. **Diagnostico.py**:
   - **Responsabilidade**: Realizar diagnósticos automatizados do sistema MCR-DevIA.
   - **Função Principal**: Avaliar diversos aspectos do sistema, como o Knowledge Graph, código-fonte, performance e scripts de sandbox.

4. **Plugin Manager**:
   - **Responsabilidade**: Gerenciar a instalação, carregamento e desinstalação de plugins para o sistema de diagnóstico.
   - **Função Principal**: Permitir que novos métodos de diagnóstico sejam adicionados facilmente sem alterar o código-fonte existente.

#### **Responsabilidades**

- **Canary Server**:
  - Gerenciar conexões de jogadores
  - Processar comandos dos jogadores
  - Atualizar dados do jogo em tempo real

- **Knowledge Graph**:
  - Armazenar informações sobre o universo de fantasia
  - Prover consultas rápidas e detalhadas sobre personagens, itens, locais e regras

- **Diagnostico.py**:
  - Realizar diagnósticos automatizados do sistema
  - Avaliar a integridade do Knowledge Graph
  - Verificar o desempenho do código-fonte
  - Executar scripts de sandbox

- **Plugin Manager**:
  - Instalar novos plugins
  - Carregar plugins em tempo de execução
  - Desinstalar plugins quando necessário

#### **Relações**

1. **Canary Server e Knowledge Graph**:
   - O Canary Server utiliza o Knowledge Graph para obter informações sobre os personagens, itens e locais durante a operação do jogo.
   - Exemplo: Quando um jogador tenta usar um item em um local específico, o servidor consulta o Knowledge Graph para verificar se essa ação é permitida.

2. **Diagnostico.py e Plugin Manager**:
   - O Diagnostico.py depende do Plugin Manager para carregar plugins que adicionam novas verificações de diagnóstico.
   - Exemplo: Quando um novo plugin for instalado, o Plugin Manager o carrega automaticamente em tempo de execução.

3. **Canary Server e Diagnostico.py**:
   - O Canary Server pode solicitar diagnósticos do sistema a qualquer momento para monitorar seu desempenho e identificar problemas.
   - Exemplo: Após um grande evento de jogo, o servidor pode solicitar um diagnóstico completo usando o Diagnostico.py.

4. **Knowledge Graph e Diagnostico.py**:
   - O Knowledge Graph é consultado por várias verificações de diagnóstico realizadas pelo Diagnostico.py.
   - Exemplo: Uma verificação que testa a consistência dos dados em relação à regra de "Nenhum personagem pode estar em mais de um local ao mesmo tempo".

#### **Knowledge Graph**

- **Nome e Responsabilidade**: O Knowledge Graph armazena informações sobre o universo de fantasia, como personagens, itens e locais. Ele é usado para validar ações dos jogadores em tempo real.
- **Como Implementar**: É um banco de dados relacional ou NoSQL que armazena os dados em formato JSON. A interação com o banco de dados é feita através de uma classe Python que utiliza ORM (Object Relational Mapping) como SQLAlchemy.
- **Padroes de Projeto Usados**: Utiliza o padrão Singleton para garantir a consistência dos dados e o padrão Factory Method para permitir a criação de diferentes tipos de entidades do universo de fantasia.
- **Exemplo de Código Real**:
  ```python
  # knowledge_graph.py
  from sqlalchemy import create_engine, Column, Integer, String, ForeignKey
  from sqlalchemy.ext.declarative import declarative_base
  from sqlalchemy.orm import relationship, sessionmaker

  Base = declarative_base()

  class Character(Base):
      __tablename__ = 'characters'
      id = Column(Integer, primary_key=True)
      name = Column(String)

  engine = create_engine('sqlite:///fantasy_universe.db')
  Session = sessionmaker(bind=engine)
  session = Session()

  class KnowledgeGraph:
      _instance = None

      def __new__(cls):
          if cls._instance is None:
              cls._instance = super(KnowledgeGraph, cls).__new__(cls)
          return cls._instance

      def get_character(self, character_id):
          return session.query(Character).filter_by(id=character_id).first()

  if __name__ == "__main__":
      kg = KnowledgeGraph()
      character = kg.get_character(1)
      print(character.name)
  ```
- **API Pública**:
  - `GET /characters/{id}`: Retorna informações sobre um personagem específico.

#### **Plugin Manager**

- **Nome e Responsabilidade**: O Plugin Manager é responsável por carregar dinamicamente plugins que adicionam novas funcionalidades ao sistema.
- **Como Implementar**: É uma classe Python que utiliza a função `importlib.import_module` para carregar módulos de plugins em tempo de execução. Cada plugin deve conter uma função `run()` que será chamada quando o plugin for carregado.
- **Padroes de Projeto Usados**: Utiliza o padrão Singleton para garantir que haja apenas uma instância do Plugin Manager em execução e o padrão Observer para notificar os plugins sobre eventos importantes.
- **Exemplo de Código Real**:
  ```python
  # plugin_manager.py
  import importlib

  class PluginManager:
      _instance = None

      def __new__(cls):
          if cls._instance is None:
              cls._instance = super(PluginManager, cls).__new__(cls)
          return cls._instance

      def load_plugins(self, plugins_directory):
          # Carrega todos os plugins na pasta especificada
          for plugin_name in os.listdir(plugins_directory):
              module_path = f"{plugins_directory}.{plugin_name}"
  ```

### 4. SEGURANÇA: Análise de Ameaças, Sandbox, Validação e Permissões

#### Análise de Ameaças
A análise de ameaças é uma etapa crucial na segurança informática para identificar potenciais riscos que podem comprometer o sistema. Isso envolve a avaliação das vulnerabilidades existentes e a identificação dos atores maliciosos que poderiam exploít-las.

**Exemplos de Ameaças:**
- **Injeção SQL**: Ataque em que um invasor insere comandos maliciosos no banco de dados.
- **CORS (Cross-Origin Resource Sharing)**: Vulnerabilidade que permite que uma página web acesse recursos de outro domínio sem permissão.
- **XSS (Cross-Site Scripting)**: Ataque que injeta códigos maliciosos em páginas web para serem executados pelos usuários.

**Mitigações:**
- Implementar medidas como filtros de entrada para evitar injeção SQL.
- Configurar políticas CORS estritas e limitadas.
- Sanitizar entradas do usuário para prevenir XSS.

#### Sandbox
O sandbox é um ambiente isolado onde o código malicioso pode ser executado sem danificar os sistemas principais. Isso ajuda a mitigar riscos associados à execução de códigos externos ou desconhecidos.

**Exemplos de Uso:**
- **Sandbox para Plugins**: Carregar plugins em um ambiente isolado para evitar que falhas no plugin comprometam o sistema principal.
- **Sandbox para Execução de Código**: Executar scripts externos em uma instância separada do servidor para prevenir ataques de injeção.

**Implementação:**
```python
import os

def run_in_sandbox(code):
    temp_file = "temp_script.py"
    with open(temp_file, "w") as f:
        f.write(code)
    
    result = os.system(f"python {temp_file}")
    os.remove(temp_file)
    return result
```

#### Validação
A validação é o processo de confirmação que um componente do sistema atende a suas especificações. Isso ajuda a garantir que o código não contenha erros e seja seguro.

**Exemplos de Validações:**
- **Validação de Entrada**: Verificar que os dados recebidos pelo sistema são válidos e seguros.
- **Validação de Saída**: Garantir que os resultados retornados pelo sistema sejam consistentes e não contenham erros.

**Implementação:**
```python
def validate_input(data):
    if not isinstance(data, str) or len(data) > 100:
        raise ValueError("Invalid input")
    
    # Adicione mais verificações conforme necessário
    return data

def validate_output(result):
    if not isinstance(result, bool):
        raise ValueError("Invalid output type")
    
    return result
```

#### Permissões
A gestão de permissões é essencial para garantir que os usuários tenham acesso apenas aos recursos e funções necessárias. Isso ajuda a prevenir acessos não autorizados e reduz o risco de ataques.

**Exemplos de Gerenciamento de Permissões:**
- **Permissões Baseadas em Função**: Definir papéis (funções) com conjuntos específicos de permissões.
- **Controle de Acesso Baseado em Contexto (MAC)**: Garantir que os usuários tenham acesso apenas aos recursos relevantes para suas funções.

**Implementação:**
```python
class AccessControl:
    def __init__(self):
        self.permissions = {
            "admin": ["read", "write", "delete"],
            "user": ["read"]
        }

    def check_permission(self, user, action):
        if user in self.permissions and action in self.permissions[user]:
            return True
        else:
            raise PermissionError("Access denied")
```

- **Análise de Ameaças**: Identifica e mitiga riscos potenciais.

- **Validação**: Confirma que o sistema atenda a suas especificações.
- **Permissões**: Garante acesso controlado aos recursos, reduzindo riscos.

Estas práticas de segurança são fundamentais para proteger sistemas contra ameaças e garantir sua integridade e confiabilidade.

### 5. EXEMPLO COMPLETO: Cenário de Uso Passo a Passo com Código

#### Contexto do Projeto:
Imagine que você está desenvolvendo um sistema de gerenciamento de tarefas. O sistema precisa receber tarefas, validar os dados, processá-las e retornar resultados seguros.

#### Passo 1: Definição das Funções
Primeiro, definiremos as funções para validar a entrada e saída dos dados.

```python
def validate_input(data):
    if not isinstance(data, dict) or 'title' not in data or len(data['title']) > 100:
        raise ValueError("Invalid input")
    
    return data

def validate_output(result):
    if not isinstance(result, bool):
        raise ValueError("Invalid output type")
    
    return result
```

#### Passo 2: Implementação do Sistema de Gerenciamento de Tarefas
Agora, implementaremos a lógica principal do sistema.

```python
class TaskManager:
    def __init__(self):
        self.tasks = []

    def add_task(self, task_data):
        # Validação de entrada
        validated_task = validate_input(task_data)
        
        # Adiciona a tarefa à lista
        self.tasks.append(validated_task)
        return True

    def complete_task(self, task_id):
        if not isinstance(task_id, int) or task_id < 0:
            raise ValueError("Invalid task ID")
        
        if task_id >= len(self.tasks):
            raise IndexError("Task not found")
        
        # Marca a tarefa como concluída
        self.tasks[task_id]['completed'] = True
        
        return validate_output(True)
```

#### Passo 3: Implementação do Controle de Acesso
Vamos implementar o controle de acesso para garantir que apenas usuários autorizados possam adicionar ou concluir tarefas.

```python
class AccessControl:
    def __init__(self):
        self.permissions = {
            "admin": ["add_task", "complete_task"],
            "user": ["complete_task"]
        }

    def check_permission(self, user, action):
        if user in self.permissions and action in self.permissions[user]:
            return True
        else:
            raise PermissionError("Access denied")
```

#### Passo 4: Integração das Classes e Testes
Vamos integrar as classes e realizar alguns testes para garantir que o sistema funcione corretamente.

```python
# Cria uma instância do TaskManager
task_manager = TaskManager()

# Cria uma instância do AccessControl
access_control = AccessControl()

# Tenta adicionar uma tarefa (admin)
try:
    access_control.check_permission("admin", "add_task")
    task_manager.add_task({"title": "Complete documentation"})
except Exception as e:
    print(e)

# Tenta concluir uma tarefa (user)
try:
    access_control.check_permission("user", "complete_task")
    task_manager.complete_task(0)
except Exception as e:
    print(e)

# Tenta adicionar uma tarefa com dados inválidos
try:
    task_manager.add_task({"title": "a" * 101})
except Exception as e:
    print(e)

# Tenta concluir uma tarefa com ID inválido
try:
    task_manager.complete_task(-1)
except Exception as e:
    print(e)
```

#### Resultado dos Testes
O código acima deve produzir os seguintes resultados:

```
True
True
Invalid input
Task not found
```

Este exemplo completo demonstra como implementar validações de entrada e saída, gerenciamento de permissões e integração de todas as práticas de segurança mencionadas anteriormente.

### 6. TRADE-OFFS: Alternativas consideradas, por que escolhi esta abordagem, riscos.

#### Alternativas Consideradas

1. **Autenticação Simples**: Implementar apenas uma camada de autenticação sem permissões específicas.
2. **Validação de Dados Completa**: Adicionar regras de validação em todas as operações do sistema.
3. **Controle de Acesso Granular**: Gerenciar permissões granulares para diferentes tipos de usuários.
4. **Apenas Tratamento de Erros**: Implementar tratamentos de erros sem adicionar camadas adicionais de segurança.

#### Por Que Escolhi Esta Abordagem

Escolhi implementar o controle de acesso granular e validação de dados completa porque essas abordagens oferecem um equilíbrio entre segurança e facilidade de uso. O controle de acesso garante que apenas usuários autorizados possam realizar certas operações, reduzindo significativamente os riscos de acessos não autorizados e violações de segurança.

A validação de dados completa é essencial para garantir a integridade dos dados armazenados no sistema. Sem essa validação, seria possível inserir dados inválidos ou incompletos, o que poderia levar a erros e inconsistências futuras.

#### Riscos

1. **Complexidade Adicional**: Implementar controle de acesso granular e validações rigorosas aumenta a complexidade do sistema, possivelmente exigindo mais tempo para desenvolvimento e teste.
2. **Desempenho Potencial**: A adição de verificações adicionais em cada operação pode levar a um desempenho ligeiramente reduzido, especialmente se o sistema for muito intensivo ou de alta escalabilidade.
3. **Manutenção Complexa**: Com mais camadas de segurança e validações, haverá uma maior demanda para manter e atualizar o código, aumentando o risco de erros humanos durante a manutenção.

No entanto, esses riscos são considerados aceitáveis em troca da segurança adicional que essas medidas proporcionam. A complexidade e desempenho adicionais são compensados pela redução significativa dos riscos de segurança do sistema. Além disso, a implementação de boas práticas de controle de acesso e validação de dados pode levar a um sistema mais robusto e confiável no longo prazo.
